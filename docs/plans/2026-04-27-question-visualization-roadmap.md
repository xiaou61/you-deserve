# 八股文全题图解化二次优化路线

## 目标

把题库从“能读懂”升级成“能看图秒懂”。每道八股文后续都补一张通俗图，帮助双非本科同学先建立直觉，再背面试话术。

## 图解分型

### 流程图

适合解释“先后步骤”的题：

- Spring MVC 请求流程
- 类加载过程
- TCP 三次握手和四次挥手
- Cache Aside 读写流程
- MQ 可靠消息链路
- 订单状态机

图像要求：从左到右或从上到下，步骤不超过 7 个，关键失败点用红色或警示标记。

### 结构图

适合解释“组件关系”的题：

- JVM 内存区域
- HashMap 结构
- Redis Cluster 槽和节点
- Kafka topic、partition、segment
- Spring IoC 容器
- MySQL Buffer Pool

图像要求：用容器、箭头、层级关系表达，不堆大段文字。

### 对比图

适合解释“容易混淆”的题：

- char vs varchar
- HashMap vs ConcurrentHashMap
- Spring MVC vs WebFlux
- Redis Pub/Sub vs Stream
- gRPC vs REST
- 乐观锁 vs 悲观锁

图像要求：左右双栏，对比维度控制在 4 到 6 个。

### 时序图

适合解释“多角色交互”的题：

- JWT 登录
- Kafka 生产者 acks
- 事务消息
- 本地消息表
- 分布式锁
- RPC 超时重试

图像要求：角色不超过 5 个，突出请求、确认、失败补偿。

### 场景图

适合解释“项目设计”的题：

- 优惠券系统
- 通知系统
- 评论系统
- 搜索系统
- Excel 导入导出
- 数据权限

图像要求：用真实业务场景类比，例如仓库、门卫、流水账、前置仓、广播站。

## 每道题的图解素材来源

优先从现有 Markdown 中抽取：

- `一句话结论`：作为图标题下方的短解释。
- `通俗解释`：作为图的视觉类比来源。
- `面试回答`：抽取节点、箭头和关键概念。
- `常见追问`：作为图下角的“面试官追问”提示。
- `记忆钩子`：作为图底部强化记忆点。

## 生成规范

- 一图只讲一个核心概念。
- 中文短句优先，每个节点不超过 12 个字。
- 不使用大段段落文字。
- 手机端也能看清，推荐 16:9 和 4:5 两套比例。
- 色彩保持克制：主色用于流程，红色只用于风险点。
- 所有图保留统一站点风格：清晰、温暖、不装腔。
