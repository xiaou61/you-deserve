# 题目质检 Batch 9：Java 81-90

审查时间：2026-05-03  
范围：排序后的第 81-90 道题，`content/questions/java/copyonwritearraylist.md` 至 `content/questions/java/hashmap-thread-unsafe.md`  
审查口径：事实准确性、表达清晰度、内容深度、面试可用度、是否存在生成模板污染。  

## 总体结论

这一批 Java 题的基础概念多数准确，`countdownlatch-cyclicbarrier-semaphore`、`final-finally-finalize`、`generic-erasure`、`hashmap-thread-unsafe` 的正文质量相对较好。主要问题仍然是自动扩写带来的“Java 题工程化套壳”：

- 10 篇都命中了通用模板，常见表达包括“数据量或并发量扩大 10 倍”“配置不生效”“灰度、压测和回滚策略”“JFR/GC 日志”，对枚举、final、泛型、HashMap 这类基础题不贴合。
- `false-sharing.md` 存在重复 `## 图解提示`；`countdownlatch-cyclicbarrier-semaphore`、`generic-erasure`、`hashmap-load-factor` 有明显图解节点截断。
- `generic-erasure.md` 部分泛型示例在正文里变成 `List 和 List`、`List ->`，疑似尖括号未用代码格式保护或生成时被截掉，影响理解，应优先修。
- `hashmap-load-factor.md` 和 `hashmap-resize.md` 的 `详细讲解/深挖理解` 把代码块内容揉成了 `text threshold = capacity * loadFactor`，表达不自然。

## 优先修复建议

1. 先修 `generic-erasure.md`：把所有泛型示例用反引号保护，避免 `List<String>` 被渲染或生成成 `List`，并补擦除限制的具体例子。
2. 清理 `false-sharing.md` 的重复 `图解提示`，并把图解改成“变量布局 -> 缓存行失效 -> 基准验证 -> 填充/隔离”的完整链路。
3. 修复 `hashmap-load-factor.md`、`hashmap-resize.md` 中代码块拼接到正文的问题。
4. Java 基础题统一把 `追问准备` 换成专项追问，不再使用“数据量或并发量扩大 10 倍”“灰度回滚”这类项目模板。
5. 对并发工具题补异常边界和工程限制：Latch finally countDown、Barrier broken、Semaphore permit 泄漏、ForkJoin 阻塞任务和公共池风险。
