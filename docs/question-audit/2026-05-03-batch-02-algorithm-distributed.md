审查时间：2026-05-03  
范围：排序后的第 11-20 道题，`content/questions/algorithm/two-pointers.md` 至 `content/questions/distributed/distributed-clock-skew.md`  
审查口径：事实准确性、表达清晰度、内容深度、面试可用度、是否存在生成模板污染。  

## 总体结论

这一批前半部分的核心短答案大多可用，尤其 `api-gateway`、`cap-base`、`consistent-hash` 的主体解释能覆盖入门面试。主要问题集中在自动扩写的后半段：

- 分布式题大量混入消息队列模板，`生产者 -> Broker -> 消费者 -> 数据库 -> 补偿任务`、`死信数量`、`人工重放`、`消息丢失/重复/乱序/积压` 出现在网关、配置中心、熔断、舱壁隔离、时钟偏移等题里，容易让读者误以为这些题都要按 MQ 链路回答。
- 算法题仍有系统设计套话，如“灰度、压测、服务日志、高并发”等，和双指针、并查集的面试重点不匹配。
- 个别题存在理解边界需要补强：并查集应讲路径压缩和按秩/按大小合并后的近似常数复杂度，也要提示删除元素不适合普通并查集；时钟偏移题不宜笼统说“缓存 TTL 都受本地时钟影响”，Redis 这类服务端 TTL 更应区分服务端时间、客户端令牌过期、Snowflake 回拨等场景。
- `图解提示` 仍有重复标题和节点截断，如一致性哈希里出现 `hash(ke…`，会影响后续 AI 生图质量。

## 优先修复建议

1. 先批量删除这一批分布式题里的 MQ 模板段落，尤其是 `生产者 -> Broker -> 消费者`、`死信数量`、`人工重放`、`消息丢失/重复/乱序/积压` 这类不属于题目的固定话术。
2. 重写 P1 分布式题的 `实战落地`：网关讲流量入口治理，舱壁讲资源隔离，熔断讲状态机和探测恢复，配置中心讲版本/灰度/回滚，缓存一致性讲失效通知和版本，时钟偏移讲 NTP/逻辑时钟/回拨处理。
3. 算法题保留现有基础解释，但把追问换成专项追问：双指针的有序性和重复元素，并查集的路径压缩、按秩合并、删除限制和典型题。
4. 清理重复或截断的 `图解提示`，保证节点是完整短句，避免 `hash(ke…` 这类内容进入后续 AI 生图。
5. 对 `distributed-clock-skew` 单独做事实精修，区分客户端本地时间、服务端时间、单调时钟和逻辑时钟，避免把所有过期机制都笼统归因到本地时钟偏移。
