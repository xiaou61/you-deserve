## 总体结论

这一批继续是 JVM 内存和 GC 排障专题。`jvm-memory-areas.md`、`oom-troubleshooting.md`、`strong-soft-weak-phantom-reference.md` 的正文相对扎实，能支撑一面/二面的基础回答。进阶题如 NMT、String Deduplication、TLAB、ZGC/Shenandoah 的核心方向正确，但“常见追问”和“图解提示”明显模板化：

- 10 篇都还有通用模板污染，尤其 `实战落地` 和 `追问准备` 继续使用“数据量或并发量扩大 10 倍”“灰度、压测和回滚策略”等泛化话术。
- 4 篇存在重复 `## 图解提示`：`jvm-native-memory-tracking.md`、`jvm-string-deduplication.md`、`jvm-tlab.md`、`jvm-zgc-shenandoah.md`。
- NMT、TLAB 等图解节点出现截断，如 `NativeMemor...`、`jcmd VM.native_...`、`TLAB...`，关键命令和概念被截断。
- 多个进阶题的 `常见追问` 是“这个点在项目里怎么落地？”并直接复用正文句子，缺少真正的面试深挖。

## 优先修复建议

1. 先合并 NMT、String Deduplication、TLAB、ZGC/Shenandoah 4 篇重复 `图解提示`，并补全 `NativeMemoryTracking`、`jcmd VM.native_memory`、`TLAB` 等截断节点。
2. 重写 4 篇进阶 JVM 题的 `常见追问`，重点补“怎么观测、怎么验证、什么时候不该用”。
3. `jvm-memory-areas.md` 建议微调方法区/元空间/Code Cache 的实现边界，避免把 HotSpot 的 Code Cache 与元空间混讲。
4. OOM、Metaspace、NMT 三题可以形成一组排障链路：错误类型 -> RSS/heap 区分 -> heap dump/NMT/classloader stats -> 修复验证。
5. Safepoint 题补日志和指标后会更像高阶面试答案，建议加入 `Time to safepoint` 和 `At safepoint` 两个排查维度。
