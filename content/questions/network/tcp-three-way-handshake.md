## 一句话结论

TCP 三次握手是为了确认客户端和服务端的发送、接收能力都正常，并同步初始序列号，避免历史连接请求造成错误连接。

## 通俗解释

建立 TCP 连接像两个人打电话确认信号：

1. 客户端说：你能听到我吗？
2. 服务端说：我能听到你，你能听到我吗？
3. 客户端说：我也能听到你。

到这里，双方才都确认“我能发、我能收、对方也能发、对方也能收”。

## 面试回答

三次握手过程是：

1. 客户端发送 SYN，进入 SYN_SENT。
2. 服务端收到后返回 SYN + ACK，进入 SYN_RCVD。
3. 客户端收到后发送 ACK，双方连接建立。

三次握手的核心目的有两个：

- 确认双方的发送和接收能力。
- 交换并确认初始序列号，保证后续可靠传输。

如果只有两次握手，服务端收到客户端的 SYN 并回复后，就可能认为连接已经建立。但客户端不一定收到了服务端的响应，服务端就会浪费资源。更糟的是，历史延迟的 SYN 抵达服务端时，也可能误建立连接。

## 常见追问

### 为什么不是四次？

因为服务端在第二次报文里就可以把“确认客户端的 SYN”和“发送自己的 SYN”合并在一起，三次已经足够完成双向收发能力确认和初始序列号同步。四次不是做不到，而是没有必要。继续展开时，可以顺手对比挥手为什么常常是四次，这样面试官会觉得你不是把两个过程割裂背的。

### 第三次握手可以携带数据吗？

理论上第三次 ACK 是可以携带数据的，因为到客户端视角，它已经确认了服务端的收发能力和序列号；但前两次一般不承载普通应用数据，因为连接还没正式建立。这个追问继续讲时，可以补上现实里的限制：即使协议允许，很多系统也不会在这里依赖首包业务数据去做复杂逻辑。

### 如果第三次 ACK 丢了，会发生什么？

客户端发完第三次 ACK 后会认为连接基本建立；如果这个 ACK 在网络中丢了，服务端还停留在半连接状态，会重发 SYN+ACK，客户端再补 ACK。也就是说，TCP 不会因为某一个 ACK 丢掉就直接彻底失败，而是靠重传机制补救。这个追问适合把“三次握手”顺手接到“可靠传输怎么兜底”。

### 线上如果大量连接卡在 SYN_SENT 或 SYN_RECV，通常先怀疑什么？

先怀疑网络和监听端，再看服务端 backlog、SYN flood、防火墙、负载均衡和连接数上限。`SYN_SENT` 多说明客户端请求发出去了但没拿到响应，`SYN_RECV` 多说明服务端收到了 SYN 但迟迟等不到最后 ACK。这个追问特别适合把协议知识接到排障场景，避免答案只停在教材图。

## 易错点

- 不要只说“为了安全”，要具体说确认收发能力和同步序列号。
- 不要把三次握手和四次挥手的状态混在一起。
- 不要忽视历史连接请求这个经典原因。

## 详细讲解

TCP 三次握手的目标不是形式上“打三次招呼”，而是在通信双方之间确认连接参数、同步初始序列号，并验证双方的发送和接收能力。TCP 是可靠字节流协议，后续每个字节都要靠序列号和确认号来保证顺序、重传和去重。因此建立连接前，双方必须知道彼此的初始序列号，不能直接开始发业务数据。

第一次握手是客户端发送 SYN，告诉服务端：我想建立连接，这是我的初始序列号 client_isn。此时客户端进入 SYN_SENT，服务端收到后知道客户端的发送能力和服务端自己的接收能力是通的。第二次握手是服务端回复 SYN+ACK，一方面确认客户端的序列号，另一方面发送自己的初始序列号 server_isn。客户端收到后，就知道服务端能接收自己的 SYN，也能发送响应回来。第三次握手是客户端再发 ACK，确认服务端的初始序列号。服务端收到这个 ACK 后，才知道自己的发送能力和客户端的接收能力也是通的。

为什么不是两次？如果只有两次，服务端发出 SYN+ACK 后就认为连接建立，但它无法确认客户端是否真的收到了自己的序列号。更经典的问题是历史重复连接请求：网络里可能有一个旧 SYN 因延迟很久才到达服务端。如果两次握手就建立连接，服务端可能为这个已经失效的请求分配资源，造成半开连接或资源浪费。第三次 ACK 可以让客户端有机会拒绝或确认这次连接，让服务端更可靠地进入 ESTABLISHED。

为什么不是四次？理论上双方都要发送 SYN 和 ACK，但服务端的 SYN 和对客户端 SYN 的 ACK 可以合并在一个报文里，所以三次就够了。挥手通常是四次，是因为关闭连接时双方的数据发送方向可以独立关闭，服务端收到 FIN 后可能还有数据没发完，ACK 和 FIN 往往不能立刻合并。

项目排查时，三次握手问题常表现为连接超时、SYN_SENT 堆积、SYN_RECV 堆积或服务端 accept 队列满。可能原因包括网络不通、防火墙拦截、服务端端口没监听、SYN backlog 不够、半连接攻击、连接数耗尽等。可以用 netstat、ss、tcpdump、服务端日志和系统指标一起看。面试里能把协议流程和排查现象连起来，会比只背 SYN/ACK 更有说服力。

图解建议画成时序图：左边客户端，右边服务端，中间三条箭头分别是 SYN、SYN+ACK、ACK。每条箭头旁边标出“我发给你的序列号”“我确认收到了你的序列号”。底部补两个结果：同步序列号、确认双向收发能力。

回答时再补一个工程视角：三次握手发生在应用层请求之前，所以接口慢不一定都是业务代码慢。DNS、连接建立、TLS 握手、连接池复用情况都会影响用户看到的耗时。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 TCP 为什么需要三次握手 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：TCP 三次握手是为了确认客户端和服务端的发送、接收能力都正常，并同步初始序列号，避免历史连接请求造成错误连接

拆开来看，第一层是背景问题：从双方收发能力确认和历史连接请求两个角度理解三次握手。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：客户端发送 SYN，进入 SYN_SENT。 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：服务端收到后返回 SYN + ACK，进入 SYN_RCVD。 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：客户端收到后发送 ACK，双方连接建立。 不要只背定义，要说出一次真实请求或一次系统调用如何经过这些环节。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：当线上出现 网络、TCP、三次握手、计网 相关的连接慢、请求超时、CPU 飙高、内存异常、文件句柄耗尽或网络抖动时，就可以从这道题切入。
- **怎么做方案**：按“客户端现象 -> 服务端日志 -> 系统指标 -> 协议状态 -> 内核资源”推进，先定位层次，再决定是代码、配置还是基础设施问题。
- **怎么验证效果**：用日志、抓包、连接状态、系统命令、监控指标和最小复现交叉验证。用抓包、日志、指标或系统命令把理论和排查动作连起来。
- **怎么兜底**：准备超时配置、限流、连接池隔离、熔断、扩容或回滚方案，避免排查期间影响继续扩大。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 TCP 为什么需要三次握手 讲成一次工程决策，而不是一个孤立知识点。重点围绕 协议状态、内核资源、时序变化和故障表现。

## 回答模板

面试时可以按这个节奏组织：

1. **先给结论**：从双方收发能力确认和历史连接请求两个角度理解三次握手。
2. **再讲机制**：它的核心不是某个名词，而是一组处理链路。把关键角色、状态变化和触发条件说清楚。
3. **补充边界**：说明什么情况下有效，什么情况下会失效，以及为什么需要配套措施。
4. **落到项目**：如果我在项目里遇到，会先看指标和日志定位问题，再用灰度、压测和回滚策略验证方案。
5. **收一句风险**：真正重要的是不要只让功能跑通，还要保证高并发、异常分支和数据状态都可控。
