## 一句话结论

HTTP 状态码按大类理解：2xx 成功，3xx 重定向，4xx 客户端问题，5xx 服务端问题。

## 通俗解释

状态码像快递结果。200 是送到了，301/302 是换地址了，404 是地址不存在，500 是快递站内部出问题。

## 面试回答

常见状态码：

- 200：请求成功。
- 201：创建成功。
- 204：成功但没有响应体。
- 301：永久重定向。
- 302：临时重定向。
- 400：请求参数错误。
- 401：未认证。
- 403：已认证但没权限。
- 404：资源不存在。
- 409：资源冲突。
- 500：服务端内部错误。
- 502/503/504：网关或服务不可用、超时相关。

接口设计里要让状态码和业务错误码配合，而不是所有结果都返回 200。

## 常见追问

### 401 和 403 区别？

401 是你还没证明你是谁；403 是知道你是谁，但你没权限访问。

### 业务失败要不要都返回 200？

不建议。参数错误、未授权、资源不存在等应该用合适 HTTP 状态码，业务错误码放在响应体里补充。

## 易错点

- 不要把 401 和 403 混淆。
- 不要所有接口都返回 200。
- 不要把 HTTP 状态码和业务状态码互相替代。

## 详细讲解

HTTP 状态码不是背数字题，它真正考的是你是否理解“服务端如何向客户端表达处理结果”。最基础的分类一定要知道：1xx 表示临时响应，2xx 表示成功，3xx 表示重定向或协商，4xx 表示客户端请求有问题，5xx 表示服务端处理失败。但面试里真正加分的，不是背出 200、404、500，而是能讲出常见状态码背后的语义边界。  

先看 2xx。200 是最常见的成功返回；201 常用于创建资源成功，比如创建订单、创建用户；204 表示成功但没有响应体，常见于删除成功或某些幂等更新。3xx 里，301 和 302 是重定向，304 则和缓存协商关系密切，表示资源未修改，客户端可以继续使用本地缓存。你如果能把 304 和缓存控制联系起来，状态码就不再是孤立知识点。  

4xx 的核心意思是“请求本身有问题”。400 往往表示参数错误或格式错误；401 表示未认证，通常是还没登录或 token 无效；403 表示已认证但无权限；404 表示资源不存在；405 表示方法不被允许；409 常见于资源冲突，比如重复创建；429 则用于限流场景。面试里经常有人把 401 和 403 混掉，这个点要特别稳。  

5xx 则表示服务端没把请求正确处理完。500 是通用内部错误；502、503、504 常出现在网关和代理链路里，分别偏向上游响应异常、服务暂时不可用、网关等待上游超时。如果你的项目里有 Nginx、Spring Cloud Gateway、服务网格，这几个状态码尤其值得结合网关链路去讲。  

工程上一个高频问题是“明明业务失败了，却总返回 200，再在 body 里写 code”。这种做法不是绝对不行，但会让客户端、网关、监控、缓存、重试策略失去基于协议语义做决策的能力。更成熟的做法是让 HTTP 状态码表达传输层和资源操作结果，再由业务码表达更细的业务语义。  

再补一个面试很爱问的点：状态码不是越细越好，而是要稳定、一致、团队能长期遵守。同一类错误今天返回 400、明天返回 200 包业务码、后天又改成 422，客户端和监控都会很痛苦。把状态码当成接口契约来维护，这句话很有工程味。  

如果想把答案和实际开发再贴近一点，还可以补一句“状态码会影响重试和告警策略”。比如 429 往往意味着限流后稍后重试，503 可能触发熔断或摘流，404 一般不该自动重试。也就是说，状态码不仅给人看，也是给网关、SDK 和运维系统看的。  

图解可以画成一棵状态码树，从 1xx 到 5xx 分层展开，再在 2xx、4xx、5xx 下挂典型场景：创建成功、未认证、无权限、资源不存在、限流、网关超时。最后收一句：选对状态码，本质是在给调用方、网关、监控系统提供准确语义，而不是单纯满足“接口能返回”。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 常见 HTTP 状态码怎么理解 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：HTTP 状态码按大类理解：2xx 成功，3xx 重定向，4xx 客户端问题，5xx 服务端问题

拆开来看，第一层是背景问题：按 2xx、3xx、4xx、5xx 分类理解接口返回状态。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：200：请求成功。 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：201：创建成功。 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：204：成功但没有响应体。 不要只背定义，要说出一次真实请求或一次系统调用如何经过这些环节。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：当线上出现 HTTP、状态码、网络、接口 相关的连接慢、请求超时、CPU 飙高、内存异常、文件句柄耗尽或网络抖动时，就可以从这道题切入。
- **怎么做方案**：按“客户端现象 -> 服务端日志 -> 系统指标 -> 协议状态 -> 内核资源”推进，先定位层次，再决定是代码、配置还是基础设施问题。
- **怎么验证效果**：用日志、抓包、连接状态、系统命令、监控指标和最小复现交叉验证。用抓包、日志、指标或系统命令把理论和排查动作连起来。
- **怎么兜底**：准备超时配置、限流、连接池隔离、熔断、扩容或回滚方案，避免排查期间影响继续扩大。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 常见 HTTP 状态码怎么理解 讲成一次工程决策，而不是一个孤立知识点。重点围绕 协议状态、内核资源、时序变化和故障表现。

## 回答模板

面试时可以按这个节奏组织：

1. **先给结论**：按 2xx、3xx、4xx、5xx 分类理解接口返回状态。
2. **再讲机制**：它的核心不是某个名词，而是一组处理链路。把关键角色、状态变化和触发条件说清楚。
3. **补充边界**：说明什么情况下有效，什么情况下会失效，以及为什么需要配套措施。
4. **落到项目**：如果我在项目里遇到，会先看指标和日志定位问题，再用灰度、压测和回滚策略验证方案。
5. **收一句风险**：真正重要的是不要只让功能跑通，还要保证高并发、异常分支和数据状态都可控。
