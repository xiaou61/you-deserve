## 一句话结论

GET 语义是获取资源，通常幂等、可缓存，参数常在 URL；POST 语义是提交处理，通常用于创建或触发操作，请求体承载数据。

## 通俗解释

GET 像问图书馆“有没有这本书”，POST 像提交一张申请表让系统处理。

## 面试回答

主要区别：

- 语义：GET 获取资源，POST 提交数据。
- 幂等：GET 应该幂等，POST 通常不保证幂等。
- 参数：GET 常放 URL query，POST 常放 body。
- 缓存：GET 更容易被浏览器和代理缓存。
- 长度：GET 受 URL 长度限制更明显。

安全性上，POST 不等于安全，只是参数不直接显示在 URL，仍然需要 HTTPS。

## 常见追问

### GET 能不能带 body？

协议层面没有绝对禁止，但兼容性和语义都不推荐，实际项目不要这样设计。

### POST 一定不幂等吗？

不一定。幂等性是接口语义和服务端实现决定的，比如带幂等键的 POST 也可以做到幂等。

## 易错点

- 不要说 POST 比 GET 天然安全。
- 不要只从参数位置区别。
- 不要忽略语义和幂等性。

## 详细讲解

GET 和 POST 的区别，核心应该从 HTTP 语义讲，而不是只说“GET 参数在 URL，POST 参数在 body”。GET 的语义是获取资源，应该是安全的、幂等的，也就是说一次请求和多次请求不应该改变服务器资源状态。POST 的语义更偏向提交数据、创建资源或触发处理，通常不承诺幂等，重复提交可能产生多条记录或多次业务动作。

参数位置只是常见表现。GET 请求通常把过滤条件、分页参数、查询关键字放在 URL query 里，方便分享、收藏、缓存和日志分析。POST 通常把复杂对象、表单、JSON 或文件放在 body 里，适合提交较大或结构化的数据。但协议并不是说 GET 绝对不能有 body，而是现实中很多中间件、网关、浏览器和服务框架对 GET body 支持不一致，所以工程上不建议依赖它。

缓存行为也不同。GET 因为语义上是读取资源，更容易被浏览器、CDN、代理缓存，配合 Cache-Control、ETag、Last-Modified 可以减少请求成本。POST 默认不容易被缓存，因为它可能改变状态。当然，是否缓存最终还是由响应头和中间件策略决定，不是方法名单独决定一切。

安全性是面试里的常见误区。有人说 POST 比 GET 安全，这是不准确的。GET 参数暴露在 URL、浏览器历史、服务器访问日志和 Referer 里，所以不适合放密码、token 等敏感信息；POST body 不会直接出现在地址栏，但如果没有 HTTPS，同样可能被窃听。真正的安全依赖 HTTPS、认证授权、CSRF 防护、输入校验和敏感字段脱敏。

接口设计时要按业务语义选。查询列表、获取详情适合 GET；创建订单、提交表单适合 POST；更新资源可用 PUT/PATCH；删除资源可用 DELETE。对于支付、下单、发券这类接口，即使用 POST，也要设计幂等键，防止用户重复点击、网关重试或网络超时导致重复处理。

图解可以画成两列：GET 一列标“读取、幂等、query、可缓存、适合查询”；POST 一列标“提交、可能改变状态、body、默认不缓存、适合创建或动作”。最后收束：GET 和 POST 的第一差异是语义，其次才是参数位置和缓存行为。

如果还要把答案讲得更像项目经验，可以用验证闭环收束：先说明这个问题在什么条件下出现，再说线上能从哪些信号确认，最后讲处理后怎么证明风险已经下降。对这道题来说，可以围绕“从语义、幂等性、参数位置和缓存理解 GET 与 POST”展开，再结合 抓包、连接状态、RTT、重传率、超时日志、网关日志、监控指标和最小压测 做交叉验证。这样回答不会停在概念层，而是把发现问题、定位原因、采取动作、验证结果串成一条完整链路。最后再补一句边界：方案要和业务容忍度匹配，不能只追求理论正确，还要考虑性能、复杂度、维护成本和失败后的恢复路径。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 GET 和 POST 有什么区别 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：GET 语义是获取资源，通常幂等、可缓存，参数常在 URL；POST 语义是提交处理，通常用于创建或触发操作，请求体承载数据

拆开来看，第一层是背景问题：从语义、幂等性、参数位置和缓存理解 GET 与 POST。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：语义：GET 获取资源，POST 提交数据。 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：幂等：GET 应该幂等，POST 通常不保证幂等。 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：参数：GET 常放 URL query，POST 常放 body。 不要只背定义，要说出一次真实请求或一次系统调用如何经过这些环节。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：当线上出现 HTTP、GET、POST、接口 相关的连接慢、请求超时、CPU 飙高、内存异常、文件句柄耗尽或网络抖动时，就可以从这道题切入。
- **怎么做方案**：按“客户端现象 -> 服务端日志 -> 系统指标 -> 协议状态 -> 内核资源”推进，先定位层次，再决定是代码、配置还是基础设施问题。
- **怎么验证效果**：用日志、抓包、连接状态、系统命令、监控指标和最小复现交叉验证。用抓包、日志、指标或系统命令把理论和排查动作连起来。
- **怎么兜底**：准备超时配置、限流、连接池隔离、熔断、扩容或回滚方案，避免排查期间影响继续扩大。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 GET 和 POST 有什么区别 讲成一次工程决策，而不是一个孤立知识点。重点围绕 协议状态、内核资源、时序变化和故障表现。

## 回答模板

面试时可以按这个节奏组织：

1. **先给结论**：从语义、幂等性、参数位置和缓存理解 GET 与 POST。
2. **再讲机制**：它的核心不是某个名词，而是一组处理链路。把关键角色、状态变化和触发条件说清楚。
3. **补充边界**：说明什么情况下有效，什么情况下会失效，以及为什么需要配套措施。
4. **落到项目**：如果我在项目里遇到，会先看指标和日志定位问题，再用灰度、压测和回滚策略验证方案。
