## 一句话结论

客户端通过 Range 头声明需要的字节范围，服务端返回 206 Partial Content 和 Content-Range。

## 通俗解释

Range 请求像只复印一本书的某几页，不用每次复印整本。

## 面试回答

可以从这几层回答：

- 请求头使用 Range: bytes=start-end。
- 服务端支持时返回 206，而不是 200 全量内容。
- Content-Range 表示返回范围和总大小。
- 文件下载、视频播放和大对象读取都常用它。

回答时最好补一句：这个点不是孤立知识点，真正落地时要结合业务场景、数据规模和失败兜底，说明你不是只背概念。

## 常见追问

### 这个点在项目里怎么落地？

服务端支持时返回 206，而不是 200 全量内容。

### 面试官继续追问时怎么展开？

Content-Range 表示返回范围和总大小。

## 易错点

- 服务端不支持 Range 却期待断点续传生效。
- CDN/代理层没有正确透传范围响应。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 HTTP Range 请求有什么用 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：客户端通过 Range 头声明需要的字节范围，服务端返回 206 Partial Content 和 Content-Range

拆开来看，第一层是背景问题：Range 请求允许客户端只请求资源的一部分，常用于断点续传和视频拖拽。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：请求头使用 Range: bytes=start-end。 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：服务端支持时返回 206，而不是 200 全量内容。 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：Content-Range 表示返回范围和总大小。 不要只背定义，要说出一次真实请求或一次系统调用如何经过这些环节。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：当线上出现 HTTP、Range、断点续传 相关的连接慢、请求超时、CPU 飙高、内存异常、文件句柄耗尽或网络抖动时，就可以从这道题切入。
- **怎么做方案**：按“客户端现象 -> 服务端日志 -> 系统指标 -> 协议状态 -> 内核资源”推进，先定位层次，再决定是代码、配置还是基础设施问题。
- **怎么验证效果**：用日志、抓包、连接状态、系统命令、监控指标和最小复现交叉验证。用抓包、日志、指标或系统命令把理论和排查动作连起来。
- **怎么兜底**：准备超时配置、限流、连接池隔离、熔断、扩容或回滚方案，避免排查期间影响继续扩大。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 HTTP Range 请求有什么用 讲成一次工程决策，而不是一个孤立知识点。重点围绕 协议状态、内核资源、时序变化和故障表现。
