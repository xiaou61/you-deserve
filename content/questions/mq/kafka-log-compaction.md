## 一句话结论

Kafka 不只按时间删除日志，也可以对同 key 的多版本消息做压缩，保留最终状态。

## 通俗解释

Log Compaction 像通讯录变更记录，最后只关心每个人当前最新手机号。

## 面试回答

可以从这几层回答：

- 它按 key 清理旧值，保留较新记录。
- 适合配置、用户状态、维表变更等场景。
- 删除语义通常通过 tombstone 记录表达。
- 它不是实时去重，后台清理有延迟。

回答时最好补一句：这个点不是孤立知识点，真正落地时要结合业务场景、数据规模和失败兜底，说明你不是只背概念。

## 常见追问

### 这题最容易被追问的边界是什么？

重点说清重复投递、顺序错乱、消息积压、事务边界、重试风暴和毒丸消息。不要只给结论，要说明哪些条件下结论成立，哪些条件下会退化或需要换方案。

### 怎么证明自己不是在背模板？

用消费延迟、offset、重试次数、死信量、事务状态、幂等表和最终数据状态做验证。能说出具体信号、反例或排查入口，答案就从概念层进入实战层。

### 和相近方案怎么区分？

可以拿同步调用、异步消息、Outbox、事务消息、延迟消息或补偿任务对比，从可靠性、顺序性、延迟、吞吐、幂等成本和补偿成本几个维度选择。

### 面试官继续深挖时怎么展开？

先围绕“它按 key 清理旧值，保留较新记录。”讲清主链路，再补“补 compaction 不保证消费者只能看到最新值、后台清理延迟、cleanup.policy=compact、min.cleanable.dirty.ratio、delete.retention.ms、key 为空无法压缩、状态恢复场景”。如果能把边界条件、异常分支和验证闭环连起来，就能接住二面、三面的追问。

## 易错点

- 以为 compaction 能保证消费者只看到最新值。
- 没有 key 的消息无法按业务实体压缩。

## 详细讲解

Kafka Log Compaction 是什么？ 这道题现在要从“能背出来”修到“能接住追问”。核心结论是：Log Compaction 会按 key 保留较新的消息，适合保存状态变更日志。回答时先把它放回消息队列语境，说明它解决什么矛盾，再围绕生产发送、Broker 存储、消费确认、重试死信和幂等补偿展开。这样能避免泛泛而谈，也能让面试官听出你知道这题的真实边界。

第一步是拆清问题背景。Log Compaction 像通讯录变更记录，最后只关心每个人当前最新手机号。在消息队列题里，背景不能写成空泛的工程套话，而要落到按生产者、Broker、消费者、业务数据库和补偿任务拆链路。如果只说“看场景”或“加组件”，读者很难知道下一步该检查什么；如果能把输入、状态、依赖和输出说清，答案就有了主线。

第二步是讲机制。可以围绕这些点展开：它按 key 清理旧值，保留较新记录。；适合配置、用户状态、维表变更等场景。；删除语义通常通过 tombstone 记录表达。；它不是实时去重，后台清理有延迟。。每个点都要回答三个问题：它在链路里的位置是什么，它改变了什么状态，失败或边界条件下会留下什么现象。把这三件事讲清楚，追问从概念跳到实战时就不会断。

第三步是补边界。这里尤其要主动说明重复投递、顺序错乱、消息积压、事务边界、重试风暴和毒丸消息。以为 compaction 能保证消费者只看到最新值。；没有 key 的消息无法按业务实体压缩。很多八股答案的问题不是方向错，而是没有说适用范围；一旦遇到数据规模变化、异常分支或版本差异，原来的结论就可能不成立。

第四步是补专项深度：补 compaction 不保证消费者只能看到最新值、后台清理延迟、cleanup.policy=compact、min.cleanable.dirty.ratio、delete.retention.ms、key 为空无法压缩、状态恢复场景。这部分适合放在面试回答后半段，用来证明你不仅知道标准答案，也知道高频追问会往哪里挖。

最后收束成一套回答节奏：先给结论，再讲机制，再补边界，最后说验证方式。对这题来说，验证可以看消费延迟、offset、重试次数、死信量、事务状态、幂等表和最终数据状态；方案选择可以和同步调用、异步消息、Outbox、事务消息、延迟消息或补偿任务对比。这样既保留八股题的清晰度，也能把答案讲成可落地、可排查、可复盘的经验。

如果面试官继续追问，可以再补一层取舍：为什么不用另一个方案、什么时候会退化、怎么发现反例或异常、出问题后如何止血。消息队列题最怕只有名词没有链路，补上生产发送、Broker 存储、消费确认、重试死信和幂等补偿之后，答案会更稳。
