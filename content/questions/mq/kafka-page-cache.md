## 一句话结论

Kafka 自身不把所有消息缓存到 JVM 堆，而是把日志文件交给 OS 缓存，减少 GC 压力并利用顺序读写。

## 通俗解释

Page Cache 像仓库门口的临时货架，热点货物不用每次进库房翻。

## 面试回答

可以从这几层回答：

- 消息追加写入日志文件，顺序 IO 友好。
- 热点数据由操作系统 Page Cache 缓存。
- 消费者读取时可能直接命中缓存。
- sendfile 等零拷贝路径能减少用户态和内核态拷贝。

回答时最好补一句：这个点不是孤立知识点，真正落地时要结合业务场景、数据规模和失败兜底，说明你不是只背概念。

## 常见追问

### 这个点在项目里怎么落地？

## 易错点

- 给 Kafka 堆内存盲目调很大。
- 忽略磁盘顺序 IO 和 Page Cache 命中率。

## 实战落地

- **什么时候会遇到**：当业务里出现 Kafka、Page Cache、高吞吐 相关的性能、并发、一致性或可维护性问题时，就可以把这道题里的知识点拿出来判断。不要等线上出故障才想概念，设计阶段就要先问容量、失败分支和边界条件。
- **怎么做方案**：先明确目标，再拆链路。围绕“输入条件、核心处理、状态变化、异常分支、观测告警”五个位置检查一遍，判断 Kafka 为什么依赖 Page Cache 应该放在哪个环节解决问题。
- **怎么验证效果**：不要只看功能能不能跑通，还要看压测指标、错误日志、慢操作、重试次数、队列积压、锁等待或资源占用等信号。结合生产问题说明如何使用、如何验证、如何兜底。
- **怎么兜底**：准备降级、限流、重试、补偿任务、人工修复脚本或回滚策略。面试里能讲出兜底，说明你不是只会写 happy path。
