## 一句话结论

Kafka 生产者选择分区通常按优先级：指定 partition、根据 key 哈希、默认分区器分配；同一个 key 通常会进入同一分区。

## 通俗解释

它像快递分拣。你指定柜台就去指定柜台；没指定但有收件城市，就按城市分；都没有就按规则均匀分。

## 面试回答

常见分区选择逻辑：

- 如果消息指定了 partition，就直接发到该分区。
- 如果有 key，通常根据 key 哈希选择分区。
- 如果没有 key，分区器会尽量在可用分区间分摊流量。

同 key 进入同一分区，可以保证同 key 消息的顺序消费，比如同一个订单的状态变更。但如果 key 分布不均，会造成热点分区。

## 常见追问

### Kafka 能保证全局顺序吗？

单分区内有序，多分区不保证全局有序。如果要全局有序只能用一个分区，但吞吐会受限。

### 分区数变更会影响什么？

按 key 哈希的分区结果可能变化，顺序性和数据分布都要评估。

## 易错点

- 不要说 Kafka 天然全局有序。
- key 设计不均会造成热点分区。
- 分区数不是越多越好，会增加管理和 rebalance 成本。

## 记忆钩子

**指定优先，其次看 key；顺序在分区内。**
