## 一句话结论

Kafka 生产者选择分区通常按优先级：指定 partition、根据 key 哈希、默认分区器分配；同一个 key 通常会进入同一分区。

## 通俗解释

它像快递分拣。你指定柜台就去指定柜台；没指定但有收件城市，就按城市分；都没有就按规则均匀分。

## 面试回答

常见分区选择逻辑：

- 如果消息指定了 partition，就直接发到该分区。
- 如果有 key，通常根据 key 哈希选择分区。
- 如果没有 key，分区器会尽量在可用分区间分摊流量。

同 key 进入同一分区，可以保证同 key 消息的顺序消费，比如同一个订单的状态变更。但如果 key 分布不均，会造成热点分区。

## 常见追问

### 这题最容易被追问的边界是什么？

重点说清重复投递、顺序错乱、消息积压、事务边界、重试风暴和毒丸消息。不要只给结论，要说明哪些条件下结论成立，哪些条件下会退化或需要换方案。

### 怎么证明自己不是在背模板？

用消费延迟、offset、重试次数、死信量、事务状态、幂等表和最终数据状态做验证。能说出具体信号、反例或排查入口，答案就从概念层进入实战层。

### 和相近方案怎么区分？

可以拿同步调用、异步消息、Outbox、事务消息、延迟消息或补偿任务对比，从可靠性、顺序性、延迟、吞吐、幂等成本和补偿成本几个维度选择。

### 面试官继续深挖时怎么展开？

先围绕“如果消息指定了 partition，就直接发到该分区。”讲清主链路，再补“补 sticky partitioner/默认分区器版本差异、分区数变化导致 key 映射变化、不可用分区选择、key 设计如何兼顾顺序和热点、扩分区后顺序风险”。如果能把边界条件、异常分支和验证闭环连起来，就能接住二面、三面的追问。

## 易错点

- 不要说 Kafka 天然全局有序。
- key 设计不均会造成热点分区。
- 分区数不是越多越好，会增加管理和 rebalance 成本。

第二步是讲机制。可以围绕这些点展开：如果消息指定了 partition，就直接发到该分区。；如果有 key，通常根据 key 哈希选择分区。；如果没有 key，分区器会尽量在可用分区间分摊流量。；同 key 进入同一分区，可以保证同 key 消息的顺序消费，比如同一个订单的状态变更。但如果 key 分布不均，会造成热点分区。。每个点都要回答三个问题：它在链路里的位置是什么，它改变了什么状态，失败或边界条件下会留下什么现象。把这三件事讲清楚，追问从概念跳到实战时就不会断。
