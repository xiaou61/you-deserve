## 一句话结论

缓存穿透是查不存在的数据，缓存击穿是热点 key 失效，缓存雪崩是大量 key 同时失效或 Redis 整体不可用。它们的共同结果都是请求打到数据库，但原因不一样。

## 通俗解释

把缓存想成数据库门口的前台：

- 穿透：有人一直问“根本不存在的人住哪”，前台查不到，只能每次都问后面的数据库。
- 击穿：所有人都问同一个大明星，刚好这位大明星的信息过期了，瞬间全挤到数据库。
- 雪崩：前台大片资料同一时间过期，或者前台直接停工，所有请求都冲向数据库。

## 面试回答

缓存穿透通常是请求查询不存在的数据，缓存和数据库都没有结果。如果不处理，每次请求都会绕过缓存打到数据库。常见方案是缓存空值、布隆过滤器、参数校验。

缓存击穿是某个热点 key 在高并发下突然过期，大量请求同时访问数据库。常见方案是互斥锁、逻辑过期、热点数据不过期。

缓存雪崩是大量 key 同一时间过期，或者 Redis 集群不可用，导致大量请求直接打到数据库。常见方案是过期时间加随机值、缓存预热、多级缓存、限流降级、Redis 高可用。

## 常见追问

### 缓存空值有什么问题？

空值也会占用缓存空间，而且如果真实数据后来创建了，可能出现短时间不一致。所以空值缓存一般设置较短过期时间。

### 布隆过滤器有什么缺点？

布隆过滤器可能误判存在，但不会误判不存在。它适合挡掉明显不存在的请求，但不能替代数据库查询。

## 易错点

- 不要把击穿和雪崩混在一起。击穿通常是单个热点 key，雪崩是大面积问题。
- 不要只说“加锁”，还要考虑锁粒度、超时和兜底。
- 不要忽视限流和降级，缓存事故本质上是系统保护问题。

## 详细讲解

缓存穿透、击穿、雪崩都是缓存系统里的高频故障，但三者的触发点不一样。缓存穿透是请求的数据根本不存在，缓存里查不到，数据库里也查不到。如果攻击者不断请求随机 ID，每次都绕过缓存打到数据库，缓存就失去了保护作用。解决思路是把“不存在”也缓存起来，比如缓存空值并设置较短 TTL；或者用布隆过滤器在入口处判断这个 key 是否可能存在，不可能存在的请求直接拦截。

缓存击穿强调一个热点 key。这个 key 平时访问量很大，缓存命中时没问题，但它突然过期了，大量请求同时发现缓存没有，于是一起去查数据库并重建缓存，数据库瞬间被打爆。解决思路是保护重建过程：可以用互斥锁，只允许一个线程查库并回填缓存，其他线程等待或返回旧值；也可以热点 key 不设置普通过期，而是逻辑过期，后台异步刷新；还可以提前预热和续期，避免热点 key 在流量高峰自然过期。

缓存雪崩是更大范围的问题。大量 key 在同一时间过期，或者 Redis 集群故障，导致大批请求同时落到数据库。它和击穿的区别是规模：击穿是一个或少数热点 key，雪崩是一大片 key 或整个缓存层。解决方式包括过期时间加随机抖动，避免同一批 key 集中过期；热点数据提前预热；Redis 做高可用；服务端限流降级；数据库侧保护连接池和慢查询；必要时返回兜底数据。

面试里不要只背三句话，要能按“请求路径”讲。正常路径是请求先查缓存，命中直接返回；未命中再查数据库，查到后回填缓存。穿透的问题发生在“数据库也查不到”；击穿的问题发生在“热点 key 未命中后的回源重建”；雪崩的问题发生在“大量 key 或整个缓存层同时不可用”。沿着这条路径讲，三者区别会非常清楚。

项目里要补充监控指标。穿透会表现为大量不存在 key、缓存命中率下降、数据库 QPS 异常升高；击穿会表现为某个热点 key 过期点附近数据库压力尖刺；雪崩会表现为整体命中率骤降、数据库连接池耗尽、接口 P99 飙升。治理时不能只靠一个方案，比如布隆过滤器有误判，空值缓存要控制 TTL 和内存，互斥锁要防死锁和超时，逻辑过期要接受短暂旧数据。

图解可以画成三列对比：左列穿透，画“非法 key -> 缓存 miss -> DB miss -> 空值/布隆拦截”；中列击穿，画“热点 key 过期 -> 大量请求 -> 互斥重建”；右列雪崩，画“大量 key 同时过期 -> 请求洪峰 -> 随机 TTL/限流/降级”。最后统一落到一句话：缓存不是只提升速度，更重要是保护数据库。

如果要联系项目，可以补一句：缓存治理一定要配合监控和开关。没有命中率、热点 key、数据库 QPS 和降级开关，方案再漂亮也很难在线上稳定运行。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 缓存穿透、击穿、雪崩有什么区别 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：缓存穿透是查不存在的数据，缓存击穿是热点 key 失效，缓存雪崩是大量 key 同时失效或 Redis 整体不可用

拆开来看，第一层是背景问题：用请求如何打到数据库来区分三类缓存问题。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：缓存穿透通常是请求查询不存在的数据，缓存和数据库都没有结果 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：如果不处理，每次请求都会绕过缓存打到数据库 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：常见方案是缓存空值、布隆过滤器、参数校验 不要把数据库或缓存讲成万能组件，要说明适用范围和兜底手段。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：当业务里出现 Redis、缓存、高并发、系统设计 相关的慢查询、锁等待、热点 key、缓存不一致、内存上涨或写入抖动时，就要把这道题放到读写链路里分析。
- **怎么做方案**：先明确数据规模和访问模式，再选择索引、事务、缓存、过期策略或分片策略。围绕“读路径、写路径、并发控制、失败补偿、容量上限”五个点展开。
- **怎么验证效果**：看 explain、慢日志、锁等待、命中率、内存、QPS、P99 延迟和错误率。把答案落到慢查询、热点 key、事务隔离、缓存一致性或容量评估上。
- **怎么兜底**：准备降级读、限流、重建索引、缓存预热、补偿任务或数据修复脚本，避免单点方案拖垮主链路。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 缓存穿透、击穿、雪崩有什么区别 讲成一次工程决策，而不是一个孤立知识点。重点围绕 数据结构、读写路径、一致性边界和性能影响。

## 回答模板

面试时可以按这个节奏组织：

1. **先给结论**：用请求如何打到数据库来区分三类缓存问题。
2. **再讲机制**：它的核心不是某个名词，而是一组处理链路。把关键角色、状态变化和触发条件说清楚。
3. **补充边界**：说明什么情况下有效，什么情况下会失效，以及为什么需要配套措施。
4. **落到项目**：如果我在项目里遇到，会先看指标和日志定位问题，再用灰度、压测和回滚策略验证方案。
5. **收一句风险**：真正重要的是不要只让功能跑通，还要保证高并发、异常分支和数据状态都可控。
