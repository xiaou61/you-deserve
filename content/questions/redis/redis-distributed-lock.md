## 一句话结论

Redis 分布式锁的基础实现是 `SET key value NX EX seconds`，加锁要带过期时间和唯一标识，释放锁要用 Lua 脚本保证“判断是自己的锁”和“删除锁”是原子操作。

## 通俗解释

分布式锁像公共会议室门口的一张占用牌。谁先挂上牌，谁就能用会议室。牌子上要写自己的名字，还要写自动失效时间，避免人走了牌子永远挂着。

释放时也不能看到有牌子就摘，必须确认牌子是自己的，不然可能把别人的锁删掉。

## 面试回答

基础加锁命令一般是：

```text
SET lock_key request_id NX EX 30
```

含义是：只有 key 不存在时才设置成功，并设置过期时间。`request_id` 是当前线程或请求的唯一标识。

释放锁时不能简单 `DEL key`，因为业务执行时间可能超过锁过期时间，锁已经被别人重新获取。正确做法是用 Lua 脚本先判断 value 是否等于自己的 request_id，如果是再删除，保证判断和删除的原子性。

更完整的方案还要考虑：

- 锁过期时间如何设置。
- 业务没执行完时是否自动续期。
- Redis 主从切换导致锁丢失的风险。
- 是否使用 Redisson 这类成熟组件。

## 常见追问

### 为什么要设置过期时间？

防止拿到锁的服务宕机后无法释放锁，导致其他请求永远拿不到锁。

### 为什么释放锁要用 Lua？

因为“判断 value 是不是自己”和“删除 key”必须是原子操作。分开执行可能在中间发生锁过期和其他线程加锁，导致误删别人的锁。

## 易错点

- 不要用 `setnx` 后再单独 `expire`，两步之间宕机会留下死锁。
- 不要释放锁时直接 `del`。
- 不要忽略锁续期和 Redis 高可用场景。

## 详细讲解

Redis 分布式锁的基本目标，是让多个进程或多台机器在访问同一份共享资源时，同一时刻只有一个持有者进入临界区。最基础的加锁命令应该用 SET key value NX EX seconds，一条命令同时完成“仅不存在时写入”和“设置过期时间”。不要用 setnx 后再 expire 两条命令，因为中间如果进程宕机，锁可能没有过期时间，形成死锁。

value 一定要是唯一标识，比如 UUID、请求 ID 或线程标识。原因是锁可能过期后被别人重新获取，如果旧持有者业务执行完直接 del key，就会误删新持有者的锁。因此释放锁必须先判断 value 是否还是自己的，再删除。这个判断和删除也要保持原子，通常用 Lua 脚本完成：if get(key) == value then del(key) end。面试里如果能说出唯一标识和 Lua 解锁，基本就避开了最危险的坑。

过期时间是分布式锁最难的取舍。设置太短，业务没执行完锁就过期，其他线程会进入临界区，导致并发问题；设置太长，持有者宕机后其他请求要等很久。可以通过压测和业务耗时设置合理 TTL，也可以使用看门狗机制自动续期，比如 Redisson 在业务线程仍然存活时定期延长锁时间。但续期不是万能的，如果业务卡死、GC 停顿、网络抖动，都可能影响锁的可靠性。

还要注意锁保护的资源是否真的具备幂等和兜底。分布式锁只能减少并发进入，不应该作为唯一一致性保障。比如扣库存、发优惠券、创建订单，还要有数据库唯一索引、状态机、乐观锁或幂等表兜底。否则锁一旦失效，就可能造成超卖或重复处理。面试里说“锁只是第一层保护，数据库约束是最后防线”，会更像生产答案。

Redis 主从切换也会带来边界问题。单实例 Redis 持锁后，如果还没同步到从库主节点就宕机，从库提升为主，另一个客户端可能重新拿到锁，出现两个持有者。RedLock 试图通过多个独立 Redis 节点投票提高可靠性，但实现复杂，也有争议。大多数业务要根据一致性要求选择：普通防重复可以用 Redis 锁加数据库兜底；强一致资源竞争可能更适合数据库锁、ZooKeeper、etcd 或业务状态机。

图解可以画成加锁流程：请求进入后执行 SET NX EX，成功则带唯一 value 执行业务，结束时 Lua 比对并删除；失败则等待、重试或快速失败。旁边画风险分支：锁过期、误删他人锁、业务超时、Redis 故障，并对应写上 TTL、唯一标识、Lua、续期和数据库兜底。

回答时还可以补充“锁粒度”。锁粒度太粗会降低并发，太细又可能保护不住共享资源。比如按商品维度锁库存，比全局锁更合理；但如果业务操作跨多个资源，还要考虑死锁顺序和超时释放。

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 Redis 分布式锁怎么实现 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：Redis 分布式锁的基础实现是 SET key value NX EX seconds，加锁要带过期时间和唯一标识，释放锁要用 Lua 脚本保证“判断是自己的锁”和“删除锁”是原子操作

拆开来看，第一层是背景问题：掌握 set nx ex、唯一标识、Lua 删除和锁续期这几个关键点。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：锁过期时间如何设置。 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：业务没执行完时是否自动续期。 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：Redis 主从切换导致锁丢失的风险。 不要把数据库或缓存讲成万能组件，要说明适用范围和兜底手段。这样回答会比单纯背八股更像做过项目的人。
