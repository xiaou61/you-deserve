## 一句话结论

过期删除处理的是 key 到期后怎么删；内存淘汰处理的是 Redis 内存达到上限后删哪些 key。它们不是一回事。

## 通俗解释

过期删除像食品到了保质期要下架。内存淘汰像货架满了，必须挑一些商品拿走给新商品腾位置。

一个看时间，一个看空间。

## 面试回答

Redis 过期删除主要有：

- 惰性删除：访问 key 时发现过期再删除。
- 定期删除：后台定期抽样检查并删除过期 key。

只靠惰性删除可能导致过期 key 长期占内存，只靠定期删除又可能消耗太多 CPU，所以 Redis 结合两者。

内存淘汰是在达到 maxmemory 后，根据策略选择 key 淘汰，比如 LRU、LFU、随机淘汰、只淘汰设置过期时间的 key 等。

## 常见追问

### Redis 为什么不扫描所有过期 key？

全量扫描成本太高，会影响主线程处理请求，所以采用定期抽样 + 惰性删除的折中策略。

### LRU 和 LFU 区别？

LRU 看最近是否使用，LFU 看使用频率。热点数据场景下 LFU 可能更合理。

## 易错点

- 不要把过期删除和内存淘汰混为一谈。
- 不要说 Redis 到期一定立刻删除。
- 不要忽略 maxmemory 配置。

## 详细讲解

Redis 过期删除和内存淘汰经常被混成一件事，但它们解决的问题完全不同。过期删除关注的是“这个 key 的生存时间到了没有”；内存淘汰关注的是“Redis 可用内存不够了，应该把谁赶走”。一个是基于时间语义，一个是基于容量压力。面试里如果能先把这两个问题拆开，后面就会顺很多。  

先说过期删除。给 key 设置了 TTL，并不代表一到那个时刻 Redis 就一定立刻把它删掉。Redis 主要结合惰性删除和定期删除。惰性删除是在访问 key 时发现它过期了，顺手删掉；定期删除则是后台周期性抽样扫描一部分设置了过期时间的 key，把其中已过期的清掉。这样设计是为了避免“为了一到点立刻删掉所有 key”带来过高 CPU 成本。代价是：逻辑上已经过期的数据，物理上可能会短暂残留。  

再说内存淘汰。它发生在配置了 maxmemory 且当前内存达到上限之后。此时 Redis 不是看某个 key 是否到期，而是按照策略决定淘汰谁。常见策略有 noeviction、allkeys-lru、volatile-lru、allkeys-lfu、volatile-ttl、allkeys-random 等。这里的 allkeys 和 volatile 就体现了淘汰范围差异：前者从所有 key 里选，后者只从设置了过期时间的 key 里选。  

很多故障都来自把这两套机制想当然地等同起来。比如有人觉得“我给所有缓存都设了 TTL，就一定不会顶满内存”，其实未必，因为过期 key 可能还没被及时清理；也有人觉得“用了 LRU 淘汰就不用考虑 TTL 了”，也不对，因为 TTL 还承担业务语义，比如缓存多久失效、多久需要回源刷新。TTL 决定的是数据新鲜度，淘汰策略决定的是内存压力下的生存优先级。  

工程上设计 Redis 缓存时，TTL 和 maxmemory 策略必须一起看。热点缓存、用户会话、排行榜、延迟队列、持久键值，它们对 TTL 和淘汰容忍度完全不同。比如做纯缓存层时，allkeys-lru 或 allkeys-lfu 常见；如果 Redis 里混放了不能随便丢的业务数据，就要格外谨慎，甚至不该和缓存共用实例。  

再补一个线上经验：当你发现命中率下降、内存频繁打满、慢查询或回源流量陡增时，要同时检查 TTL 分布和淘汰策略统计，不能只盯一种机制。很多缓存故障其实是“过期设计不合理”和“淘汰策略不匹配”叠加出来的。  

图解可以画成两条分支：第一条是 key 设置 TTL 后，经过惰性删除和定期删除处理；第二条是 Redis 内存接近上限，触发 maxmemory 策略，从候选 key 中做淘汰。旁边再标一句：过期不等于立删，淘汰不等于过期。这样就能把很多面试里常见的混淆点一次讲明白。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 Redis 过期删除和内存淘汰有什么区别 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：过期删除处理的是 key 到期后怎么删；内存淘汰处理的是 Redis 内存达到上限后删哪些 key

拆开来看，第一层是背景问题：区分 key 到期后的删除策略和内存不足时的淘汰策略。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：Redis 过期删除主要有： - 惰性删除：访问 key 时发现过期再删除 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：- 定期删除：后台定期抽样检查并删除过期 key 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：只靠惰性删除可能导致过期 key 长期占内存，只靠定期删除又可能消耗太多 CPU，所以 Redis 结合两者 不要把数据库或缓存讲成万能组件，要说明适用范围和兜底手段。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：当业务里出现 Redis、过期删除、内存淘汰、缓存 相关的慢查询、锁等待、热点 key、缓存不一致、内存上涨或写入抖动时，就要把这道题放到读写链路里分析。
- **怎么做方案**：先明确数据规模和访问模式，再选择索引、事务、缓存、过期策略或分片策略。围绕“读路径、写路径、并发控制、失败补偿、容量上限”五个点展开。
- **怎么验证效果**：看 explain、慢日志、锁等待、命中率、内存、QPS、P99 延迟和错误率。把答案落到慢查询、热点 key、事务隔离、缓存一致性或容量评估上。
- **怎么兜底**：准备降级读、限流、重建索引、缓存预热、补偿任务或数据修复脚本，避免单点方案拖垮主链路。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 Redis 过期删除和内存淘汰有什么区别 讲成一次工程决策，而不是一个孤立知识点。重点围绕 数据结构、读写路径、一致性边界和性能影响。
