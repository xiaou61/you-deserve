## 一句话结论

缓存击穿是热点 key 失效瞬间大量请求同时打到数据库，互斥锁方案让只有一个请求去查库并重建缓存。

## 通俗解释

它像食堂热门菜刚卖完，所有人都冲进后厨催。互斥锁就是只让一个人去催菜，其他人在窗口等一会。

## 面试回答

处理流程：

1. 请求查询缓存未命中。
2. 尝试获取互斥锁。
3. 获取成功的请求查数据库并写回缓存。
4. 获取失败的请求短暂等待后重试缓存。
5. 如果数据库也没有数据，要缓存空值防穿透。

互斥锁可以用 Redis `SET NX EX` 实现，但要设置过期时间，避免锁不释放。热点 key 还可以配合逻辑过期、永不过期加异步刷新等方案。

## 常见追问

### 这题最容易被追问的边界是什么？

重点说清大 key、热 key、缓存击穿、主从切换、Cluster 跨槽、Lua 阻塞和内存碎片。不要只给结论，要说明哪些条件下结论成立，哪些条件下会退化或需要换方案。

### 怎么证明自己不是在背模板？

用INFO、SLOWLOG、MEMORY USAGE、OBJECT ENCODING、CLIENT LIST、命中率和延迟做验证。能说出具体信号、反例或排查入口，答案就从概念层进入实战层。

### 和相近方案怎么区分？

可以拿String、Hash、ZSet、Lua、Pipeline、Stream、Cluster、哨兵或本地缓存对比，从读写性能、一致性、内存成本、可用性和运维复杂度几个维度选择。

### 面试官继续深挖时怎么展开？

先围绕“请求查询缓存未命中。”讲清主链路，再补“补双重检查、锁过期时间、锁续期、获取锁失败的等待/降级、逻辑过期返回旧值、热点 key 永不过期 + 异步刷新、空值缓存和数据库保护”。如果能把边界条件、异常分支和验证闭环连起来，就能接住二面、三面的追问。

## 易错点

- 锁必须有过期时间。
- 等待请求不要无限阻塞。
- 数据库空结果也要处理。

## 详细讲解

缓存击穿为什么常用互斥锁解决？ 这道题现在要从“能背出来”修到“能接住追问”。核心结论是：热点 key 过期瞬间大量请求回源会打爆数据库，互斥锁让只有一个请求重建缓存。回答时先把它放回Redis语境，说明它解决什么矛盾，再围绕命令语义、key 设计、过期策略、内存指标和客户端行为展开。这样能避免泛泛而谈，也能让面试官听出你知道这题的真实边界。

第一步是拆清问题背景。它像食堂热门菜刚卖完，所有人都冲进后厨催。互斥锁就是只让一个人去催菜，其他人在窗口等一会。在Redis题里，背景不能写成空泛的工程套话，而要落到从数据结构、命令原子性、过期淘汰、集群路由或客户端重试解释。如果只说“看场景”或“加组件”，读者很难知道下一步该检查什么；如果能把输入、状态、依赖和输出说清，答案就有了主线。

第二步是讲机制。可以围绕这些点展开：请求查询缓存未命中。；尝试获取互斥锁。；获取成功的请求查数据库并写回缓存。；获取失败的请求短暂等待后重试缓存。。每个点都要回答三个问题：它在链路里的位置是什么，它改变了什么状态，失败或边界条件下会留下什么现象。把这三件事讲清楚，追问从概念跳到实战时就不会断。

第三步是补边界。这里尤其要主动说明大 key、热 key、缓存击穿、主从切换、Cluster 跨槽、Lua 阻塞和内存碎片。锁必须有过期时间。；等待请求不要无限阻塞。；数据库空结果也要处理。很多八股答案的问题不是方向错，而是没有说适用范围；一旦遇到数据规模变化、异常分支或版本差异，原来的结论就可能不成立。

第四步是补专项深度：补双重检查、锁过期时间、锁续期、获取锁失败的等待/降级、逻辑过期返回旧值、热点 key 永不过期 + 异步刷新、空值缓存和数据库保护。这部分适合放在面试回答后半段，用来证明你不仅知道标准答案，也知道高频追问会往哪里挖。

最后收束成一套回答节奏：先给结论，再讲机制，再补边界，最后说验证方式。对这题来说，验证可以看INFO、SLOWLOG、MEMORY USAGE、OBJECT ENCODING、CLIENT LIST、命中率和延迟；方案选择可以和String、Hash、ZSet、Lua、Pipeline、Stream、Cluster、哨兵或本地缓存对比。这样既保留八股题的清晰度，也能把答案讲成可落地、可排查、可复盘的经验。

如果面试官继续追问，可以再补一层取舍：为什么不用另一个方案、什么时候会退化、怎么发现反例或异常、出问题后如何止血。Redis题最怕只有名词没有链路，补上命令语义、key 设计、过期策略、内存指标和客户端行为之后，答案会更稳。
