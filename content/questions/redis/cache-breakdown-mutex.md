## 一句话结论

缓存击穿是热点 key 失效瞬间大量请求同时打到数据库，互斥锁方案让只有一个请求去查库并重建缓存。

## 通俗解释

它像食堂热门菜刚卖完，所有人都冲进后厨催。互斥锁就是只让一个人去催菜，其他人在窗口等一会。

## 面试回答

处理流程：

1. 请求查询缓存未命中。
2. 尝试获取互斥锁。
3. 获取成功的请求查数据库并写回缓存。
4. 获取失败的请求短暂等待后重试缓存。
5. 如果数据库也没有数据，要缓存空值防穿透。

互斥锁可以用 Redis `SET NX EX` 实现，但要设置过期时间，避免锁不释放。热点 key 还可以配合逻辑过期、永不过期加异步刷新等方案。

## 常见追问

### 互斥锁会不会让请求变慢？

会让部分请求等待，但比所有请求打爆数据库更可控。热点场景更关注系统稳定性。

### 逻辑过期和互斥锁有什么区别？

逻辑过期可以先返回旧值，再异步刷新缓存，用户体验更平滑，但数据会短暂旧。

## 易错点

- 锁必须有过期时间。
- 等待请求不要无限阻塞。
- 数据库空结果也要处理。
