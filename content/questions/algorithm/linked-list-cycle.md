## 一句话结论

判断链表是否有环常用快慢指针：慢指针一次走一步，快指针一次走两步。如果有环，快指针最终会追上慢指针。

## 通俗解释

两个人在操场跑步，一个跑得快，一个跑得慢。如果跑道是环形，快的人迟早会从后面追上慢的人。如果是一条直路，快的人会先到终点。

## 面试回答

算法步骤：

1. 定义 slow 和 fast，都从头节点出发。
2. slow 每次走一步，fast 每次走两步。
3. 如果 fast 或 fast.next 为空，说明没有环。
4. 如果 slow 和 fast 相遇，说明有环。

时间复杂度 O(n)，空间复杂度 O(1)。

## 常见追问

### 这题最容易被追问的边界是什么？

重点说清空输入、重复元素、极值、无解、越界、负数、溢出和复杂度退化。不要只给结论，要说明哪些条件下结论成立，哪些条件下会退化或需要换方案。

### 怎么证明自己不是在背模板？

用边界样例、时间复杂度、空间复杂度、单调性、递归深度和反例做验证。能说出具体信号、反例或排查入口，答案就从概念层进入实战层。

### 和相近方案怎么区分？

可以拿暴力解、排序、哈希、双指针、二分、堆、动态规划或图遍历对比，从正确性、复杂度、边界样例和实现难度几个维度选择。

### 面试官继续深挖时怎么展开？

先围绕“定义 slow 和 fast，都从头节点出发。”讲清主链路，再补“把入口推导公式更清楚：slow = a + b，fast = a + b + k(b+c) 且 fast = 2slow，推出 a = k(b+c)-b，所以头节点和相遇点同步走会在入口相遇；替换为单节点自环、无环、找入口、环长计算等专项追问”。如果能把边界条件、异常分支和验证闭环连起来，就能接住二面、三面的追问。

## 易错点

- 循环条件要检查 `fast != null && fast.next != null`。
- 不要用 HashSet 时忘了还有 O(1) 空间方案。
- 找入口和判断有环是两个层次。

## 详细讲解

如何判断链表是否有环？ 这道题现在要从“能背出来”修到“能接住追问”。核心结论是：用快慢指针判断环，并理解为什么快指针一定能追上慢指针。回答时先把它放回算法语境，说明它解决什么矛盾，再围绕输入规模、状态定义、边界样例和复杂度推导展开。这样能避免泛泛而谈，也能让面试官听出你知道这题的真实边界。

第一步是拆清问题背景。两个人在操场跑步，一个跑得快，一个跑得慢。如果跑道是环形，快的人迟早会从后面追上慢的人。如果是一条直路，快的人会先到终点。在算法题里，背景不能写成空泛的工程套话，而要落到先给暴力思路，再说明为什么当前数据特征允许优化，最后落到状态、指针、堆、哈希或递归栈这些可验证结构。如果只说“看场景”或“加组件”，读者很难知道下一步该检查什么；如果能把输入、状态、依赖和输出说清，答案就有了主线。

第二步是讲机制。可以围绕这些点展开：定义 slow 和 fast，都从头节点出发。；slow 每次走一步，fast 每次走两步。；如果 fast 或 fast.next 为空，说明没有环。；如果 slow 和 fast 相遇，说明有环。。每个点都要回答三个问题：它在链路里的位置是什么，它改变了什么状态，失败或边界条件下会留下什么现象。把这三件事讲清楚，追问从概念跳到实战时就不会断。

第三步是补边界。这里尤其要主动说明空输入、重复元素、极值、无解、越界、负数、溢出和复杂度退化。循环条件要检查 fast != null && fast.next != null。；不要用 HashSet 时忘了还有 O(1) 空间方案。；找入口和判断有环是两个层次。很多八股答案的问题不是方向错，而是没有说适用范围；一旦遇到数据规模变化、异常分支或版本差异，原来的结论就可能不成立。
