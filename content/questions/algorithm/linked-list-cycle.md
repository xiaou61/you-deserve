## 一句话结论

判断链表是否有环常用快慢指针：慢指针一次走一步，快指针一次走两步。如果有环，快指针最终会追上慢指针。

## 通俗解释

两个人在操场跑步，一个跑得快，一个跑得慢。如果跑道是环形，快的人迟早会从后面追上慢的人。如果是一条直路，快的人会先到终点。

## 面试回答

算法步骤：

1. 定义 slow 和 fast，都从头节点出发。
2. slow 每次走一步，fast 每次走两步。
3. 如果 fast 或 fast.next 为空，说明没有环。
4. 如果 slow 和 fast 相遇，说明有环。

时间复杂度 O(n)，空间复杂度 O(1)。

## 常见追问

### 怎么找环的入口？

相遇后，让一个指针回到头节点，另一个留在相遇点。两个指针每次都走一步，再次相遇的位置就是环入口。

### 为什么一定会相遇？

进入环后，快指针每次比慢指针多走一步，相对距离会不断缩小，最终追上。

## 易错点

- 循环条件要检查 `fast != null && fast.next != null`。
- 不要用 HashSet 时忘了还有 O(1) 空间方案。
- 找入口和判断有环是两个层次。

## 详细讲解

判断链表是否有环，最经典的方法是快慢指针，也叫 Floyd 判圈算法。题目本身不复杂，但它考的是你能不能把“指针运动”和“数学关系”讲清楚。链表有环的意思是某个节点的 next 最终会指回前面某个节点，导致沿着 next 一直走不会走到 null，而是会在环里无限循环。  

为什么快慢指针能判断有环？设慢指针每次走一步，快指针每次走两步。如果链表无环，快指针一定先走到 null，因为它速度更快，终点又是真实存在的。如果链表有环，慢指针进入环后会一直在环里转，快指针也会进入环，而且它每次都比慢指针多走一步，相当于在环上不断追赶慢指针。只要两者都进入同一个有限长度的环，快指针迟早会追上慢指针，于是两者相遇。  

这题容易被问到第二层：为什么相遇后还能找到入环点？设头节点到入环点距离为 a，入环点到第一次相遇点距离为 b，环剩余长度为 c。快指针走过的总距离是慢指针的两倍，于是可以推导出：从头节点再出发一个指针，从第一次相遇点再出发一个指针，两者都每次走一步，最终会在入环点相遇。这个推导不用背复杂公式，面试里能说清“慢指针走了 a+b，快指针多绕了若干圈，因此头到入环点的距离，等于相遇点再走若干步到入环点的距离”就够了。  

实现层面的细节也很重要。循环条件通常写成 while (fast != null && fast.next != null)，因为快指针每次要走两步，必须先保证两步都合法。很多人写成只判断 fast != null，第二步就会空指针异常。空链表、单节点、单节点自环这几种边界情况也要考虑。  

如果面试官追问“能不能用 HashSet”，答案当然可以：遍历链表，把访问过的节点放进集合，若再次遇到同一节点就说明有环。这种方法思路直白，但空间复杂度 O(n)；快慢指针的优势是 O(1) 额外空间。对比讲出来，会显得你有方案取舍。  

再往上一步，很多链表题其实都能借这个题建立“指针运动模型”。比如找中点、找倒数第 N 个节点、判断相交、删除节点，本质都和“不同指针代表不同状态”有关。你能把有环判断和其他链表模板联系起来，说明你掌握的是套路背后的共性。工程上虽然未必经常手写链表，但这种思维方式在理解消息链、调用链、状态推进和指针式结构时都很有用，这也是面试官爱问它的原因。  

图解非常适合画：一条链表前半段是直线，后半段连成一个环；慢指针每次走一格，快指针每次走两格，在环内相遇。再补一张进阶图，从头节点和相遇点分别发出两个一步一走的指针，最终在入环点汇合。这样这道题就从“背模板”变成了“看得见的运动过程”。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 如何判断链表是否有环 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：判断链表是否有环常用快慢指针：慢指针一次走一步，快指针一次走两步

拆开来看，第一层是背景问题：用快慢指针判断环，并理解为什么快指针一定能追上慢指针。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：定义 slow 和 fast，都从头节点出发。 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：slow 每次走一步，fast 每次走两步。 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：如果 fast 或 fast.next 为空，说明没有环。 不要直接背模板，要解释为什么这个模板适合当前问题。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：看到 算法、链表、快慢指针 相关题目时，先识别输入规模、是否有序、是否需要去重、是否存在重复子问题，再决定用哪类模板。
- **怎么做方案**：先说暴力解法，再给出优化方向。围绕“状态定义、转移关系、数据结构、边界样例、复杂度”五个点组织，避免一上来就写代码。
- **怎么验证效果**：至少准备空输入、单元素、重复元素、极大值、边界窗口或无解场景。能主动讲测试样例，面试官会更相信你不是背模板。
- **怎么兜底**：如果最优解一时想不到，先交代可运行方案和复杂度，再逐步优化到更好的时间或空间复杂度。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 如何判断链表是否有环 讲成一次工程决策，而不是一个孤立知识点。重点围绕 问题抽象、状态定义、复杂度和边界样例。

3. **补充边界**：说明什么情况下有效，什么情况下会失效，以及为什么需要配套措施。
