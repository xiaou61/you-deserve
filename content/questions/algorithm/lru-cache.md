## 一句话结论

LRU 通常用哈希表 + 双向链表实现。哈希表负责 O(1) 定位节点，双向链表负责维护最近使用顺序。

## 通俗解释

LRU 像桌面只放有限本书。你刚看过的书放最前面，很久没看的书慢慢被挤到最后。桌面满了，就把最后那本最久没看的拿走。

哈希表让你快速找到书，链表让你快速调整书的位置。

## 面试回答

设计思路：

1. 哈希表 `Map<key, Node>` 存 key 到链表节点的映射。
2. 双向链表按最近使用顺序排列，头部最新，尾部最旧。
3. `get` 时，如果 key 存在，把节点移动到头部。
4. `put` 时，如果 key 存在，更新值并移动到头部；如果不存在，插入头部。
5. 超过容量时，删除尾部节点，并从 Map 中移除。

这样 `get` 和 `put` 都可以做到 O(1)。

## 常见追问

### 为什么要双向链表？

删除一个已知节点时，需要快速连接它的前后节点。单链表不知道前驱节点，删除不方便。

### Java 里有没有现成结构？

可以用 `LinkedHashMap` 实现 LRU，它内部维护了链表顺序。

## 易错点

- 不要只用数组或普通队列，删除中间节点会慢。
- 不要忘记更新已有 key 时也要移动到头部。
- 不要忘记容量超限时同步删除 Map。

## 详细讲解

手写 LRU 缓存这道题，本质考的是“你能不能把需求翻译成数据结构组合”。LRU 的完整要求通常有两个：第一，get 和 put 都要尽量快，理想是 O(1)；第二，当容量满了时，要淘汰“最近最少使用”的元素。如果只用数组或链表，淘汰顺序好维护，但查找太慢；如果只用 HashMap，查找快，但不知道哪个元素最久没用。所以标准解法几乎总是 HashMap 加双向链表。  

HashMap 负责通过 key O(1) 找到节点；双向链表负责维护访问顺序。链表头表示最近刚访问过的节点，链表尾表示最久没访问的节点。每次 get(key) 命中，除了返回值，还要把这个节点移到链表头，因为它刚被使用过。每次 put(key, value) 分两种情况：如果 key 已存在，就更新值并移动到头部；如果 key 不存在，就新建节点插到头部，同时写入 HashMap。如果插入后容量超限，就删掉尾部节点，并把它从 HashMap 里也移除。  

为什么必须用双向链表，而不是单向链表？因为节点移动和删除要求 O(1)。单向链表删除一个中间节点时，你还得先找到它的前驱；双向链表拿到节点本身后，就能直接通过 prev 和 next 把它摘掉。很多实现还会额外放两个哨兵节点 head 和 tail，这样插入头部、删除尾部、删除中间节点都能统一处理，少写很多 if 判断。  

这题真正容易错的，不在思路，而在细节。比如：更新已有 key 时，不能新建节点后忘删旧节点；淘汰尾节点时，要先从链表摘除，再从 Map 删除；容量为 0 或 1 时边界怎么处理；get 未命中返回什么；节点复用时指针是否残留旧引用。很多人能说出“HashMap + 双向链表”，但现场写代码会因为指针更新顺序出 bug。  

面试里还可以顺带讲 JDK 的 LinkedHashMap。它内部就维护了访问顺序，通过重写 removeEldestEntry 可以快速做一个 LRU，这说明你知道标准题解，也知道工程里不一定手写轮子。但如果面试官要求手写，你还是得能把核心操作讲清楚：addFirst、removeNode、moveToHead、removeLast 四个动作就是整题骨架。  

图解建议画成两层：上层是 HashMap，把 key 指到链表节点；下层是双向链表，从头到尾表示“最近使用 -> 最久未使用”。然后演示三个动作：一次 get 命中把中间节点移到头部；一次 put 新增插到头部；一次容量溢出删除尾节点。最后收一句：LRU 的关键不是“背模板”，而是理解为什么需要同时满足快速定位和快速维护顺序。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 如何手写 LRU 缓存 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：LRU 通常用哈希表 + 双向链表实现

拆开来看，第一层是背景问题：用哈希表 + 双向链表实现 O(1) 查询、更新和淘汰。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：哈希表 Map 存 key 到链表节点的映射。 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：双向链表按最近使用顺序排列，头部最新，尾部最旧。 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：get 时，如果 key 存在，把节点移动到头部。 不要直接背模板，要解释为什么这个模板适合当前问题。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：看到 算法、LRU、哈希表、双向链表 相关题目时，先识别输入规模、是否有序、是否需要去重、是否存在重复子问题，再决定用哪类模板。

- **怎么验证效果**：至少准备空输入、单元素、重复元素、极大值、边界窗口或无解场景。能主动讲测试样例，面试官会更相信你不是背模板。
- **怎么兜底**：如果最优解一时想不到，先交代可运行方案和复杂度，再逐步优化到更好的时间或空间复杂度。
