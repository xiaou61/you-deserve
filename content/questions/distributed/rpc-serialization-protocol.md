## 一句话结论

RPC 序列化就是把内存对象转换成网络可传输的字节，再在对端还原；协议选择会影响性能、体积、兼容性和跨语言能力。

## 通俗解释

它像搬家打包。你家里的东西不能原样塞进车里，必须装箱、贴标签，到新家再拆箱。装得好，搬得快；装得乱，容易丢东西。

## 面试回答

RPC 调用跨进程、跨机器，方法参数和返回值必须经过序列化。常见考虑点：

- 性能：编码解码速度是否快。
- 体积：网络传输字节是否小。
- 兼容性：字段新增删除是否容易兼容。
- 跨语言：不同语言是否都能解析。
- 可读性：排查问题时是否容易看懂。

Java 原生序列化使用方便但性能和安全性一般，实际项目里常见 JSON、Hessian、Kryo、Protobuf 等方案。对外接口更重视通用性，对内高频调用更重视性能和体积。

## 常见追问

### Protobuf 为什么常用于 RPC？

它体积小、性能好、跨语言能力强，并且通过 schema 定义接口契约，适合服务间通信。

### JSON 有什么优势？

可读性强、调试方便、生态通用，但体积和性能通常不如二进制协议。

## 易错点

- 不要只说“转成 JSON”，要说明对象到字节再还原。
- 不要忽略版本兼容。
- 不要把序列化性能和网络延迟混为一谈。

## 记忆钩子

**RPC 传的不是对象，是对象打包后的字节。**
