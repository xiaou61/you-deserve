## 一句话结论

常见限流算法有固定窗口、滑动窗口、漏桶和令牌桶。实际系统常用令牌桶或滑动窗口来兼顾平滑和突发流量。

## 通俗解释

限流像景区入口控流。固定窗口像每分钟只放 100 人；漏桶像固定速度出水；令牌桶像桶里攒门票，有票才能进，攒了一些票时可以短暂应对突发。

## 面试回答

常见算法：

- 固定窗口：实现简单，但窗口边界可能瞬间放过两倍流量。
- 滑动窗口：把时间切得更细，限流更平滑。
- 漏桶：请求按固定速率流出，能削峰但不适合突发。
- 令牌桶：按固定速率生成令牌，请求拿到令牌才能通过，允许一定突发。

限流通常放在网关、接口、用户、IP、业务 key 等维度上，并配合降级返回。

## 常见追问

### 这类分布式题最容易在哪个边界翻车？

最常见的是把主链路讲顺了，却漏掉超时、重试、幂等、补偿和脑裂边界。像「常见限流算法有哪些」这种题，如果不主动补这几个点，面试官通常会继续追着问。

### 线上排查时你会先看什么？

先看入口流量、下游错误率、超时分布、重试次数、熔断/限流命中，再看状态是不是已经写了一半。分布式问题最怕只盯日志，不把链路和指标一起对。

### 规模再涨一截，原来的结论会先变在哪里？

通常会先体现在协调成本、热点分布、重试风暴、选主抖动或一致性代价上。答题时最好补一句：流量变大后，我会先判断瓶颈是网络、存储还是协调层。

### 如果要讲成项目经历，该怎么组织？

可以按“为什么要做 常见限流算法有哪些 -> 当时系统卡在哪 -> 方案怎么拆 -> 出问题怎么兜底 -> 最后靠什么指标证明有效”这条线来讲，会比纯概念更像你自己做过。

## 易错点

- 不要只说“用 Redis 计数”，要说算法。
- 不要忽略限流维度。
- 不要限流后直接让用户看到奇怪错误。

## 详细讲解

常见限流算法有哪些？ 核心结论是：理解固定窗口、滑动窗口、漏桶和令牌桶的区别。回答时先把它放回分布式语境，说明它解决什么矛盾，再围绕角色边界、一致性语义、超时重试和故障恢复展开。

第一步是拆清问题背景。限流像景区入口控流。固定窗口像每分钟只放 100 人；漏桶像固定速度出水；令牌桶像桶里攒门票，有票才能进，攒了一些票时可以短暂应对突发。在分布式题里，背景不能写成空泛的工程套话，而要落到把请求、协调者、参与节点、存储状态和补偿任务串成时序，而不是只罗列组件名。如果只说“看场景”或“加组件”，读者很难知道下一步该检查什么；如果能把输入、状态、依赖和输出说清，答案就有了主线。

第二步是讲机制。可以围绕这些点展开：固定窗口：实现简单，但窗口边界可能瞬间放过两倍流量。；滑动窗口：把时间切得更细，限流更平滑。；漏桶：请求按固定速率流出，能削峰但不适合突发。；令牌桶：按固定速率生成令牌，请求拿到令牌才能通过，允许一定突发。。每个点都要回答三个问题：它在链路里的位置是什么，它改变了什么状态，失败或边界条件下会留下什么现象。把这三件事讲清楚，追问从概念跳到实战时就不会断。

第三步是补边界。这里尤其要主动说明网络分区、时钟漂移、旧 leader、重复请求、超时重试、幂等和脑裂。不要只说“用 Redis 计数”，要说算法。；不要忽略限流维度。；不要限流后直接让用户看到奇怪错误。很多八股答案的问题不是方向错，而是没有说适用范围；一旦遇到数据规模变化、异常分支或版本差异，原来的结论就可能不成立。

第四步是补专项深度：补分布式限流的 Redis Lua 原子性、本地预分配令牌、热点 key、fail open/fail closed、限流命中率和误伤率；替换为限流专项链路。这部分适合放在面试回答后半段，用来证明你不仅知道标准答案，也知道高频追问会往哪里挖。

最后收束成一套回答节奏：先给结论，再讲机制，再补边界，最后说验证方式。对这题来说，验证可以看租约、任期、版本号、TraceId、重试次数、状态机、延迟和错误率；方案选择可以和本地事务、最终一致、TCC、Saga、Raft、租约、fencing token 或配置中心方案对比。这样既保留八股题的清晰度，也能把答案讲成可落地、可排查、可复盘的经验。

如果面试官继续追问，可以再补一层取舍：为什么不用另一个方案、什么时候会退化、怎么发现反例或异常、出问题后如何止血。分布式题最怕只有名词没有链路，补上角色边界、一致性语义、超时重试和故障恢复之后，答案会更稳。
