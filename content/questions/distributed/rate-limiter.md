## 一句话结论

常见限流算法有固定窗口、滑动窗口、漏桶和令牌桶。实际系统常用令牌桶或滑动窗口来兼顾平滑和突发流量。

## 通俗解释

限流像景区入口控流。固定窗口像每分钟只放 100 人；漏桶像固定速度出水；令牌桶像桶里攒门票，有票才能进，攒了一些票时可以短暂应对突发。

## 面试回答

常见算法：

- 固定窗口：实现简单，但窗口边界可能瞬间放过两倍流量。
- 滑动窗口：把时间切得更细，限流更平滑。
- 漏桶：请求按固定速率流出，能削峰但不适合突发。
- 令牌桶：按固定速率生成令牌，请求拿到令牌才能通过，允许一定突发。

限流通常放在网关、接口、用户、IP、业务 key 等维度上，并配合降级返回。

## 常见追问

### 限流和熔断有什么区别？

限流是主动控制进入系统的流量；熔断是下游异常时暂时停止调用，保护系统。

### 限流后请求怎么处理？

可以直接拒绝、排队、返回友好提示、降级到缓存结果，具体看业务。

## 易错点

- 不要只说“用 Redis 计数”，要说算法。
- 不要忽略限流维度。
- 不要限流后直接让用户看到奇怪错误。

## 详细讲解

限流的目标是保护系统，把超过承载能力的流量挡在外面或降级处理。常见算法有固定窗口、滑动窗口、漏桶、令牌桶。面试里要讲清楚每种算法解决什么问题、有什么缺点，以及业务上如何选择限流维度和阈值。限流不是为了拒绝用户，而是为了让核心服务在流量异常时仍然可用。

固定窗口最简单，比如每分钟最多 100 次。实现上用 key 加当前分钟计数即可。但它有边界突刺问题：用户在上一分钟最后一秒打 100 次，下一分钟第一秒再打 100 次，短时间内实际承受 200 次。滑动窗口把时间切成多个小格，或者记录最近一段时间请求时间戳，统计更平滑，但实现和存储成本更高。

漏桶算法把请求倒进桶里，桶以固定速率流出。桶满了就拒绝新请求。它适合把突发流量整形成稳定速率，下游处理会很平滑，但对突发不友好。令牌桶则按固定速率生成令牌，请求拿到令牌才能通过。桶里可以积累一定令牌，所以允许短时间突发，长期平均速率受限。很多网关和客户端限流更喜欢令牌桶，因为它兼顾平均限制和突发弹性。

限流要先确定维度。可以按 IP、用户 ID、接口、租户、设备、商户、API key、全局 QPS 等维度限流。维度太粗会误伤正常用户，太细可能挡不住整体洪峰。阈值也不能拍脑袋，要基于压测、下游容量、历史峰值和错误预算。核心接口和非核心接口的策略也不同，非核心可以更早降级，核心要保留容量。

单机限流适合服务实例本地保护，简单高效；分布式限流需要多个实例共享计数，常用 Redis Lua 保证原子性，或者在 API 网关统一限流。分布式限流要考虑 Redis 自身成为瓶颈，所以高流量场景可能用本地预分配令牌、分片计数、近似限流等方式降低中心依赖。

图解可以画四个算法对比：固定窗口边界突刺，滑动窗口平滑统计，漏桶匀速流出，令牌桶积累令牌允许突发。底部画落地链路：请求进入网关，按用户和接口生成限流 key，Redis Lua 判断，超限返回 429 或降级响应，监控记录通过量、拒绝量和下游错误率。

如果还要把答案讲得更像项目经验，可以用验证闭环收束：先说明这个问题在什么条件下出现，再说线上能从哪些信号确认，最后讲处理后怎么证明风险已经下降。对这道题来说，可以围绕“理解固定窗口、滑动窗口、漏桶和令牌桶的区别”展开，再结合 TraceId、消费延迟、重试次数、死信数量、下游耗时和最终数据状态 做交叉验证。这样回答不会停在概念层，而是把发现问题、定位原因、采取动作、验证结果串成一条完整链路。最后再补一句边界：方案要和业务容忍度匹配，不能只追求理论正确，还要考虑性能、复杂度、维护成本和失败后的恢复路径。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 常见限流算法有哪些 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：常见限流算法有固定窗口、滑动窗口、漏桶和令牌桶

拆开来看，第一层是背景问题：理解固定窗口、滑动窗口、漏桶和令牌桶的区别。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：固定窗口：实现简单，但窗口边界可能瞬间放过两倍流量。 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：滑动窗口：把时间切得更细，限流更平滑。 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：漏桶：请求按固定速率流出，能削峰但不适合突发。 不要只说引入中间件，要说明消息丢失、重复、乱序和积压如何处理。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：当链路里出现 限流、高并发、令牌桶、漏桶 相关的超时、重复、丢失、乱序、积压或状态不一致时，这道题就是排查入口。
- **怎么做方案**：按“生产者 -> Broker/注册中心 -> 消费者 -> 数据库 -> 补偿任务”拆链路，明确每一步的状态、重试和幂等键。
- **怎么验证效果**：看消息堆积、消费延迟、重试次数、死信数量、接口超时、TraceId 链路和最终数据状态。按生产链路讲清楚请求进入、状态推进、异常补偿和监控告警。
- **怎么兜底**：准备幂等表、去重键、补偿任务、死信处理、限流降级和人工重放工具，保证失败可见、可追踪、可恢复。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 常见限流算法有哪些 讲成一次工程决策，而不是一个孤立知识点。重点围绕 链路状态、失败重试、幂等、顺序和最终一致性。

## 回答模板

面试时可以按这个节奏组织：

1. **先给结论**：理解固定窗口、滑动窗口、漏桶和令牌桶的区别。
2. **再讲机制**：它的核心不是某个名词，而是一组处理链路。把关键角色、状态变化和触发条件说清楚。
3. **补充边界**：说明什么情况下有效，什么情况下会失效，以及为什么需要配套措施。
4. **落到项目**：如果我在项目里遇到，会先看指标和日志定位问题，再用灰度、压测和回滚策略验证方案。
5. **收一句风险**：真正重要的是不要只让功能跑通，还要保证高并发、异常分支和数据状态都可控。
