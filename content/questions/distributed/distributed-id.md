## 一句话结论

分布式 ID 要保证全局唯一，常见方案有 UUID、数据库自增/号段、Redis 自增、雪花算法。真实项目要结合趋势递增、性能和可用性选择。

## 通俗解释

分布式 ID 像全国多个窗口同时发号码，不能重复，还最好大致有顺序，方便排队和查账。

## 面试回答

常见方案：

- UUID：本地生成，简单，但太长且无序，不利于数据库索引。
- 数据库自增：简单但单点压力大。
- 号段模式：一次从数据库拿一段 ID，本地发号，性能更好。
- Redis 自增：性能高，但依赖 Redis 可用性和持久化。
- 雪花算法：按时间戳、机器 ID、序列号生成，趋势递增，性能好。

雪花算法要注意时钟回拨和机器 ID 分配问题。

## 常见追问

### 为什么 UUID 不适合作为 MySQL 主键？

UUID 长且随机，插入 B+ 树时容易导致页分裂和索引膨胀。

### 雪花算法时钟回拨怎么办？

可以等待时间追上、使用备用序列、记录上次时间戳并报警，严重场景要有统一时钟治理。

## 易错点

- 不要只说 UUID。
- 不要忽略数据库索引性能。
- 不要忘记时钟回拨风险。

## 记忆钩子

**分布式发号：不能重，最好顺，还要发得快。**
