## 一句话结论

分布式系统里超时、重试、重复消息很常见，幂等能保证同一个业务操作重复执行也只产生一次有效结果。

## 通俗解释

你点外卖时按钮卡了一下，连点三次。系统应该只生成一单，而不是给你送三份。

## 面试回答

分布式系统更需要幂等，因为：

- 网络超时后调用方可能重试。
- MQ 可能重复投递。
- 支付回调可能多次通知。
- 用户可能重复点击。
- 服务调用链中某一步成功但响应丢失。

常见实现方式：

- 业务唯一键。
- 数据库唯一索引。
- 去重表。
- 状态机流转校验。
- Redis token 防重复提交。

幂等设计要围绕业务语义，而不只是接口层拦截。

## 常见追问

### 查询接口需要幂等吗？

普通查询天然幂等，因为重复查询不会改变状态。更需要关注的是创建、扣款、发券、发货这类写操作。

### Redis token 防重有什么风险？

要考虑 token 过期、原子删除、异常重试和用户体验。

## 易错点

- 幂等不是简单防重复点击。
- 唯一键要选业务唯一标识，而不是随便生成。
- 幂等失败可能造成严重资损。

## 记忆钩子

**分布式一定会重试，幂等保证重复不重复生效。**
