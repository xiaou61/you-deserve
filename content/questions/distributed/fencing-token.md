## 一句话结论

fencing token 是每次加锁时生成的递增令牌，资源方只接受更大的令牌，用来防止过期锁持有者继续写数据。

## 通俗解释

它像排队号码牌。即使前一个人因为卡顿晚到了窗口，窗口也会看号码，只服务最新有效号码，过期号码不能插队。

## 面试回答

普通分布式锁可能遇到这样的问题：

1. 客户端 A 拿到锁。
2. A 长时间 GC 或网络卡顿，锁过期。
3. 客户端 B 拿到新锁并操作资源。
4. A 恢复后以为自己还持有锁，又写了一次资源。

fencing token 的做法是：每次成功拿锁时返回一个递增 token，资源服务在写入时校验 token，只接受比已处理 token 更新的请求。这样过期客户端即使恢复，也会因为 token 旧被拒绝。

## 易错点

- 不要把 fencing token 说成普通 UUID。
- token 必须递增，资源方必须校验。
- 不是所有业务都需要，但强一致写入场景很有价值。

分布式锁里的 fencing token 是什么？ 这道题现在要从“能背出来”修到“能接住追问”。核心结论是：fencing token 是递增令牌，用来让资源方拒绝过期锁持有者的写入，补充分布式锁的安全性。回答时先把它放回分布式语境，说明它解决什么矛盾，再围绕角色边界、一致性语义、超时重试和故障恢复展开。这样能避免泛泛而谈，也能让面试官听出你知道这题的真实边界。

第一步是拆清问题背景。它像排队号码牌。即使前一个人因为卡顿晚到了窗口，窗口也会看号码，只服务最新有效号码，过期号码不能插队。在分布式题里，背景不能写成空泛的工程套话，而要落到把请求、协调者、参与节点、存储状态和补偿任务串成时序，而不是只罗列组件名。如果只说“看场景”或“加组件”，读者很难知道下一步该检查什么；如果能把输入、状态、依赖和输出说清，答案就有了主线。

第二步是讲机制。可以围绕这些点展开：客户端 A 拿到锁。；A 长时间 GC 或网络卡顿，锁过期。；客户端 B 拿到新锁并操作资源。；A 恢复后以为自己还持有锁，又写了一次资源。。每个点都要回答三个问题：它在链路里的位置是什么，它改变了什么状态，失败或边界条件下会留下什么现象。把这三件事讲清楚，追问从概念跳到实战时就不会断。
