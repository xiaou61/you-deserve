## 一句话结论

fencing token 是每次加锁时生成的递增令牌，资源方只接受更大的令牌，用来防止过期锁持有者继续写数据。

## 通俗解释

它像排队号码牌。即使前一个人因为卡顿晚到了窗口，窗口也会看号码，只服务最新有效号码，过期号码不能插队。

## 面试回答

普通分布式锁可能遇到这样的问题：

1. 客户端 A 拿到锁。
2. A 长时间 GC 或网络卡顿，锁过期。
3. 客户端 B 拿到新锁并操作资源。
4. A 恢复后以为自己还持有锁，又写了一次资源。

fencing token 的做法是：每次成功拿锁时返回一个递增 token，资源服务在写入时校验 token，只接受比已处理 token 更新的请求。这样过期客户端即使恢复，也会因为 token 旧被拒绝。

## 常见追问

### Redis 分布式锁加过期时间不就够了吗？

过期时间能避免死锁，但不能完全避免“锁过期后旧客户端继续写”的问题，所以关键资源还需要资源侧校验。

### token 存在哪里？

通常要由真正被保护的资源方保存最近处理过的最大 token，比如数据库记录版本、存储系统元数据等。

## 易错点

- 不要把 fencing token 说成普通 UUID。
- token 必须递增，资源方必须校验。
- 不是所有业务都需要，但强一致写入场景很有价值。

## 实战落地

- **什么时候会遇到**：当链路里出现 分布式锁、Fencing Token、一致性、并发控制 相关的超时、重复、丢失、乱序、积压或状态不一致时，这道题就是排查入口。
- **怎么做方案**：按“生产者 -> Broker/注册中心 -> 消费者 -> 数据库 -> 补偿任务”拆链路，明确每一步的状态、重试和幂等键。
- **怎么验证效果**：看消息堆积、消费延迟、重试次数、死信数量、接口超时、TraceId 链路和最终数据状态。按生产链路讲清楚请求进入、状态推进、异常补偿和监控告警。
- **怎么兜底**：准备幂等表、去重键、补偿任务、死信处理、限流降级和人工重放工具，保证失败可见、可追踪、可恢复。
