## 一句话结论

双亲委派是类加载器收到请求后先交给父加载器加载，父加载器加载不了才自己加载，用来保证核心类库安全和类加载一致性。

## 通俗解释

双亲委派像公司审批。普通员工先交给上级，上级能处理就不往下乱分；上级处理不了，才让具体部门自己办。

## 面试回答

类加载器通常包括启动类加载器、扩展类加载器、应用类加载器和自定义类加载器。

双亲委派流程是：

1. 当前类加载器收到加载请求。
2. 先委托给父加载器。
3. 父加载器继续向上委托。
4. 顶层加载不了时，再逐层向下尝试加载。

好处是避免核心类被篡改，比如用户自己写一个 `java.lang.String`，也不会轻易替代 JDK 自带的类。

但一些场景需要打破双亲委派，比如 SPI 需要父加载器加载的接口找到子加载器中的实现，Tomcat 需要不同 Web 应用隔离类，热部署也可能用自定义类加载器重新加载类。

## 常见追问

### 打破双亲委派是不是不安全？

不是绝对不安全，而是要在隔离、插件化、SPI 等特定场景下有控制地使用。

### Tomcat 为什么需要自己的类加载机制？

因为不同 Web 应用可能依赖不同版本的类库，需要互相隔离，不能都由同一个应用类加载器统一加载。

## 易错点

- 不要说双亲委派是父类加载子类。
- 不要把“父加载器”理解成 Java 继承关系。
- 打破双亲委派不是乱来，而是为了解决扩展和隔离问题。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 双亲委派模型是什么？为什么要打破它 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：双亲委派是类加载器收到请求后先交给父加载器加载，父加载器加载不了才自己加载，用来保证核心类库安全和类加载一致性

拆开来看，第一层是背景问题：双亲委派优先让父加载器加载类，保证核心类安全；SPI、热部署等场景可能需要打破。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：当前类加载器收到加载请求。 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：先委托给父加载器。 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：父加载器继续向上委托。 不要只背概念名词，要能说出触发条件、底层机制和排查入口。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：当业务代码出现 Java、JVM、类加载器、双亲委派 相关的并发异常、性能抖动、配置不生效、对象行为和预期不一致时，就可以用这道题定位原因。
- **怎么做方案**：先看触发条件，再看运行时机制。围绕“调用入口、对象状态、线程边界、框架代理、异常日志”五个位置检查，判断 双亲委派模型是什么？为什么要打破它 是设计问题、用法问题还是环境问题。
- **怎么验证效果**：用单元测试、压测、日志、线程栈、JFR/GC 日志或本地最小复现确认。结合真实业务代码说明什么时候会踩坑、怎么定位、怎么替换方案。
- **怎么兜底**：准备替代 API、隔离开关、降级策略、配置回滚和监控告警。面试里能讲出兜底，说明你不是只会写 happy path。
