## 一句话结论

GC 调优常看堆大小、垃圾收集器、GC 日志、停顿时间目标、新生代比例、对象晋升阈值等参数，但核心是先观测再调整。

## 通俗解释

GC 调优像调空调。不是一上来乱按按钮，而是先看房间大小、人数、温度变化，再决定风量、温度和模式。

## 面试回答

常见参数可以按几类记：

- 堆大小：`-Xms`、`-Xmx`。
- 新生代相关：`-Xmn`、`-XX:SurvivorRatio`。
- 元空间：`-XX:MetaspaceSize`、`-XX:MaxMetaspaceSize`。
- 收集器：如 G1、ZGC、Parallel GC 等。
- GC 日志：开启 GC 日志用于分析频率、耗时、回收效果。
- 停顿目标：例如 G1 的 `-XX:MaxGCPauseMillis`。

调优思路是先通过监控和 GC 日志判断问题：是内存泄漏、对象创建太快、老年代压力大，还是停顿时间不达标。确认原因后再调整参数或改代码。

## 常见追问

### JVM 题最容易讲得太空的地方在哪？

就是只讲术语，不讲观测。像「JVM GC 调优常看哪些参数」这种题，如果不顺手补一句会看什么日志、火焰图、JFR 或诊断命令，内容就容易飘。

### 排查时第一眼看什么？

先确认是 CPU、内存、停顿还是线程卡死，再决定去看 GC 日志、jstack、堆外内存、类加载还是编译行为。JVM 排查最怕一上来就盲打命令。

### 为什么同一个结论在不同线上环境会不一样？

因为堆大小、GC 选择、对象分配模式、容器限制和流量形态都在变。答题时主动提版本差异和运行参数，通常会显得更稳。

### 怎么讲得像真处理过线上问题？

别只背定义，可以直接说：如果 JVM GC 调优常看哪些参数 相关指标异常，我会先看哪几个信号，再判断是短期抖动还是结构性问题，最后再决定调参数、改代码还是换收集器。

## 易错点

- 不要把调优等同于背参数。
- 不要没看日志就说改 `-Xmx`。
- 很多 GC 问题本质是代码对象创建或缓存失控。

## 详细讲解

JVM GC 调优常看哪些参数？ 核心结论是：GC 调优不是乱改参数，而是围绕堆大小、收集器、日志、停顿目标和对象晋升来分析。回答时先把它放回JVM语境，说明它解决什么矛盾，再围绕运行时结构、参数开关、日志指标、排障工具和版本差异展开。

第一步是拆清问题背景。GC 调优像调空调。不是一上来乱按按钮，而是先看房间大小、人数、温度变化，再决定风量、温度和模式。在JVM题里，背景不能写成空泛的工程套话，而要落到把内存区域、类加载、JIT、GC 或 safepoint 放到真实运行时链路里解释。如果只说“看场景”或“加组件”，读者很难知道下一步该检查什么；如果能把输入、状态、依赖和输出说清，答案就有了主线。

第二步是讲机制。可以围绕这些点展开：堆大小：-Xms、-Xmx。；新生代相关：-Xmn、-XX:SurvivorRatio。；元空间：-XX:MetaspaceSize、-XX:MaxMetaspaceSize。；收集器：如 G1、ZGC、Parallel GC 等。。每个点都要回答三个问题：它在链路里的位置是什么，它改变了什么状态，失败或边界条件下会留下什么现象。把这三件事讲清楚，追问从概念跳到实战时就不会断。

第三步是补边界。这里尤其要主动说明堆外内存、容器限制、GC 停顿、类加载泄漏、JIT 退化和版本默认值变化。不要把调优等同于背参数。；不要没看日志就说改 -Xmx。；很多 GC 问题本质是代码对象创建或缓存失控。很多八股答案的问题不是方向错，而是没有说适用范围；一旦遇到数据规模变化、异常分支或版本差异，原来的结论就可能不成立。

第四步是补专项深度：补 JDK 9+ GC 日志参数、G1 的 MaxGCPauseMillis、InitiatingHeapOccupancyPercent、G1HeapRegionSize、ParallelGCThreads/ConcGCThreads，以及调参前后用同一压测流量验证。这部分适合放在面试回答后半段，用来证明你不仅知道标准答案，也知道高频追问会往哪里挖。

最后收束成一套回答节奏：先给结论，再讲机制，再补边界，最后说验证方式。对这题来说，验证可以看GC 日志、JFR、jcmd、jstack、jmap、NMT、CodeCache、Safepoint 和线程状态；方案选择可以和GC 参数、内存拆分、对象生命周期分析、类加载排查或运行时采样对比。这样既保留八股题的清晰度，也能把答案讲成可落地、可排查、可复盘的经验。
