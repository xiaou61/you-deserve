## 一句话结论

GC 调优常看堆大小、垃圾收集器、GC 日志、停顿时间目标、新生代比例、对象晋升阈值等参数，但核心是先观测再调整。

## 通俗解释

GC 调优像调空调。不是一上来乱按按钮，而是先看房间大小、人数、温度变化，再决定风量、温度和模式。

## 面试回答

常见参数可以按几类记：

- 堆大小：`-Xms`、`-Xmx`。
- 新生代相关：`-Xmn`、`-XX:SurvivorRatio`。
- 元空间：`-XX:MetaspaceSize`、`-XX:MaxMetaspaceSize`。
- 收集器：如 G1、ZGC、Parallel GC 等。
- GC 日志：开启 GC 日志用于分析频率、耗时、回收效果。
- 停顿目标：例如 G1 的 `-XX:MaxGCPauseMillis`。

调优思路是先通过监控和 GC 日志判断问题：是内存泄漏、对象创建太快、老年代压力大，还是停顿时间不达标。确认原因后再调整参数或改代码。

## 常见追问

### GC 调优第一步是什么？

第一步不是改参数，而是收集证据：堆使用曲线、GC 日志、接口耗时、对象分配情况。

### 堆越大越好吗？

不是。堆大可以减少 GC 频率，但也可能增加单次回收停顿和内存成本。

## 易错点

- 不要把调优等同于背参数。
- 不要没看日志就说改 `-Xmx`。
- 很多 GC 问题本质是代码对象创建或缓存失控。

## 实战落地

- **什么时候会遇到**：当业务代码出现 Java、JVM、GC 调优、线上排查 相关的并发异常、性能抖动、配置不生效、对象行为和预期不一致时，就可以用这道题定位原因。
- **怎么做方案**：先看触发条件，再看运行时机制。围绕“调用入口、对象状态、线程边界、框架代理、异常日志”五个位置检查，判断 JVM GC 调优常看哪些参数 是设计问题、用法问题还是环境问题。
- **怎么验证效果**：用单元测试、压测、日志、线程栈、JFR/GC 日志或本地最小复现确认。结合真实业务代码说明什么时候会踩坑、怎么定位、怎么替换方案。
- **怎么兜底**：准备替代 API、隔离开关、降级策略、配置回滚和监控告警。面试里能讲出兜底，说明你不是只会写 happy path。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 JVM GC 调优常看哪些参数 讲成一次工程决策，而不是一个孤立知识点。重点围绕 对象模型、运行时行为、边界条件和框架默认行为。

## 回答模板

2. **再讲机制**：它的核心不是某个名词，而是一组处理链路。把关键角色、状态变化和触发条件说清楚。
3. **补充边界**：说明什么情况下有效，什么情况下会失效，以及为什么需要配套措施。
4. **落到项目**：如果我在项目里遇到，会先看指标和日志定位问题，再用灰度、压测和回滚策略验证方案。
