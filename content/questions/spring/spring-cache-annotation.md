## 一句话结论

@Cacheable、@CachePut、@CacheEvict 让业务方法和缓存操作解耦，适合读多写少的数据。

## 通俗解释

缓存注解像自动门，进出方便，但门禁规则和清理策略要先设计好。

## 面试回答

可以从这几层回答：

- @Cacheable 命中缓存时不执行方法，未命中才调用方法并写缓存。
- @CachePut 通常用于更新后同步写缓存。
- @CacheEvict 用于删除缓存，可在更新或删除数据后执行。
- key 要包含影响结果的关键参数，避免串数据。

## 常见追问

### @Cacheable 方法一定会执行吗？

不一定。缓存命中时目标方法不会执行，未命中时才执行并回填。

### TTL 在注解里配置吗？

通常由具体缓存实现和 CacheManager 配置控制，比如 RedisCacheConfiguration 或 Caffeine 策略。

### 为什么自调用会导致缓存不生效？

缓存注解依赖 Spring 代理，同类内部 this 调用绕过代理，所以不会进入缓存拦截器。

### 更新数据库后先删缓存还是先改库？

常见做法是先改库，事务提交后删缓存，并配合重试、消息补偿或延迟双删处理异常窗口。

## 易错点

- 方法内部自调用导致缓存注解不生效。
- 只加 Cacheable，不设计更新和删除时的失效策略。

## 详细讲解

Spring Cache 注解怎么用，这类题不能只答“有 `@Cacheable`、`@CachePut`、`@CacheEvict` 三个注解”。先用一句话压住主题，比如 Spring Cache 用注解抽象缓存读写，但一致性和 key 设计仍要自己负责，然后马上把读者带回真实链路：一次查询先查缓存，没命中才查数据库并回填；一次更新改完库之后，要不要同步写缓存、删缓存、还是延迟双删。

更顺的讲法，是先把三个注解放到读写流程里。`@Cacheable` 负责“命中就直接返回，未命中才执行方法”；`@CachePut` 更像“方法无论如何都执行，执行完把结果写回缓存”；`@CacheEvict` 则是在更新或删除后做失效。真正值钱的不是背名字，而是知道它们分别站在读路径、写路径和失效路径的哪个位置。

这类题最容易答浅的地方，是把缓存注解当成“自动提速开关”。实际上 key 怎么设计、TTL 谁来配、空值要不要缓存、批量查询怎么避免 key 混乱、数据库更新后怎样避免脏数据，这些都不是注解自动替你解决的。像“方法内部自调用导致缓存注解不生效”这种边界，也是在提醒你：缓存能力最终还是靠 Spring 代理接住方法调用，不是写上去就一定生效。

继续追问时，通常会落到“@Cacheable 方法一定会执行吗”和“TTL 在注解里配置吗”。这里比较好的答法，是直接给抓手：会看缓存命中率、回源次数、Redis/Caffeine 配置、CacheManager 具体实现、是否开启了 null 值缓存、更新后删缓存的日志顺序、缓存 key 实际长什么样、同类内部调用有没有绕开代理。缓存题越靠近读写链和观测指标，越像真的做过性能治理。

如果放回业务现场，故障表现会非常真实。key 设计不稳会直接串数据；只加了 `@Cacheable` 没处理失效策略，会让更新后的旧值一直挂着；TTL 配太长会放大脏数据窗口，配太短又会让数据库频繁回源；自调用绕过代理以后，你以为加了缓存，实际上每次都在打数据库。把这些坏法说出来，比单纯介绍注解语法更有说服力。

这里还有明显的取舍。Spring Cache 最大的价值是把缓存逻辑从业务代码里抽出来，让读路径更干净；但代价是很多一致性策略被你藏到了注解和配置背后，团队如果没有统一规范，后续排障会很痛。所以它适合“读多写少、缓存边界比较清晰”的场景，不适合拿来硬扛复杂的跨服务一致性问题。

最后收口时，可以把这题讲成“读请求怎么命中缓存、写请求怎么更新或失效缓存、哪些调用会绕开代理导致缓存不生效、最终靠哪些指标确认缓存真的帮系统减压”的闭环。这样答案既有机制，也有工程判断。

## 记忆钩子

**Cache 注解不是魔法：key、失效和自调用都要讲清楚。**
