## 一句话结论

普通 @EventListener 会在发布时同步触发，可能早于事务提交；需要事务结果确定后处理时应使用 @TransactionalEventListener。

## 通俗解释

事务事件像下单后发通知，要等订单真正提交成功再发更稳。

## 面试回答

可以从这几层回答：

- ApplicationEventPublisher 默认发布后由监听器立即处理。
- @TransactionalEventListener 可以绑定 AFTER_COMMIT、AFTER_ROLLBACK 等阶段。
- 发送 MQ、短信、清缓存等副作用通常更适合提交后执行。
- 异步事件还要处理线程池、异常和上下文丢失问题。

## 常见追问

### @EventListener 默认是同步还是异步？

默认同步，在发布事件的线程里执行。需要异步时要开启异步并给监听器或执行器做配置。

### AFTER_COMMIT 适合做什么？

适合提交成功后发送 MQ、清缓存、发通知等副作用，但仍要考虑监听器失败后的重试和补偿。

### 没有事务时 TransactionalEventListener 会执行吗？

默认不执行，除非设置 fallbackExecution=true，但这样就失去事务阶段语义，要谨慎使用。

### 事务事件能替代 MQ 吗？

不能。它解决本进程内事件时机和解耦，不解决跨服务可靠投递、堆积、重放和消费确认。

## 易错点

- 事务还没提交就通知外部系统。
- 监听器异常反过来影响主事务但业务没意识到。

## 详细讲解

事务内发布 Spring 事件要注意什么，这类题本质上是在问“副作用应该挂在事务的哪个时间点”。先用一句话压住主线，比如 事务内发布事件要区分立即执行和事务提交后执行，然后马上把读者带到真实业务里：下单成功后发券、发短信、写审计日志、删缓存，这些动作到底是在数据库提交前做，还是确认提交成功后再做，结论完全不同。

最顺的讲法，是先把两个机制拆开。`@EventListener` 默认是发布后立刻同步执行，所以它和发布方共享同一线程，也可能共享同一事务语境；`@TransactionalEventListener` 则是把监听动作挂到事务阶段上，像 `AFTER_COMMIT`、`AFTER_ROLLBACK`、`BEFORE_COMMIT` 都有不同含义。这里真正值钱的不是背枚举名，而是知道“副作用为什么要等结果确定”。

这类题最容易答虚的地方，是把“事件已经发出”误当成“业务已经成功”。如果你在事务还没提交时就发 MQ、调下游、发优惠券，外部系统看到的可能是一个最终会回滚的业务结果；反过来，如果所有副作用都强行放在提交后，又要面对监听器失败没人兜底的问题。所以这题不是简单地选同步还是异步，而是在选主流程一致性和后续补偿策略。

继续深挖时，通常会落到“@EventListener 默认是同步还是异步”和“AFTER_COMMIT 适合做什么”这两个方向。这里不要只说“看日志”，而要直接说抓手：会看监听器线程名、事务日志、提交点前后的业务日志顺序、是否设置了 `fallbackExecution`、监听器异常有没有被统一接住、消息或通知有没有补偿记录。只要把这些证据说出来，面试官就知道你不是在背注解。

如果放进项目现场，故障会很具体。最常见的是订单回滚了，短信却已经发出；或者提交成功了，但提交后监听器挂掉，缓存没删、MQ 没发，主流程还以为事情已经结束。还有一种很隐蔽的坑，是监听器里做了重 IO 或远程调用，结果虽然逻辑没错，却把下单、支付这类主链路延迟拖高。读者知道这些坏法，才会真的记住事务事件为什么要谨慎。

这一层还带着明显的取舍。应用内事务事件接入轻、代码整洁，适合同进程副作用编排；但它不解决跨服务可靠投递，也不天然具备堆积、重试、死信和消费确认。很多团队会把“数据库提交后发 MQ”做成 outbox 或本地消息表，本质上就是承认：只靠 Spring 事件能解决时机问题，但解决不了全部可靠性问题。

最后收口时，可以把这题讲成一条完整链路：业务方法里什么时候发布事件，监听器在哪个事务阶段执行，失败会不会影响主事务，提交后失败又靠什么补偿，最终怎么通过日志和业务结果确认副作用真的落地。这样答案听起来就会非常像真实项目里的设计判断。

## 图解提示

适合画一张事务事件时机图：业务方法开启事务 -> 写数据库 -> 发布事件 -> 分叉成同步 `@EventListener` 立即执行 和 `@TransactionalEventListener(AFTER_COMMIT)` 提交后执行 -> 事务提交或回滚 -> 触发不同副作用。重点标清“提交前看到的状态”和“提交后才能安全执行的动作”。

## 记忆钩子

**事务事件看时机：立即监听不等提交，提交后用 TransactionalEventListener。**
