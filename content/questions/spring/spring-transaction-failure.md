## 一句话结论

Spring 事务失效的核心原因通常是事务代理没有生效，或者异常没有触发回滚规则，也可能是传播行为、数据库引擎等配置不符合预期。

## 通俗解释

Spring 事务像给方法外面套了一层“保护壳”。请求经过这层保护壳时，Spring 才能在方法前开启事务，在异常时回滚事务。

如果你从类内部直接调用方法，相当于绕过了保护壳；如果异常被你自己吞掉，Spring 也不知道出事了，自然不会回滚。

## 面试回答

常见事务失效场景有：

1. 同类内部方法调用。`this.xxx()` 不会经过 Spring 代理，`@Transactional` 不生效。
2. 方法不是 public。Spring AOP 默认主要拦截 public 方法。
3. 异常被捕获，没有继续抛出，事务拦截器感知不到异常。
4. 默认只回滚运行时异常，受检异常需要配置 `rollbackFor`。
5. 数据库表不支持事务，比如 MySQL MyISAM 引擎。
6. 事务传播行为配置不符合预期。
7. 对象没有交给 Spring 容器管理，注解自然不生效。

## 常见追问

### 为什么同类调用会失效？

Spring 声明式事务基于 AOP 代理。外部调用代理对象时，代理能增强方法；但类内部 `this` 调用直接走原对象方法，不经过代理。

### 怎么解决同类调用问题？

可以把事务方法拆到另一个 Spring Bean 中，或者通过代理对象调用，但更推荐拆分职责，让事务边界更清晰。

## 易错点

- 不要只说“注解没生效”，要说清楚代理机制。
- 不要忘记默认回滚规则是运行时异常。
- 不要忽略数据库引擎和事务传播行为。

## 详细讲解

Spring 事务失效最常见的根因，是没有理解它依赖代理机制。声明式事务不是数据库自动识别注解，而是 Spring 在 Bean 外面包了一层代理对象。当外部调用代理对象的方法时，代理会在目标方法执行前开启事务，方法正常返回后提交，抛出符合规则的异常时回滚。如果调用没有经过代理，这层事务增强就没有机会执行，注解写得再漂亮也不会生效。

第一个高频坑是同类内部调用。比如一个 Service 里的 methodA 直接调用 this.methodB，而 methodB 上有 Transactional。这个调用发生在目标对象内部，没有经过 Spring 代理，因此 methodB 的事务不会被增强。解决思路不是“多加一个注解”，而是调整调用边界：把事务方法挪到另一个 Bean，通过代理调用；或者从 Spring 容器里拿到当前代理再调用，但这通常不如拆职责清晰。

第二个坑是方法可见性和代理方式。Spring 默认基于 AOP 代理，事务通常要求 public 方法从外部被调用。private、final 方法，或者 final 类在某些代理模式下不能被正常增强。JDK 动态代理基于接口，CGLIB 基于子类，不同代理方式对方法形态有要求。面试时不用展开太底层，但要能说明“事务注解不是贴在哪都能生效，它要能被代理拦截”。

第三个坑是异常类型。Spring 默认对 RuntimeException 和 Error 回滚，对受检异常不回滚。很多业务代码 catch 了异常后只打印日志不往外抛，事务代理看到方法正常返回，就会提交。还有些代码抛的是自定义受检异常，如果没有配置 rollbackFor，也不会按预期回滚。回答时可以说：要么让异常继续抛出，要么显式设置 rollbackFor，要么在 catch 后手动标记 rollback-only，但不要吞异常。

第四个坑是传播行为。外层方法和内层方法都加事务时，REQUIRED 会加入同一个事务；REQUIRES_NEW 会挂起外层事务，新开一个事务；NESTED 则依赖保存点。传播行为用错，就会出现“我以为一起回滚，结果内层已经提交”或“内层回滚影响外层”的情况。项目里尤其要注意日志、审计、消息表、库存扣减这些动作是否应该和主业务同事务。

最后还要看数据库层。MySQL 如果表引擎是 MyISAM，本身不支持事务；如果没有被同一个事务管理器管理，或者手动拿连接提交，也会破坏事务边界。图解可以画成“外部调用 -> 代理对象 -> 开启事务 -> 目标方法 -> 异常判断 -> 提交或回滚”，旁边列出失效分支：内部调用、非 public、异常被吞、rollbackFor 缺失、传播行为不对、数据库不支持。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 Spring 事务为什么会失效 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：Spring 事务失效的核心原因通常是事务代理没有生效，或者异常没有触发回滚规则，也可能是传播行为、数据库引擎等配置不符合预期

拆开来看，第一层是背景问题：从代理机制、异常类型、传播行为和数据库引擎角度排查 Spring 事务失效。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：同类内部方法调用。this.xxx() 不会经过 Spring 代理，@Transactional 不生效。 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：方法不是 public。Spring AOP 默认主要拦截 public 方法。 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：异常被捕获，没有继续抛出，事务拦截器感知不到异常。 不要只背概念名词，要能说出触发条件、底层机制和排查入口。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：当业务代码出现 Java、Spring、事务、AOP 相关的并发异常、性能抖动、配置不生效、对象行为和预期不一致时，就可以用这道题定位原因。
- **怎么做方案**：先看触发条件，再看运行时机制。围绕“调用入口、对象状态、线程边界、框架代理、异常日志”五个位置检查，判断 Spring 事务为什么会失效 是设计问题、用法问题还是环境问题。
- **怎么验证效果**：用单元测试、压测、日志、线程栈、JFR/GC 日志或本地最小复现确认。结合真实业务代码说明什么时候会踩坑、怎么定位、怎么替换方案。
- **怎么兜底**：准备替代 API、隔离开关、降级策略、配置回滚和监控告警。面试里能讲出兜底，说明你不是只会写 happy path。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 Spring 事务为什么会失效 讲成一次工程决策，而不是一个孤立知识点。重点围绕 对象模型、运行时行为、边界条件和框架默认行为。

## 回答模板

面试时可以按这个节奏组织：

1. **先给结论**：从代理机制、异常类型、传播行为和数据库引擎角度排查 Spring 事务失效。
2. **再讲机制**：它的核心不是某个名词，而是一组处理链路。把关键角色、状态变化和触发条件说清楚。
3. **补充边界**：说明什么情况下有效，什么情况下会失效，以及为什么需要配套措施。
4. **落到项目**：如果我在项目里遇到，会先看指标和日志定位问题，再用灰度、压测和回滚策略验证方案。
5. **收一句风险**：真正重要的是不要只让功能跑通，还要保证高并发、异常分支和数据状态都可控。
