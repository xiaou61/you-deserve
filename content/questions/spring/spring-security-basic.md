## 一句话结论

认证解决“你是谁”，授权解决“你能做什么”。Spring Security 通过过滤器链在请求进入业务代码前完成安全检查。

## 通俗解释

进公司先刷工牌，这是认证；刷完工牌后能不能进机房、财务室，这是授权。

很多同学把登录成功和拥有所有权限混在一起，这是不对的。

## 面试回答

认证是确认用户身份，比如用户名密码登录、短信登录、Token 校验。授权是在身份确认后，判断这个用户是否有权限访问某个资源或执行某个操作。

Spring Security 的核心在过滤器链。请求进入 Controller 前，会先经过一系列安全过滤器，完成认证信息提取、身份校验、异常处理和权限判断。

在前后端分离项目里，常见做法是登录后发放 Token，后续请求携带 Token，服务端解析并构建认证上下文，再做权限判断。

## 常见追问

### Session 和 JWT 有什么区别？

Session 通常由服务端保存登录状态，客户端保存 session id。JWT 把部分身份信息放在令牌里，服务端可以无状态校验，但要注意过期、撤销和泄露风险。

### 权限控制放前端可以吗？

前端可以做展示控制，但真正的权限校验必须在后端做，因为前端逻辑可以被绕过。

## 易错点

- 不要把认证和授权混为一谈。
- 不要认为 JWT 天然更安全。
- 不要只在前端做权限控制。

## 详细讲解

Spring Security 里认证和授权是两个阶段。认证 authentication 解决“你是谁”，授权 authorization 解决“你能做什么”。用户输入用户名密码、携带 JWT、OAuth2 登录、短信验证码登录，这些都属于认证方式。认证成功后，系统会得到一个表示当前用户身份的 Authentication 对象，里面包含 principal、credentials、authorities 等信息，并放入 SecurityContext。

认证流程通常经过一组过滤器。请求进入 Spring Security FilterChain 后，某个认证过滤器会尝试从请求里提取凭证，比如表单登录提取用户名密码，JWT 过滤器提取 Authorization 头。然后交给 AuthenticationManager，后者再委托 AuthenticationProvider 校验凭证。校验成功后，把认证结果放入 SecurityContextHolder，后续代码就能知道当前用户是谁。

授权发生在认证之后。系统拿到当前用户的角色或权限，再根据 URL 规则、方法注解或自定义权限判断，决定是否允许访问。比如 hasRole、hasAuthority、PreAuthorize 都是授权表达。认证失败通常返回 401，表示你还没登录或凭证无效；授权失败通常返回 403，表示你已经登录但没有权限。这个区分非常适合面试回答。

角色和权限也要分清。角色通常是权限集合，比如 ADMIN、USER；权限更细，比如 order:read、order:write。简单系统可以只用角色，复杂系统通常要做 RBAC，把用户、角色、权限、资源关联起来。多租户系统还要把租户 ID、部门、数据范围纳入授权判断，否则登录成功的人可能越权访问别人的数据。

JWT 场景下，服务端可能不存 session，而是每次请求解析 token 得到用户身份。但这不代表不需要授权，token 只是认证载体，真正能不能访问接口还要看 authorities、资源归属和业务数据权限。token 过期、刷新、注销、黑名单、密钥轮换也要考虑。

图解可以画请求链：请求进入过滤器链，认证过滤器提取凭证，Provider 校验，成功后写 SecurityContext；然后进入授权判断，匹配 URL 或方法权限，成功放行，失败返回 401 或 403。最后一句：认证是门禁刷卡确认身份，授权是进门后判断你能进哪些房间。

如果还要把答案讲得更像项目经验，可以用验证闭环收束：先说明这个问题在什么条件下出现，再说线上能从哪些信号确认，最后讲处理后怎么证明风险已经下降。对这道题来说，可以围绕“把你是谁和你能做什么分清楚，理解安全过滤链的基本位置”展开，再结合 日志、单元测试、线程栈、JFR/GC 日志、最小复现和配置对照 做交叉验证。这样回答不会停在概念层，而是把发现问题、定位原因、采取动作、验证结果串成一条完整链路。最后再补一句边界：方案要和业务容忍度匹配，不能只追求理论正确，还要考虑性能、复杂度、维护成本和失败后的恢复路径。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 Spring Security 认证和授权有什么区别 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：认证解决“你是谁”，授权解决“你能做什么”

拆开来看，第一层是背景问题：把你是谁和你能做什么分清楚，理解安全过滤链的基本位置。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：认证是确认用户身份，比如用户名密码登录、短信登录、Token 校验 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：授权是在身份确认后，判断这个用户是否有权限访问某个资源或执行某个操作 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：Spring Security 的核心在过滤器链 不要只背概念名词，要能说出触发条件、底层机制和排查入口。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：当业务代码出现 Java、Spring Security、认证、授权 相关的并发异常、性能抖动、配置不生效、对象行为和预期不一致时，就可以用这道题定位原因。
- **怎么做方案**：先看触发条件，再看运行时机制。围绕“调用入口、对象状态、线程边界、框架代理、异常日志”五个位置检查，判断 Spring Security 认证和授权有什么区别 是设计问题、用法问题还是环境问题。
- **怎么验证效果**：用单元测试、压测、日志、线程栈、JFR/GC 日志或本地最小复现确认。结合真实业务代码说明什么时候会踩坑、怎么定位、怎么替换方案。
- **怎么兜底**：准备替代 API、隔离开关、降级策略、配置回滚和监控告警。面试里能讲出兜底，说明你不是只会写 happy path。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 Spring Security 认证和授权有什么区别 讲成一次工程决策，而不是一个孤立知识点。重点围绕 对象模型、运行时行为、边界条件和框架默认行为。

## 回答模板

面试时可以按这个节奏组织：

1. **先给结论**：把你是谁和你能做什么分清楚，理解安全过滤链的基本位置。
2. **再讲机制**：它的核心不是某个名词，而是一组处理链路。把关键角色、状态变化和触发条件说清楚。
3. **补充边界**：说明什么情况下有效，什么情况下会失效，以及为什么需要配套措施。
4. **落到项目**：如果我在项目里遇到，会先看指标和日志定位问题，再用灰度、压测和回滚策略验证方案。
5. **收一句风险**：真正重要的是不要只让功能跑通，还要保证高并发、异常分支和数据状态都可控。
