## 一句话结论

`@Autowired` 是 Spring 提供的，默认按类型注入；`@Resource` 来自 Java 规范，默认按名称注入，名称找不到再按类型匹配。

## 通俗解释

按类型像说“给我一个程序员”，按名称像说“我要张三”。如果有多个程序员，光说类型就可能不知道该找谁。

## 面试回答

`@Autowired` 默认按类型注入。如果同一类型有多个 Bean，需要配合 `@Qualifier` 或使用字段名、参数名辅助匹配。

`@Resource` 默认按名称查找 Bean，找不到时再按类型。它不是 Spring 独有注解。

实际项目中更推荐构造器注入，依赖更清晰，也方便测试。

## 常见追问

### 多个同类型 Bean 怎么办？

用 `@Qualifier` 指定名称，或用 `@Primary` 标记默认实现，也可以直接按名称使用 `@Resource`。

### 为什么推荐构造器注入？

必需依赖更明确，支持 final 字段，便于单测，也能更早暴露循环依赖。

### 字段注入有什么问题？

隐藏依赖、难以脱离容器测试、对象构造不完整，也不利于重构。

### @Resource 包名变化要注意什么？

Spring Boot 3/Jakarta 生态下常见 `jakarta.annotation.Resource`，旧项目可能还是 `javax.annotation.Resource`。

## 易错点

- 不要说 @Autowired 永远只按类型，不看名称。
- 不要忽略 @Resource 默认按名称。
- 不要把注入方式选择只看写法方便。

## 详细讲解

@Autowired 和 @Resource 有什么区别，这类题表面是在比两个注解，实际是在问“Spring 到底按什么规则帮你找依赖”。先用一句话压住主题，比如 区分按类型注入、按名称注入和来源规范，然后把视角放到依赖解析这一刻：容器看到一个字段、setter 或构造器参数时，到底是先按类型找、先按名字找，还是结合限定条件继续缩小范围。

更顺的讲法，是把两套解析思路拆开。`@Autowired` 更偏 Spring 自己的按类型注入体系，如果同类型候选有多个，再借助 `@Qualifier`、`@Primary`、参数名等信息继续判断；`@Resource` 则更接近“先按名称锁定，再回退到类型匹配”的语义，而且它本身来自标准注解体系。只要把“先看什么、再看什么”这条决策链说清楚，答案就已经很像工程实践。

这题最容易答混的地方，是把注入结果看成理所当然。很多启动失败都不是 Bean 没定义，而是候选太多、名称不对、默认实现没标、或者你以为字段名会自动生效，结果匹配顺序和自己想的不一样。所以面试官追这题，往往不是想听你背一句“一个按类型一个按名称”，而是看你知不知道歧义出现时容器会怎么选。

继续追问时，常见方向是“多个同类型 Bean 怎么办”和“为什么推荐构造器注入”。这里比较好的答法，是直接讲验证方式：看启动异常是不是 `NoUniqueBeanDefinitionException`，看 `@Qualifier` 和 Bean 名称是否对得上，构造器参数名有没有参与匹配，再打印最终注入对象的实际类型。依赖注入题越贴近“容器到底选中了谁”，越不像背八股。

放回项目里，差异也很实际。字段注入写起来快，但依赖隐藏、测试不友好；构造器注入会把必需依赖显式暴露出来，也更容易发现循环依赖；`@Resource` 在老项目或名称约定很强的场景里会更直观。读者一旦知道这些差异，答题时就不会只停在注解表面。

这里也有取舍。Spring 风格的 `@Autowired` 和生态整合更深，组合懒加载、条件装配、限定符时更自然；`@Resource` 语义更偏标准，但在复杂候选解析场景下通常没有 Spring 这套组合能力细。像 不要说 @Autowired 永远只按类型，不看名称 这种提醒，本质上就是在告诉你：真正重要的是匹配规则，而不是标签名字。

最后收口时，可以把这题讲成“容器如何挑选依赖、多个候选时靠什么继续缩小范围、为什么构造器注入更稳、出了问题你看什么异常和匹配规则去定位”的闭环。这样答案会明显更像真正处理过注入问题的人在讲经验。
