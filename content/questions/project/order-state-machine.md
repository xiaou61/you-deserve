## 一句话结论

订单状态机是把订单从创建、支付、发货到完成或取消的状态流转规则固定下来，保证状态只能按合法路径变化。

## 通俗解释

订单状态像坐地铁，必须按线路走。不能从“未支付”直接跳到“已完成”，也不能付款后又随便回到“待支付”。

## 面试回答

设计订单状态机要考虑：

- 状态：待支付、已支付、已发货、已完成、已取消、退款中、已退款等。
- 事件：支付成功、超时关闭、发货、确认收货、申请退款。
- 流转规则：每个事件只允许从指定状态流转到目标状态。
- 幂等控制：重复支付回调、重复取消不能导致状态错乱。
- 并发控制：同一订单状态更新要加条件更新或锁。
- 日志记录：保存状态变更历史，方便排查。

实现上可以用枚举、状态流转表或状态机框架。简单项目用枚举加合法流转校验就够。

## 常见追问

### 支付回调和超时取消同时发生怎么办？

更新订单时带上当前状态条件，比如只允许待支付订单变成已支付或已取消，谁先成功谁生效，失败方按当前状态做补偿。

### 为什么要记录状态流转日志？

方便排查订单争议、定位回调重复、分析异常流转。

## 易错点

- 不要让业务代码随意 set 状态。
- 不要忽略重复回调和并发更新。
- 状态设计要覆盖取消、退款等异常路径。

## 记忆钩子

**订单状态像地铁线路，只能按站点合法换乘。**
