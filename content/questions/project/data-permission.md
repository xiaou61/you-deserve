## 一句话结论

数据权限解决的是“用户能操作哪些数据”的问题，常按本人、部门、角色、租户、组织树等维度控制查询和操作范围。

## 通俗解释

菜单权限决定你能进哪个房间，数据权限决定你进房间后能看哪几个柜子。

## 面试回答

权限通常分两层：

- 功能权限：能不能访问某个菜单或接口。
- 数据权限：能不能看到某些数据。

常见数据范围：

- 仅本人。
- 本部门。
- 本部门及下级部门。
- 指定部门。
- 全部数据。
- 租户隔离。

实现方式可以在 SQL 层追加过滤条件，也可以用 MyBatis 插件、AOP 或统一查询组件实现。关键是避免遗漏接口和越权访问。

## 常见追问

### 这题最容易被追问的边界是什么？

重点说清重复请求、超时回调、权限越权、资金库存不一致、灰度误伤和人工修复。不要只给结论，要说明哪些条件下结论成立，哪些条件下会退化或需要换方案。

### 怎么证明自己不是在背模板？

用状态表、流水表、唯一索引、审计日志、对账结果、告警、Runbook 和回滚记录做验证。能说出具体信号、反例或排查入口，答案就从概念层进入实战层。

### 和相近方案怎么区分？

可以拿同步事务、异步补偿、状态机、Outbox、定时扫描、人工审核或多级降级对比，从一致性要求、用户体验、资金库存风险、实现成本和人工兜底成本几个维度选择。

### 面试官继续深挖时怎么展开？

先围绕“功能权限：能不能访问某个菜单或接口。”讲清主链路，再补“补 RBAC + ABAC、租户隔离、部门树闭包表/路径枚举、数据权限 SQL 条件统一注入、避免手写漏条件、权限缓存失效、越权测试、审计日志、导出接口也必须套数据权限”。如果能把边界条件、异常分支和验证闭环连起来，就能接住二面、三面的追问。

## 易错点

- 不要把菜单权限当成数据权限。
- 后端必须做权限过滤。
- 管理员、租户、部门树边界要定义清楚。

## 详细讲解

数据权限怎么设计？ 这道题现在要从“能背出来”修到“能接住追问”。核心结论是：数据权限控制用户能看哪些数据，常基于部门、角色、本人、租户等维度过滤查询范围。回答时先把它放回项目设计语境，说明它解决什么矛盾，再围绕业务状态机、幂等约束、失败补偿、灰度回滚和审计指标展开。这样能避免泛泛而谈，也能让面试官听出你知道这题的真实边界。

第一步是拆清问题背景。菜单权限决定你能进哪个房间，数据权限决定你进房间后能看哪几个柜子。在项目设计题里，背景不能写成空泛的工程套话，而要落到先讲业务目标和数据模型，再讲主链路、异常分支、补偿任务和运营后台。如果只说“看场景”或“加组件”，读者很难知道下一步该检查什么；如果能把输入、状态、依赖和输出说清，答案就有了主线。

第二步是讲机制。可以围绕这些点展开：功能权限：能不能访问某个菜单或接口。；数据权限：能不能看到某些数据。；仅本人。；本部门。。每个点都要回答三个问题：它在链路里的位置是什么，它改变了什么状态，失败或边界条件下会留下什么现象。把这三件事讲清楚，追问从概念跳到实战时就不会断。

第三步是补边界。这里尤其要主动说明重复请求、超时回调、权限越权、资金库存不一致、灰度误伤和人工修复。不要把菜单权限当成数据权限。；后端必须做权限过滤。；管理员、租户、部门树边界要定义清楚。很多八股答案的问题不是方向错，而是没有说适用范围；一旦遇到数据规模变化、异常分支或版本差异，原来的结论就可能不成立。

第四步是补专项深度：补 RBAC + ABAC、租户隔离、部门树闭包表/路径枚举、数据权限 SQL 条件统一注入、避免手写漏条件、权限缓存失效、越权测试、审计日志、导出接口也必须套数据权限。这部分适合放在面试回答后半段，用来证明你不仅知道标准答案，也知道高频追问会往哪里挖。

最后收束成一套回答节奏：先给结论，再讲机制，再补边界，最后说验证方式。对这题来说，验证可以看状态表、流水表、唯一索引、审计日志、对账结果、告警、Runbook 和回滚记录；方案选择可以和同步事务、异步补偿、状态机、Outbox、定时扫描、人工审核或多级降级对比。这样既保留八股题的清晰度，也能把答案讲成可落地、可排查、可复盘的经验。

如果面试官继续追问，可以再补一层取舍：为什么不用另一个方案、什么时候会退化、怎么发现反例或异常、出问题后如何止血。项目设计题最怕只有名词没有链路，补上业务状态机、幂等约束、失败补偿、灰度回滚和审计指标之后，答案会更稳。
