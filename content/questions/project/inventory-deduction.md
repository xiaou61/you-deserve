## 一句话结论

库存扣减的核心是防止超卖，可以用数据库条件更新保证最终安全，用 Redis 预扣提升性能，再配合消息和补偿保证一致性。

## 通俗解释

库存像电影院座位。卖票时必须确认座位还空着再锁定，不能两个人同时买到同一个座位。

## 面试回答

常见方案：

- 数据库扣减：`update stock set num = num - 1 where sku_id = ? and num > 0`，简单可靠。
- Redis 预扣：高并发下先在 Redis 扣库存，成功后异步创建订单。
- MQ 削峰：把请求写入队列，后端按能力处理。
- 幂等控制：同一用户同一活动不能重复扣。
- 补偿机制：订单超时未支付要释放库存。

如果是普通业务，数据库条件扣减通常足够；秒杀场景才需要 Redis 和 MQ 组合。

## 常见追问

### Redis 扣了库存但订单创建失败怎么办？

需要补偿回滚库存，或者通过消息和任务扫描保证最终一致。

### 什么时候扣库存更合适？

下单扣库存能减少超卖风险，支付扣库存能减少库存占用。具体看业务更怕超卖还是更怕占用。

## 易错点

- 不要先查库存再单独更新，容易并发超卖。
- 不要忽略订单取消释放库存。
- Redis 方案也要考虑数据库最终一致。

## 记忆钩子

**库存扣减先防超卖，再谈性能。**
