## 一句话结论

接口幂等性保证同一个请求重复执行多次，最终业务效果和执行一次一致。常见方案有唯一业务键、幂等 token、数据库唯一索引、状态机和去重表。

## 通俗解释

支付按钮点一次和点三次，结果都应该只扣一次钱。这就是幂等。

网络重试、用户连点、MQ 重复投递都会导致请求重复，所以后端必须兜底。

## 面试回答

常见方案：

1. 唯一业务键，比如订单号、支付流水号。
2. 数据库唯一索引，作为最终兜底。
3. 幂等 token，提交前先生成 token，提交时校验并删除。
4. 状态机，只允许合法状态流转，比如未支付才能支付。
5. 去重表，记录已处理请求 ID。
6. Redis setnx 做短期防重，但关键业务仍建议数据库兜底。

## 常见追问

### 防重复提交和幂等是一回事吗？

防重复提交是入口防护，幂等是后端业务效果保证。前端防抖不能替代后端幂等。

### 数据库唯一索引为什么重要？

因为它是强一致兜底。即使并发请求同时进来，数据库也能阻止重复插入。

## 易错点

- 不要只靠前端按钮置灰。
- 不要只靠 Redis，关键业务要有数据库兜底。
- 不要把幂等和接口不报错混为一谈。

## 详细讲解

接口幂等性要解决的问题，是同一个业务请求因为超时重试、用户连点、网关重放、消息重复消费等原因被执行多次时，系统不能重复扣款、重复下单、重复发券、重复写账。这里要先讲清楚：幂等不是“接口只能调一次”，也不是“第二次一定什么都不做”，而是“同一业务语义的重复请求，不再产生新的副作用”。  

最典型的场景有三个。第一，客户端请求超时，不知道服务端到底成功没，于是重试；第二，用户手快连点提交按钮；第三，异步消息系统为了可靠性把同一消息投递多次。这些情况你不能靠“理论上不会发生”回避，因为分布式系统里它们非常常见。  

设计幂等的第一步，通常是定义幂等键。这个键可以是业务单号、支付流水号、客户端生成的 requestId、消息唯一 ID，也可以是“用户 ID + 业务动作 + 时间窗口”组合。关键是它要能唯一标识“这是不是同一次业务请求”。有了幂等键之后，系统才能在重复请求进来时识别出来。  

具体实现方式很多，但核心都围绕“先判重，再决定执行还是返回既有结果”。常见方式有数据库唯一索引、防重表、状态机表、Redis setnx、分布式锁、消息消费去重表等。比如创建订单时用业务单号做唯一索引；支付回调时用第三方流水号去重；消息消费时先写消费记录表，写成功才继续执行。不同方案的关键差异在于一致性要求、吞吐量和故障恢复方式。  

面试里别忘了说“处理中”状态。很多人只想到成功去重，但如果一个请求第一次刚执行到一半，第二次就进来了，系统要么返回“处理中”，要么等待已有结果，而不是直接再执行一次。也就是说，幂等设计通常不是简单布尔值，而是和业务状态流转绑在一起的。  

再往工程实践里落一步，幂等一定要和事务边界一起考虑。最怕的是“判重通过了，但业务写库失败”或者“业务成功了，幂等记录没落下”，这样重复请求又会穿透进来。所以真正稳的方案，要么把判重和业务结果放进同一事务，要么设计补偿和状态恢复机制。  

面试里如果你能再补一句“幂等键生命周期也要设计”，会更完整。比如保留多久、失败记录是否允许重试、历史幂等记录怎么清理、跨天跨月的唯一性怎么保证，这些都会直接影响系统长期运行成本和正确性。  

再往前收一句也很好用：幂等是分布式系统里对“不确定请求结果”的一种工程妥协，不是为了让系统什么都不做，而是为了让重复做不会把业务做坏。  

图解可以画成一条判重链：请求进入 -> 取幂等键 -> 查幂等记录/唯一约束 -> 未处理则执行并落状态 -> 已成功则返回已有结果 -> 处理中则拒绝或等待。旁边再补一句：真正难的不是写一个锁，而是保证幂等判断和业务落库之间的原子性。这样答案会非常扎实。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 接口幂等性怎么设计 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：接口幂等性保证同一个请求重复执行多次，最终业务效果和执行一次一致

拆开来看，第一层是背景问题：掌握唯一业务键、幂等 token、数据库唯一索引和状态机等常见方案。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：唯一业务键，比如订单号、支付流水号。 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：数据库唯一索引，作为最终兜底。 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：幂等 token，提交前先生成 token，提交时校验并删除。 不要只画架构图，要把风险、取舍、回滚和监控讲完整。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：当你要把 幂等、接口设计、项目、高并发 相关能力做成真实系统，而不是只写一个 demo 时，就要回答容量、权限、一致性、降级和可观测性。
- **怎么做方案**：先明确业务目标和约束，再拆“入口流量、核心服务、数据模型、异步任务、异常补偿、监控告警、发布回滚”。
- **怎么验证效果**：不要只看功能能不能跑通，还要做压测、灰度、告警演练、失败注入和数据核对。按需求约束、核心链路、异常分支、上线验证来组织回答。
- **怎么兜底**：准备开关、限流、降级、补偿任务、人工审核、数据修复脚本和回滚预案。项目题能讲出这些，可信度会明显更高。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 接口幂等性怎么设计 讲成一次工程决策，而不是一个孤立知识点。重点围绕 业务目标、容量边界、数据一致性、灰度降级和可观测性。

## 回答模板

面试时可以按这个节奏组织：

1. **先给结论**：掌握唯一业务键、幂等 token、数据库唯一索引和状态机等常见方案。
2. **再讲机制**：它的核心不是某个名词，而是一组处理链路。把关键角色、状态变化和触发条件说清楚。
3. **补充边界**：说明什么情况下有效，什么情况下会失效，以及为什么需要配套措施。
4. **落到项目**：如果我在项目里遇到，会先看指标和日志定位问题，再用灰度、压测和回滚策略验证方案。
5. **收一句风险**：真正重要的是不要只让功能跑通，还要保证高并发、异常分支和数据状态都可控。
