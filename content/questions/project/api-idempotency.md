## 一句话结论

接口幂等性保证同一个请求重复执行多次，最终业务效果和执行一次一致。常见方案有唯一业务键、幂等 token、数据库唯一索引、状态机和去重表。

## 通俗解释

支付按钮点一次和点三次，结果都应该只扣一次钱。这就是幂等。

网络重试、用户连点、MQ 重复投递都会导致请求重复，所以后端必须兜底。

## 面试回答

常见方案：

1. 唯一业务键，比如订单号、支付流水号。
2. 数据库唯一索引，作为最终兜底。
3. 幂等 token，提交前先生成 token，提交时校验并删除。
4. 状态机，只允许合法状态流转，比如未支付才能支付。
5. 去重表，记录已处理请求 ID。
6. Redis setnx 做短期防重，但关键业务仍建议数据库兜底。

## 常见追问

### 这题最容易被追问的边界是什么？

重点说清重复请求、超时回调、权限越权、资金库存不一致、灰度误伤和人工修复。不要只给结论，要说明哪些条件下结论成立，哪些条件下会退化或需要换方案。

### 怎么证明自己不是在背模板？

用状态表、流水表、唯一索引、审计日志、对账结果、告警、Runbook 和回滚记录做验证。能说出具体信号、反例或排查入口，答案就从概念层进入实战层。

### 和相近方案怎么区分？

可以拿同步事务、异步补偿、状态机、Outbox、定时扫描、人工审核或多级降级对比，从一致性要求、用户体验、资金库存风险、实现成本和人工兜底成本几个维度选择。

### 面试官继续深挖时怎么展开？

先围绕“唯一业务键，比如订单号、支付流水号。”讲清主链路，再补“补 in-progress 状态、防并发穿透、幂等结果缓存、幂等键 TTL、请求参数摘要校验、数据库唯一约束兜底，以及支付/发券/消息消费各自的差异化设计”。如果能把边界条件、异常分支和验证闭环连起来，就能接住二面、三面的追问。

## 易错点

- 不要只靠前端按钮置灰。
- 不要只靠 Redis，关键业务要有数据库兜底。
- 不要把幂等和接口不报错混为一谈。

## 详细讲解

接口幂等性怎么设计？ 这道题现在要从“能背出来”修到“能接住追问”。核心结论是：掌握唯一业务键、幂等 token、数据库唯一索引和状态机等常见方案。回答时先把它放回项目设计语境，说明它解决什么矛盾，再围绕业务状态机、幂等约束、失败补偿、灰度回滚和审计指标展开。这样能避免泛泛而谈，也能让面试官听出你知道这题的真实边界。

第一步是拆清问题背景。支付按钮点一次和点三次，结果都应该只扣一次钱。这就是幂等。 网络重试、用户连点、MQ 重复投递都会导致请求重复，所以后端必须兜底。在项目设计题里，背景不能写成空泛的工程套话，而要落到先讲业务目标和数据模型，再讲主链路、异常分支、补偿任务和运营后台。如果只说“看场景”或“加组件”，读者很难知道下一步该检查什么；如果能把输入、状态、依赖和输出说清，答案就有了主线。

第二步是讲机制。可以围绕这些点展开：唯一业务键，比如订单号、支付流水号。；数据库唯一索引，作为最终兜底。；幂等 token，提交前先生成 token，提交时校验并删除。；状态机，只允许合法状态流转，比如未支付才能支付。。每个点都要回答三个问题：它在链路里的位置是什么，它改变了什么状态，失败或边界条件下会留下什么现象。把这三件事讲清楚，追问从概念跳到实战时就不会断。

第三步是补边界。这里尤其要主动说明重复请求、超时回调、权限越权、资金库存不一致、灰度误伤和人工修复。不要只靠前端按钮置灰。；不要只靠 Redis，关键业务要有数据库兜底。；不要把幂等和接口不报错混为一谈。很多八股答案的问题不是方向错，而是没有说适用范围；一旦遇到数据规模变化、异常分支或版本差异，原来的结论就可能不成立。

第四步是补专项深度：补 in-progress 状态、防并发穿透、幂等结果缓存、幂等键 TTL、请求参数摘要校验、数据库唯一约束兜底，以及支付/发券/消息消费各自的差异化设计。这部分适合放在面试回答后半段，用来证明你不仅知道标准答案，也知道高频追问会往哪里挖。

最后收束成一套回答节奏：先给结论，再讲机制，再补边界，最后说验证方式。对这题来说，验证可以看状态表、流水表、唯一索引、审计日志、对账结果、告警、Runbook 和回滚记录；方案选择可以和同步事务、异步补偿、状态机、Outbox、定时扫描、人工审核或多级降级对比。这样既保留八股题的清晰度，也能把答案讲成可落地、可排查、可复盘的经验。

如果面试官继续追问，可以再补一层取舍：为什么不用另一个方案、什么时候会退化、怎么发现反例或异常、出问题后如何止血。项目设计题最怕只有名词没有链路，补上业务状态机、幂等约束、失败补偿、灰度回滚和审计指标之后，答案会更稳。
