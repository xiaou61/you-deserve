## 一句话结论

普通后台列表可以用 offset 分页；数据量大或滚动加载场景更适合游标分页。深分页要避免大 offset 扫描。

## 通俗解释

offset 分页像让图书管理员先数过前 10000 本书，再给你后 10 本。页数越深，前面白数的越多。

游标分页像说“从上次看到的那本书后面继续拿 10 本”，更省力。

## 面试回答

常见分页方式：

- offset 分页：`limit offset, size`，实现简单，适合普通后台。
- 游标分页：基于上次最后一条记录的 id 或时间继续查询，性能更稳定。

深分页优化方式：

- 使用覆盖索引减少回表。
- 先查主键再回表。
- 用游标分页替代大 offset。
- 限制最大页数。

## 常见追问

### 游标分页有什么缺点？

不方便直接跳到任意页，更适合无限滚动或按时间流式浏览。

### 为什么 limit 100000, 10 慢？

数据库要扫描并丢弃前 100000 条，再返回 10 条，扫描成本高。

## 易错点

- 不要所有场景都用 offset。
- 不要忽略排序字段稳定性。
- 不要随手 select * 做深分页。

## 详细讲解

分页设计看起来简单，真正线上容易出问题的是深分页性能、排序稳定性和数据变化带来的重复漏读。最常见的 offset 分页是 limit offset, size，比如第 1000 页每页 20 条，数据库可能要先扫描并丢弃前 19980 条，再返回后 20 条。页数越深，丢弃成本越高，尤其当排序字段没有合适索引时，还可能触发 filesort 和大量回表。

offset 分页的优点是简单，适合后台管理、页数不深、数据量不大、需要跳页的场景。它的缺点是深分页慢，而且在翻页过程中如果有新增或删除，用户可能看到重复数据或漏数据。比如第一页看完后有新记录插入到最前面，第二页按 offset 查时，原第一页的最后一条可能又被推到第二页。

游标分页，也叫 seek pagination，是基于上一页最后一条记录继续查。比如按 create_time 和 id 倒序排序，下一页条件写成 create_time 小于上一页最后时间，或者 create_time 相等时 id 小于上一页最后 id。这样数据库可以沿着索引继续向后扫，不需要跳过大量行，性能更稳定。它的缺点是不方便任意跳到第 N 页，也不天然返回精确总页数。

排序条件必须稳定。只按 create_time 排序，如果同一秒有很多记录，翻页边界可能不确定，所以通常要加唯一字段做兜底排序，比如 order by create_time desc, id desc。对应索引也要配合，比如 (user_id, create_time, id) 或 (status, create_time, id)，让 where 和 order by 尽量走同一棵联合索引，减少排序和回表成本。

总数也是成本点。很多接口喜欢返回 total，但大表 count 可能很贵，尤其条件复杂时。可以根据场景选择精确 count、缓存 count、异步统计、只返回 hasMore，或者在搜索系统里使用近似总数。不要为了一个不关键的总页数拖慢核心列表接口。

图解可以画两条路径：offset 分页从索引头部扫过大量记录再丢弃，游标分页从上次最后一条继续向后读取。旁边标注三条设计原则：稳定排序、联合索引、限制深跳页。最后回答时补一句：分页不是只写 limit，而是要保证性能稳定、结果顺序稳定、数据变化时体验可接受。

如果还要把答案讲得更像项目经验，可以用验证闭环收束：先说明这个问题在什么条件下出现，再说线上能从哪些信号确认，最后讲处理后怎么证明风险已经下降。对这道题来说，可以围绕“区分 offset 分页和游标分页，并理解深分页性能问题”展开，再结合 接口访问日志、审计日志、权限变更记录、灰度记录、数据核对、告警和回滚记录 做交叉验证。这样回答不会停在概念层，而是把发现问题、定位原因、采取动作、验证结果串成一条完整链路。最后再补一句边界：方案要和业务容忍度匹配，不能只追求理论正确，还要考虑性能、复杂度、维护成本和失败后的恢复路径。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 分页查询怎么设计才稳 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：普通后台列表可以用 offset 分页；数据量大或滚动加载场景更适合游标分页

拆开来看，第一层是背景问题：区分 offset 分页和游标分页，并理解深分页性能问题。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：offset 分页：limit offset, size，实现简单，适合普通后台。 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：游标分页：基于上次最后一条记录的 id 或时间继续查询，性能更稳定。 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：使用覆盖索引减少回表。 不要只画架构图，要把风险、取舍、回滚和监控讲完整。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：当你要把 分页、接口设计、MySQL、项目 相关能力做成真实系统，而不是只写一个 demo 时，就要回答容量、权限、一致性、降级和可观测性。
- **怎么做方案**：先明确业务目标和约束，再拆“入口流量、核心服务、数据模型、异步任务、异常补偿、监控告警、发布回滚”。
- **怎么验证效果**：不要只看功能能不能跑通，还要做压测、灰度、告警演练、失败注入和数据核对。按需求约束、核心链路、异常分支、上线验证来组织回答。
- **怎么兜底**：准备开关、限流、降级、补偿任务、人工审核、数据修复脚本和回滚预案。项目题能讲出这些，可信度会明显更高。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 分页查询怎么设计才稳 讲成一次工程决策，而不是一个孤立知识点。重点围绕 业务目标、容量边界、数据一致性、灰度降级和可观测性。
