## 一句话结论

Linux 排查不要散背命令，要按方向查：CPU 看 `top`，内存看 `free`，磁盘看 `df/du`，网络看 `ss/curl`，日志看 `tail/grep/journalctl`。

## 通俗解释

线上排查像体检。你不能一上来乱开药，要先看是发烧、缺氧、堵车，还是病历里有异常。

Linux 命令就是不同检查项。

## 面试回答

可以按问题类型回答：

- CPU：`top`、`ps`，看哪个进程占用高。
- 内存：`free -h`、`top`，看内存和 swap。
- 磁盘空间：`df -h` 看分区，`du -sh *` 找大目录。
- 网络端口：`ss -lntp` 看监听端口，`curl` 验证接口。
- 日志：`tail -f` 实时看日志，`grep` 搜关键错误。
- 进程：`ps -ef` 查进程，`kill` 处理异常进程。

如果是 Java 服务，还可以补充 `jps`、`jstack`、`jmap` 这类 JVM 工具，但要注意生产环境使用风险。

## 常见追问

### 接口突然变慢怎么排查？

先确认是应用慢、数据库慢、网络慢还是机器资源瓶颈。看监控和日志，再查 CPU、内存、线程、慢 SQL、下游依赖。

### 磁盘满了怎么办？

先用 `df -h` 确认哪个分区满，再用 `du` 找大目录。日志文件要通过规范方式清理或轮转，不要盲目删业务数据。

## 易错点

- 不要只背命令，要说排查思路。
- 不要在线上随便执行高风险命令。
- 不要忽略日志和监控。

## 详细讲解

线上 Linux 服务排查，最重要的是先定方向，不要一上来就乱敲命令。先问清楚现象：是接口变慢、错误率升高、进程退出、CPU 高、内存高、磁盘满、连接超时，还是某个下游异常。然后按 CPU、内存、磁盘、网络、日志五条线展开。命令只是工具，排查顺序和证据链更重要。

CPU 高时，先用 top 看整体负载和哪个进程占用高，再用 top -H -p pid 或 pidstat 看线程级别。Java 服务可以把线程 ID 转成十六进制，在 jstack 里找对应线程栈，判断是业务循环、锁竞争、GC 线程、正则、序列化还是加密计算。load average 高不一定等于 CPU 忙，也可能是不可中断 IO 等待，要结合 us、sy、wa、队列长度一起看。

内存问题先用 free 看系统可用内存，用 ps 或 top 看进程 RSS，用 pmap 或 smaps 看进程内存分布。Java 服务要区分堆、元空间、直接内存、线程栈、mmap 和 native 内存。系统发生 OOM 时，要看 dmesg 或 journalctl 里的 OOM killer 日志，确认是谁被杀、当时内存状态如何。不要只看 JVM 堆没满就排除内存问题。

磁盘问题常见是 df 显示分区满，du 定位目录，lsof | grep deleted 找已删除但仍被进程占用的文件。iostat 可以看磁盘 IO 是否打满，await、util 等指标能反映等待。日志暴涨、临时文件、上传目录、数据库文件都可能撑爆磁盘。磁盘满会引发写日志失败、数据库不可写、服务启动失败等连锁问题。

网络问题可以用 ss 或 netstat 看监听端口、连接状态、TIME_WAIT、CLOSE_WAIT、ESTABLISHED 数量。curl 可以验证本机到目标服务是否通，telnet 或 nc 可以测端口，tcpdump 可以抓包看握手、重传、RST。DNS 问题用 nslookup 或 dig，路由问题用 ip route、traceroute。连接池耗尽和网络不通表现都可能是超时，需要结合服务端日志和连接状态判断。

图解可以画成排查漏斗：现象 -> 资源方向 -> 命令定位 -> 日志确认 -> 临时止血 -> 根因复盘。每个方向列 2 到 3 个核心命令。最后强调：线上排查要先止血，比如扩容、重启、清理磁盘、限流、回滚；再保留现场和日志做根因分析，避免只靠重启解决一切。复盘时要补监控、告警阈值和自动化脚本，让同类问题下次更早暴露、更快恢复。

面试时可以按一次真实故障来讲：先看告警和时间线，再定位资源瓶颈，然后验证假设，最后止血和复盘。这比单纯背命令列表更像做过线上运维。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 线上 Linux 服务排查常用哪些命令 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：Linux 排查不要散背命令，要按方向查：CPU 看 top，内存看 free，磁盘看 df/du，网络看 ss/curl，日志看 tail/grep/journalctl

拆开来看，第一层是背景问题：按 CPU、内存、磁盘、网络、日志几个方向组织 Linux 排查命令。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：CPU：top、ps，看哪个进程占用高。 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：内存：free -h、top，看内存和 swap。 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：磁盘空间：df -h 看分区，du -sh * 找大目录。 不要只画架构图，要把风险、取舍、回滚和监控讲完整。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：当你要把 Linux、排查、部署、工程化 相关能力做成真实系统，而不是只写一个 demo 时，就要回答容量、权限、一致性、降级和可观测性。
- **怎么做方案**：先明确业务目标和约束，再拆“入口流量、核心服务、数据模型、异步任务、异常补偿、监控告警、发布回滚”。
- **怎么验证效果**：不要只看功能能不能跑通，还要做压测、灰度、告警演练、失败注入和数据核对。按需求约束、核心链路、异常分支、上线验证来组织回答。
- **怎么兜底**：准备开关、限流、降级、补偿任务、人工审核、数据修复脚本和回滚预案。项目题能讲出这些，可信度会明显更高。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 线上 Linux 服务排查常用哪些命令 讲成一次工程决策，而不是一个孤立知识点。重点围绕 业务目标、容量边界、数据一致性、灰度降级和可观测性。

## 回答模板

面试时可以按这个节奏组织：

1. **先给结论**：按 CPU、内存、磁盘、网络、日志几个方向组织 Linux 排查命令。
2. **再讲机制**：它的核心不是某个名词，而是一组处理链路。把关键角色、状态变化和触发条件说清楚。
3. **补充边界**：说明什么情况下有效，什么情况下会失效，以及为什么需要配套措施。
4. **落到项目**：如果我在项目里遇到，会先看指标和日志定位问题，再用灰度、压测和回滚策略验证方案。
5. **收一句风险**：真正重要的是不要只让功能跑通，还要保证高并发、异常分支和数据状态都可控。
