## 一句话结论

Linux 排查不要散背命令，要按方向查：CPU 看 `top`，内存看 `free`，磁盘看 `df/du`，网络看 `ss/curl`，日志看 `tail/grep/journalctl`。

## 通俗解释

线上排查像体检。你不能一上来乱开药，要先看是发烧、缺氧、堵车，还是病历里有异常。

Linux 命令就是不同检查项。

## 面试回答

可以按问题类型回答：

- CPU：`top`、`ps`，看哪个进程占用高。
- 内存：`free -h`、`top`，看内存和 swap。
- 磁盘空间：`df -h` 看分区，`du -sh *` 找大目录。
- 网络端口：`ss -lntp` 看监听端口，`curl` 验证接口。
- 日志：`tail -f` 实时看日志，`grep` 搜关键错误。
- 进程：`ps -ef` 查进程，`kill` 处理异常进程。

如果是 Java 服务，还可以补充 `jps`、`jstack`、`jmap` 这类 JVM 工具，但要注意生产环境使用风险。

## 常见追问

### 这题最容易被追问的边界是什么？

重点说清配置漂移、资源打满、探针误判、发布失败、依赖异常和回滚窗口。不要只给结论，要说明哪些条件下结论成立，哪些条件下会退化或需要换方案。

### 怎么证明自己不是在背模板？

用发布记录、容器状态、日志、CPU、内存、磁盘、网络、P95/P99 和告警做验证。能说出具体信号、反例或排查入口，答案就从概念层进入实战层。

### 和相近方案怎么区分？

可以拿蓝绿、金丝雀、滚动发布、限流、降级、扩容、回滚或手工止血对比，从发布风险、恢复速度、资源成本、用户影响和运维复杂度几个维度选择。

### 面试官继续深挖时怎么展开？

先围绕“CPU：top、ps，看哪个进程占用高。”讲清主链路，再补“把 追问准备 改成 Linux 专项：CPU 高、内存泄漏、磁盘满、端口不通、日志暴涨分别怎么排查”。如果能把边界条件、异常分支和验证闭环连起来，就能接住二面、三面的追问。

## 易错点

- 不要只背命令，要说排查思路。
- 不要在线上随便执行高风险命令。
- 不要忽略日志和监控。

## 详细讲解

线上 Linux 服务排查常用哪些命令？ 这道题现在要从“能背出来”修到“能接住追问”。核心结论是：按 CPU、内存、磁盘、网络、日志几个方向组织 Linux 排查命令。回答时先把它放回工程化语境，说明它解决什么矛盾，再围绕发布流程、资源边界、排障命令、回滚策略和监控告警展开。这样能避免泛泛而谈，也能让面试官听出你知道这题的真实边界。

第一步是拆清问题背景。线上排查像体检。你不能一上来乱开药，要先看是发烧、缺氧、堵车，还是病历里有异常。 Linux 命令就是不同检查项。在工程化题里，背景不能写成空泛的工程套话，而要落到按环境、镜像、配置、健康检查、流量切换和回滚链路展开。如果只说“看场景”或“加组件”，读者很难知道下一步该检查什么；如果能把输入、状态、依赖和输出说清，答案就有了主线。

第二步是讲机制。可以围绕这些点展开：CPU：top、ps，看哪个进程占用高。；内存：free -h、top，看内存和 swap。；磁盘空间：df -h 看分区，du -sh * 找大目录。；网络端口：ss -lntp 看监听端口，curl 验证接口。。每个点都要回答三个问题：它在链路里的位置是什么，它改变了什么状态，失败或边界条件下会留下什么现象。把这三件事讲清楚，追问从概念跳到实战时就不会断。

第三步是补边界。这里尤其要主动说明配置漂移、资源打满、探针误判、发布失败、依赖异常和回滚窗口。不要只背命令，要说排查思路。；不要在线上随便执行高风险命令。；不要忽略日志和监控。很多八股答案的问题不是方向错，而是没有说适用范围；一旦遇到数据规模变化、异常分支或版本差异，原来的结论就可能不成立。

第四步是补专项深度：把 追问准备 改成 Linux 专项：CPU 高、内存泄漏、磁盘满、端口不通、日志暴涨分别怎么排查。这部分适合放在面试回答后半段，用来证明你不仅知道标准答案，也知道高频追问会往哪里挖。

最后收束成一套回答节奏：先给结论，再讲机制，再补边界，最后说验证方式。对这题来说，验证可以看发布记录、容器状态、日志、CPU、内存、磁盘、网络、P95/P99 和告警；方案选择可以和蓝绿、金丝雀、滚动发布、限流、降级、扩容、回滚或手工止血对比。这样既保留八股题的清晰度，也能把答案讲成可落地、可排查、可复盘的经验。

如果面试官继续追问，可以再补一层取舍：为什么不用另一个方案、什么时候会退化、怎么发现反例或异常、出问题后如何止血。工程化题最怕只有名词没有链路，补上发布流程、资源边界、排障命令、回滚策略和监控告警之后，答案会更稳。
