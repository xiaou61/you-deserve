## 一句话结论

CPU 飙高排查要按“机器 -> 进程 -> 线程 -> 代码栈 -> 业务原因”逐层缩小范围。

## 通俗解释

它像找教室里谁在吵。先确定哪栋楼吵，再确定哪间教室，最后找到具体是哪几个同学在说话。

## 面试回答

Java 服务 CPU 高常见排查路径：

1. 用 `top` 看整体 CPU 和高占用进程。
2. 用 `top -Hp <pid>` 或类似方式定位高 CPU 线程。
3. 把线程 id 转成十六进制。
4. 用 `jstack <pid>` 查对应线程栈。
5. 判断是死循环、锁竞争、热点计算、频繁 GC 还是外部调用异常。

如果是容器环境，还要结合容器 CPU 限额、节点资源和监控看是否被限流。

## 常见追问

### 这题最容易被追问的边界是什么？

重点说清配置漂移、资源打满、探针误判、发布失败、依赖异常和回滚窗口。不要只给结论，要说明哪些条件下结论成立，哪些条件下会退化或需要换方案。

### 怎么证明自己不是在背模板？

用发布记录、容器状态、日志、CPU、内存、磁盘、网络、P95/P99 和告警做验证。能说出具体信号、反例或排查入口，答案就从概念层进入实战层。

### 和相近方案怎么区分？

可以拿蓝绿、金丝雀、滚动发布、限流、降级、扩容、回滚或手工止血对比，从发布风险、恢复速度、资源成本、用户影响和运维复杂度几个维度选择。

### 面试官继续深挖时怎么展开？

先围绕“用 top 看整体 CPU 和高占用进程。”讲清主链路，再补“补 pidstat、jcmd Thread.print、GC 日志/jstat、容器 CPU throttle、perf top 或火焰图、保留现场后再重启”。如果能把边界条件、异常分支和验证闭环连起来，就能接住二面、三面的追问。

## 详细讲解

线上 CPU 飙高怎么排查？ 这道题现在要从“能背出来”修到“能接住追问”。核心结论是：CPU 飙高要先定位进程，再定位线程，最后结合栈信息判断是死循环、热点计算还是 GC。回答时先把它放回工程化语境，说明它解决什么矛盾，再围绕发布流程、资源边界、排障命令、回滚策略和监控告警展开。这样能避免泛泛而谈，也能让面试官听出你知道这题的真实边界。

第一步是拆清问题背景。它像找教室里谁在吵。先确定哪栋楼吵，再确定哪间教室，最后找到具体是哪几个同学在说话。在工程化题里，背景不能写成空泛的工程套话，而要落到按环境、镜像、配置、健康检查、流量切换和回滚链路展开。如果只说“看场景”或“加组件”，读者很难知道下一步该检查什么；如果能把输入、状态、依赖和输出说清，答案就有了主线。

第二步是讲机制。可以围绕这些点展开：用 top 看整体 CPU 和高占用进程。；用 top -Hp 或类似方式定位高 CPU 线程。；把线程 id 转成十六进制。；用 jstack 查对应线程栈。。每个点都要回答三个问题：它在链路里的位置是什么，它改变了什么状态，失败或边界条件下会留下什么现象。把这三件事讲清楚，追问从概念跳到实战时就不会断。

第三步是补边界。这里尤其要主动说明配置漂移、资源打满、探针误判、发布失败、依赖异常和回滚窗口。不要直接重启了事，要保留现场。；线程 id 需要转十六进制才能和 jstack 对上。；CPU 高要结合业务流量和 GC 指标看。很多八股答案的问题不是方向错，而是没有说适用范围；一旦遇到数据规模变化、异常分支或版本差异，原来的结论就可能不成立。

第四步是补专项深度：补 pidstat、jcmd Thread.print、GC 日志/jstat、容器 CPU throttle、perf top 或火焰图、保留现场后再重启。这部分适合放在面试回答后半段，用来证明你不仅知道标准答案，也知道高频追问会往哪里挖。

最后收束成一套回答节奏：先给结论，再讲机制，再补边界，最后说验证方式。对这题来说，验证可以看发布记录、容器状态、日志、CPU、内存、磁盘、网络、P95/P99 和告警；方案选择可以和蓝绿、金丝雀、滚动发布、限流、降级、扩容、回滚或手工止血对比。这样既保留八股题的清晰度，也能把答案讲成可落地、可排查、可复盘的经验。

如果面试官继续追问，可以再补一层取舍：为什么不用另一个方案、什么时候会退化、怎么发现反例或异常、出问题后如何止血。工程化题最怕只有名词没有链路，补上发布流程、资源边界、排障命令、回滚策略和监控告警之后，答案会更稳。
