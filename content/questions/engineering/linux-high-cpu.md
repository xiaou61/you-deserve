## 一句话结论

CPU 飙高排查要按“机器 -> 进程 -> 线程 -> 代码栈 -> 业务原因”逐层缩小范围。

## 通俗解释

它像找教室里谁在吵。先确定哪栋楼吵，再确定哪间教室，最后找到具体是哪几个同学在说话。

## 面试回答

Java 服务 CPU 高常见排查路径：

1. 用 `top` 看整体 CPU 和高占用进程。
2. 用 `top -Hp <pid>` 或类似方式定位高 CPU 线程。
3. 把线程 id 转成十六进制。
4. 用 `jstack <pid>` 查对应线程栈。
5. 判断是死循环、锁竞争、热点计算、频繁 GC 还是外部调用异常。

如果是容器环境，还要结合容器 CPU 限额、节点资源和监控看是否被限流。

## 常见追问

### jstack 看到线程一直 RUNNABLE 说明什么？

说明线程正在运行或等待 CPU，不一定就是 bug，要结合栈顶方法、持续时间和业务流量判断。

### CPU 高一定是代码死循环吗？

不一定，也可能是流量突增、序列化开销、正则回溯、加解密、GC 或容器 CPU 配额过低。

## 易错点

- 不要直接重启了事，要保留现场。
- 线程 id 需要转十六进制才能和 jstack 对上。
- CPU 高要结合业务流量和 GC 指标看。
