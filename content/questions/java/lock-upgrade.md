## 一句话结论

synchronized 会根据竞争情况从无锁、偏向锁、轻量级锁逐步升级到重量级锁，目标是在低竞争场景减少阻塞和系统调用成本。

## 通俗解释

一开始会议室没人抢，常来的那个人可以默认使用。后来有人偶尔竞争，就先用轻量方式协调。竞争激烈后，只能请管理员正式排队管理。

## 面试回答

锁升级大致过程：

- 无锁：对象没有被线程占用。
- 偏向锁：偏向第一个获得锁的线程，适合几乎没有竞争的场景。
- 轻量级锁：多个线程交替竞争时，通过 CAS 自旋尝试获取。
- 重量级锁：竞争激烈时，线程阻塞，由操作系统互斥量参与。

升级通常不可逆，目的是根据竞争程度选择成本更合适的同步方式。

## 常见追问

### 自旋一定好吗？

不一定。竞争时间短时自旋避免阻塞切换；竞争时间长时自旋会浪费 CPU。

### 现在偏向锁还重要吗？

不同 JDK 版本对偏向锁支持和默认策略不同，面试里重点理解锁优化思想，不要死背版本行为。

## 易错点

- 不要说 synchronized 一定是重量级锁。
- 不要忽略 JVM 对 synchronized 的优化。
- 不要脱离 JDK 版本谈细节。

## 详细讲解

synchronized 锁升级过程这道题，核心不是让你背对象头 bit 位，而是理解 JVM 为什么不一开始就上最重的锁。因为线程竞争强度并不总是一样：有的同步块几乎永远只有一个线程进，有的只是偶发短暂竞争，有的则是真的很多线程同时抢。JVM 为了在不同场景下平衡开销，才设计了从偏向到轻量到重量的逐步升级思路。  

最开始可以理解为无锁状态，对象还没被用于同步。偏向锁的设计假设是：很多同步块其实总是同一个线程反复进入，如果每次都做完整竞争流程太浪费，所以干脆把对象“偏向”某个线程。后续还是这个线程来进同步块时，基本不用再做昂贵竞争判断，成本很低。  

当出现第二个线程来竞争时，偏向锁就不再适合了。JVM 会撤销偏向，转向轻量级锁。轻量级锁通常会借助线程栈中的锁记录和 CAS 尝试，在用户态尽量快速完成竞争。如果竞争只是很短暂，比如另一个线程很快就释放，自旋几次可能就拿到锁，这比直接挂起唤醒线程更划算。  

如果竞争继续加剧，自旋多次也拿不到锁，JVM 就会把锁膨胀成重量级锁，底层更依赖操作系统互斥量和线程阻塞唤醒机制。重量级锁的特点是切换和调度成本更高，但在高竞争场景下更稳定，因为它不会让线程一直白白自旋耗 CPU。  

面试里最好补一句：锁升级的目标是让低竞争场景尽量轻、让高竞争场景能扛住，不是让开发者手工去“选择升级到哪一层”。而且随着 JDK 版本演进，一些实现细节和优化策略也会变化，所以比起死记某个位模式，更重要的是知道它背后的性能权衡逻辑。  

工程里再补一句也很加分：如果真的出现锁竞争严重，业务层更该关注的是缩小临界区、减少共享状态、换更合适的并发模型，而不是幻想靠理解锁升级就能自动把问题解决。JVM 优化能帮忙，但设计层面的竞争才是根因。  

如果想再把这题答得更贴近排障，可以补一句：线上看见 blocked 线程很多、CPU 空转、自旋严重时，真正关心的是竞争是否过高以及锁粒度是否过粗，而不是去猜对象头现在到底是哪一种位模式。这个角度非常务实。  

再补一句也很适合收尾：锁升级这套机制说明 JVM 已经在尽力帮你优化同步成本，所以当业务层仍然卡在锁竞争上，往往意味着问题已经不在语法层，而在共享设计本身。把这句话说出来，会显得你很有工程判断。  

图解可以画成一条升级链：无锁 -> 偏向锁 -> 轻量级锁 -> 重量级锁，每一步旁边都写上“适合什么竞争强度”和“代价是什么”。最后收一句：锁升级是 JVM 根据竞争情况自动切换同步策略，本质是在拿不同成本换不同场景下的吞吐和延迟表现。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 synchronized 锁升级过程是什么 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：synchronized 会根据竞争情况从无锁、偏向锁、轻量级锁逐步升级到重量级锁，目标是在低竞争场景减少阻塞和系统调用成本

拆开来看，第一层是背景问题：理解偏向锁、轻量级锁、重量级锁的演进思路和性能优化目标。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：无锁：对象没有被线程占用。 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：偏向锁：偏向第一个获得锁的线程，适合几乎没有竞争的场景。 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：轻量级锁：多个线程交替竞争时，通过 CAS 自旋尝试获取。 不要只背概念名词，要能说出触发条件、底层机制和排查入口。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：当业务代码出现 Java、synchronized、锁升级、JVM 相关的并发异常、性能抖动、配置不生效、对象行为和预期不一致时，就可以用这道题定位原因。
- **怎么做方案**：先看触发条件，再看运行时机制。围绕“调用入口、对象状态、线程边界、框架代理、异常日志”五个位置检查，判断 synchronized 锁升级过程是什么 是设计问题、用法问题还是环境问题。
- **怎么验证效果**：用单元测试、压测、日志、线程栈、JFR/GC 日志或本地最小复现确认。结合真实业务代码说明什么时候会踩坑、怎么定位、怎么替换方案。
- **怎么兜底**：准备替代 API、隔离开关、降级策略、配置回滚和监控告警。面试里能讲出兜底，说明你不是只会写 happy path。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 synchronized 锁升级过程是什么 讲成一次工程决策，而不是一个孤立知识点。重点围绕 对象模型、运行时行为、边界条件和框架默认行为。
