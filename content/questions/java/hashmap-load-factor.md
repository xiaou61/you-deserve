## 一句话结论

0.75 不是数学定理，而是 JDK 在空间占用、扩容频率和冲突概率之间做出的工程折中：再小会更浪费，再大链冲突和树化概率都会上升。

## 通俗解释

储物柜不能刚放一半就换大柜，也不能塞到快爆才换。0.75 就像柜子用到七八成时扩容，比较均衡。

## 面试回答

这题别只背“经验值”，要把它和扩容行为绑起来说：

1. 负载因子决定阈值，公式是 `capacity * loadFactor`。
2. 负载因子小，桶还没怎么用满就扩容，内存浪费但碰撞少。
3. 负载因子大，空间利用率更高，但链表变长、树化概率上升，查询和迁移成本都会变差。
4. 默认 0.75 代表 JDK 认为“大多数通用业务”里，这是时间和空间较平衡的点；优化时通常先调初始容量，而不是先改负载因子。

## 常见追问

### 为什么不直接把默认值设成 1？

因为那等于尽量把桶塞满再扩容，空间是省了，但碰撞会明显增加。HashMap 的查询效率依赖分布够散，链越长，CPU cache 命中和遍历成本都会变差。

### 什么时候会考虑改负载因子？

极少数对空间特别敏感、并且 key 分布稳定的场景才会考虑；大多数业务更推荐按预估元素量设置初始容量。面试里主动说“生产里改默认负载因子不是第一优先级”，会显得更稳。

### 它和树化阈值有什么关系？

负载因子越大，冲突链变长的概率越高，更容易逼近树化条件；但树化还受桶数组长度限制，不是单纯链长到了就一定变红黑树。这个点能说明你知道 HashMap 在看多个条件，不是单点判断。

### 怎么讲这题才不空？

可以举一个 100 万条数据的缓存例子：如果初始容量太小，会在装载过程中多次扩容；如果乱把负载因子调高，链冲突和 RT 又会上去。把“空间、时间、扩容次数”三个指标一起讲，答案就落地了。

## 易错点

- 负载因子不是越大越好。
- 扩容阈值不是固定值，而是容量乘负载因子。
- 优化 HashMap 常先设置初始容量，而不是乱改负载因子。
