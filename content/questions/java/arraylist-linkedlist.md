## 一句话结论

两者的核心差异不是“一个增删快一个查询快”这么粗，而是数组连续内存和链表节点跳转带来的访问路径、缓存命中率和真实业务成本完全不同。

## 通俗解释

ArrayList 像一排连续座位，你知道第几个座位就能直接走过去。LinkedList 像一串手拉手的人，想找第 N 个通常要一个个数过去。

所以 ArrayList 查得快，LinkedList 找位置慢。链表插入删除本身快，但前提是你已经找到了位置。

## 面试回答

面试里建议直接按“结构 -> 操作 -> 实际选型”来答：

1. ArrayList 基于动态数组，按下标随机访问快，CPU cache 友好。
2. LinkedList 基于双向链表，头尾插删容易，但按索引访问必须走节点。
3. ArrayList 在中间插入删除要搬移元素；LinkedList 虽然改指针便宜，但前提是你已经遍历到目标位置。
4. 真实业务里，大多数场景更常用 ArrayList；只有明确存在头尾频繁操作或需要队列/双端队列语义时，才会考虑 LinkedList。

## 常见追问

### 为什么很多时候 LinkedList 理论上增删快，实践里却不占优？

因为“找到位置”本身就要遍历，节点又分散在堆上，缓存局部性差。真实系统里内存访问和对象分配成本常常比“改一下指针”更贵，所以面试里不能只背大 O 结论。

### 如果我要实现队列，用哪个更合适？

如果只是 FIFO，优先看 `ArrayDeque`，通常比 LinkedList 更轻；LinkedList 的优势在于它天然实现了 Deque 接口，但每个节点都有额外指针开销。能主动提 ArrayDeque，会比只在两者里二选一更像做过工程的人。

### ArrayList 的扩容要不要顺带讲？

值得提一句。因为很多业务最终选 ArrayList，面试官常会继续问扩容、初始容量和 fail-fast。你可以顺势说“它常用，但也有扩容拷贝成本”，让答案更完整。

### 哪些信号说明你真的应该换成 LinkedList？

比如确实存在大量头尾插删、且元素量不大、又不关心随机访问。这种条件同时满足才更像 LinkedList 的主场；如果只是“可能会增删”，通常还不够。

## 易错点

- 不要简单背“ArrayList 查快，LinkedList 增删快”，这句话太粗。
- 不要忽略 LinkedList 的节点额外内存开销。
- 不要忘记 ArrayList 在真实业务里通常更常用。
