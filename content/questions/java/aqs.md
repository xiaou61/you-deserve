## 一句话结论

AQS 不是某一种锁，而是一套“共享状态 + 等待队列 + 获取/释放模板方法”的同步器骨架，ReentrantLock、Semaphore、CountDownLatch 都是在它上面长出来的。

## 通俗解释

AQS 像排队办业务的大厅。state 表示窗口是否可用或还有几个名额，拿不到资源的人进入队列等待，轮到谁再被唤醒。

## 面试回答

这题最好按“state、队列、模板方法”三件套来答：

1. `state` 表示同步状态，子类通过 CAS 或特定规则修改它。
2. 获取资源失败的线程会封装成 Node 挂到等待队列里，随后 park 挂起。
3. 资源释放后，AQS 会按规则唤醒后继节点继续竞争。
4. 子类不是重写整个并发框架，而是重点实现 `tryAcquire/tryRelease` 或共享模式下对应方法，所以它非常适合做锁和同步器模板。

## 常见追问

### 为什么说 AQS 是“框架”而不是“锁”？

因为它并不直接决定你是独占还是共享、可重入还是不可重入、公平还是非公平；这些都是子类在 state 语义和获取释放规则上再定义出来的。把这层抽象讲出来，答案就立住了。

### CLH 队列在这题里要讲到什么程度？

不用把每个字段都背全，但至少要知道它是一个等待线程的双向队列，节点拿不到锁会挂起，前驱释放后再被唤醒。能说出 `waitStatus`、前驱/后继关系，就已经比纯背概念更深一层。

### 公平锁和非公平锁在 AQS 里怎么落地？

核心差别在于获取资源前是否先看队列中有没有前驱节点，例如 `hasQueuedPredecessors()`。这句要是能自然说出来，面试官一般会点头。

### `Condition` 和同步队列是什么关系？

Condition 有自己的等待队列，线程先从同步队列获取锁，调用 `await` 后转去 Condition 队列，`signal` 再转回同步队列重新竞争。这个过程很适合拿来接深挖。

## 易错点

- 不要只背 AQS 全称，要讲 state 和队列。
- 不要把 AQS 说成某一种锁。
- 不要忽略 CAS 和阻塞唤醒配合。
