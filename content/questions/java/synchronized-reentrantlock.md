## 一句话结论

`synchronized` 是 JVM 内置锁，使用简单，自动释放；`ReentrantLock` 是显式锁，功能更灵活，支持可中断、公平锁、多个条件队列，但必须手动释放。

## 通俗解释

`synchronized` 像自动门，进出规则简单，走完自动关门。`ReentrantLock` 像手动门锁，能设置更多规则，但你必须记得关门。

## 面试回答

区别可以从几个方面说：

1. 使用方式：synchronized 是关键字；ReentrantLock 是类。
2. 释放方式：synchronized 自动释放；ReentrantLock 要在 finally 中 unlock。
3. 可中断：ReentrantLock 支持等待锁时响应中断。
4. 公平性：ReentrantLock 可以选择公平锁；synchronized 不支持手动设置公平。
5. 条件队列：ReentrantLock 可以创建多个 Condition。
6. 实现层面：synchronized 由 JVM 支持，ReentrantLock 基于 AQS。

一般简单同步优先 synchronized；需要高级能力时再考虑 ReentrantLock。

## 常见追问

### ReentrantLock 为什么要放 finally 解锁？

因为它不会自动释放锁。如果业务代码抛异常但没有 unlock，可能导致其他线程永远拿不到锁。

### synchronized 性能是不是一定差？

不是。现代 JVM 对 synchronized 做了很多优化，普通场景完全可以使用。

## 易错点

- 不要说 ReentrantLock 一定比 synchronized 快。
- 不要忘记 finally 释放锁。
- 不要只背区别，要说选择建议。

## 详细讲解

synchronized 和 ReentrantLock 都能实现互斥，并且都是可重入锁。可重入的意思是同一个线程已经拿到某把锁后，再次进入需要同一把锁的代码不会把自己锁死，而是计数加一，退出时逐层释放。它们都能保证临界区内的原子性、可见性和有序性，区别主要在使用方式、能力边界和可控制程度。

synchronized 是 JVM 关键字，可以修饰实例方法、静态方法或代码块。锁对象可以是当前实例、Class 对象或任意对象。它的优点是语法简单，进入代码块自动加锁，退出时无论正常返回还是抛异常都会自动释放锁，不容易忘记 unlock。现代 JVM 对 synchronized 做了大量优化，比如偏向锁曾经存在、轻量级锁、锁粗化、锁消除、自适应自旋等，所以不能再简单说 synchronized 性能一定差。

ReentrantLock 是 JUC 包里的显式锁，需要手动 lock 和 unlock。它的优点是功能更丰富：tryLock 可以尝试获取锁，拿不到就返回，避免永久等待；lockInterruptibly 可以响应中断；构造时可以选择公平锁，让等待时间更久的线程优先获得锁；Condition 可以创建多个条件队列，比 synchronized 的 wait/notify 更灵活。缺点是必须在 finally 里 unlock，否则异常路径可能导致锁永远不释放。

选择时可以按复杂度判断。普通临界区、简单同步、方法级互斥，优先 synchronized，代码更短，释放更安全。如果需要超时获取锁、可中断等待、公平锁、多条件队列，选择 ReentrantLock。比如生产者消费者有多个等待条件，用 Condition 会更清晰；需要避免死等，用 tryLock 加超时更安全。

面试里还可以补充锁和等待通知的关系。synchronized 搭配 wait、notify、notifyAll，要求在持有对象监视器时调用；ReentrantLock 搭配 Condition 的 await、signal、signalAll，一个锁可以创建多个 Condition，把不同等待队列分开。这样能避免 notifyAll 唤醒一堆无关线程。

图解可以画左右对比：左边 synchronized 进入代码块自动加锁，退出自动释放；右边 ReentrantLock 显式 lock，try/finally unlock。中间列能力差异：自动释放、可中断、公平性、tryLock、多个条件队列。最后强调：二者不是谁完全替代谁，简单场景用内置锁，复杂并发控制用显式锁。
