## 一句话结论

拒绝策略不是背四个类名，而是要说明系统满载时你是抛错、反压、丢新任务还是丢旧任务，以及业务能不能承受这个后果。

## 通俗解释

拒绝策略像餐厅满座后的处理方式：拒客、让老板接待、丢老单或静默丢弃。

## 面试回答

建议按“默认行为 + 业务取舍”来答：

1. `AbortPolicy` 直接抛 `RejectedExecutionException`，最利于显式失败。
2. `CallerRunsPolicy` 让提交线程自己执行任务，相当于把压力反推给上游。
3. `DiscardPolicy` 静默丢弃，除非业务天然允许丢，否则风险很高。
4. `DiscardOldestPolicy` 会丢队列中最旧的任务，更像为“过期数据不值得做”这类特殊场景准备。
5. 真正上线时，拒绝策略通常要配合监控、日志和自定义降级动作，而不是只靠 JDK 默认值。

## 常见追问

### 为什么很多业务更偏向显式失败？

因为抛错最容易被调用方感知，也便于监控和补偿。静默丢弃虽然“看起来没报错”，但最容易把数据丢失变成黑洞，后续追查成本反而最高。

### `CallerRunsPolicy` 为什么能形成反压？

提交任务的线程被迫亲自执行，提交速度自然会慢下来。如果上游就是 Web 线程或消息消费线程，这种拖慢会直接传回入口，从而限制继续灌任务的速度。

### 可以无限重试提交吗？

通常不行。线程池已经在告诉你容量打满了，这时无脑重试只会继续放大拥塞。更合理的是记录拒绝次数、快速失败、限流、降级，或者切到备用通道。

### 自定义拒绝策略时最该补什么？

至少补拒绝计数、结构化日志和告警指标；如果任务允许延迟执行，再补缓存队列、补偿表或降级逻辑。能把“失败后怎么办”说出来，这题就不再只是 API 记忆。

## 易错点

- 线上使用静默丢弃但没有监控。
- 拒绝后无限重试把调用方也拖垮。
