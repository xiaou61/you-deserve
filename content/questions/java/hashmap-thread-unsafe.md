## 一句话结论

HashMap 线程不安全，是因为它的 `put`、扩容、链表/红黑树调整都没有同步控制；多个线程同时修改时，可能出现数据覆盖、元素丢失、结构异常，旧版本里还可能因为扩容链表成环导致死循环。

## 通俗解释

你可以把 HashMap 想成一排抽屉。每个 key 会算出自己应该放进哪个抽屉。单线程时，一个人按顺序放东西，基本不会乱。

但多线程时，相当于好几个人同时往同一个抽屉塞东西，而且没有人排队。A 刚看到抽屉是空的，准备放；B 也看到抽屉是空的，也准备放。最后谁后写进去，谁就把前一个人的结果覆盖掉了。

扩容时更明显：HashMap 要把旧抽屉里的东西搬到新抽屉。如果两个线程同时搬，链表指针可能被改乱，数据就可能丢失或结构异常。

## 面试回答

可以从三个点回答：

1. `put` 操作不是原子的。多个线程同时写入同一个桶位时，可能互相覆盖。
2. 扩容过程不是线程安全的。多个线程同时 resize，会同时迁移节点，可能导致数据丢失或结构破坏。
3. HashMap 本身没有锁，也没有 volatile 等并发可见性保证，读线程可能看到中间状态。

如果面试官继续追问解决方案，可以说：多线程场景应该使用 `ConcurrentHashMap`，或者在外层加锁，但一般不推荐自己给 HashMap 粗暴加锁，因为性能和可维护性都不如并发容器。

## 常见追问

### HashMap 读操作安全吗？

如果没有线程在写，多个线程只读通常问题不大。但只要存在并发写，读也不安全，因为读线程可能看到扩容或链表调整中的中间状态。

### Hashtable 和 ConcurrentHashMap 有什么区别？

Hashtable 是老容器，基本方法都加了 synchronized，锁粒度粗，性能差。ConcurrentHashMap 是专门为并发设计的，JDK 8 以后主要通过 CAS + synchronized 控制桶级别并发，锁粒度更小。

## 易错点

- 不要只说“HashMap 没加锁”，要说清楚哪些操作会出问题。
- 不要把 HashMap 的线程不安全等同于“必然报错”。很多时候它是悄悄丢数据。
- 不要在并发写场景用 `Collections.synchronizedMap` 当默认答案，面试里优先说 ConcurrentHashMap。

## 详细讲解

HashMap 的线程不安全，最核心不是“源码里没有 synchronized”这么一句话，而是它的内部结构默认只为单线程修改服务。HashMap 底层是一张数组，数组里的每个位置叫桶，发生哈希冲突时，桶里会挂链表或红黑树。单线程 put 时，流程大致是先计算 hash，再定位桶位，再判断桶里有没有相同 key，最后插入、覆盖或触发扩容。这个流程拆开看有很多步骤，但 HashMap 没有把这些步骤包装成并发场景下的原子操作。

最常见的问题是覆盖。假设两个线程同时 put 到同一个桶，线程 A 看到桶为空，准备把自己的节点放进去；线程 B 也在几乎同一时间看到桶为空，也准备放。因为它们之间没有锁，也没有 CAS 协调，最后数组桶位只能保存最后一次写入的节点，另一个线程的写入就像从来没发生过。更麻烦的是，这种错误不一定抛异常，它可能只是业务上少了一条数据，所以线上表现会很隐蔽。

第二个风险来自扩容。HashMap 在元素数量超过阈值后会 resize，把旧数组里的节点搬到更大的新数组。扩容不是简单复制，它要重新计算节点在新数组中的位置，并调整链表或树结构。如果多个线程同时触发扩容，或者一个线程正在扩容另一个线程还在 put，旧表、新表、链表指针和元素数量都会同时变化。JDK 7 里曾经因为头插法迁移链表，在并发扩容时可能形成环，导致 get 时死循环。JDK 8 改进了迁移方式，但这不代表 HashMap 变成线程安全了，并发修改仍然可能导致元素丢失、覆盖、结构状态不一致。

第三个点是内存可见性。HashMap 的 table、size、节点 next 指针等字段不是为了并发读写设计的。一个线程修改完结构，另一个线程未必立刻看到完整结果；读线程可能看到扩容过程中的中间态，也可能看到旧数组或尚未连接完整的节点链。面试里如果只说“读操作没问题”，要补一句前提：没有并发写时多线程只读通常可以；一旦有写，读也不安全。

图解时可以画成三段：左边是两个线程同时进入 put，中间是同一个桶位被竞争，右边分成覆盖丢失、扩容迁移混乱、读到中间态三个后果。最后再接一个解决方案节点：多线程写优先使用 ConcurrentHashMap，或者在外层明确加锁。ConcurrentHashMap 的价值在于它把并发控制做在容器内部，JDK 8 之后主要用 CAS 加桶级 synchronized 控制冲突范围，比给整个 HashMap 粗暴加一把大锁更可控。

项目回答里还可以补一个判断标准：如果集合在初始化后只读，HashMap 可以接受；如果存在并发写，就必须换容器或加同步边界。这个补充能让答案从“背源码问题”变成“会做工程选择”。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 HashMap 为什么线程不安全 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：HashMap 线程不安全，是因为它的 put、扩容、链表/红黑树调整都没有同步控制；多个线程同时修改时，可能出现数据覆盖、元素丢失、结构异常，旧版本里还可能因为扩容链表成环导致死循环

拆开来看，第一层是背景问题：从 put 覆盖、扩容迁移、结构破坏三个角度理解 HashMap 的线程安全问题。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：put 操作不是原子的。多个线程同时写入同一个桶位时，可能互相覆盖。 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：扩容过程不是线程安全的。多个线程同时 resize，会同时迁移节点，可能导致数据丢失或结构破坏。 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：HashMap 本身没有锁，也没有 volatile 等并发可见性保证，读线程可能看到中间状态。 不要只背概念名词，要能说出触发条件、底层机制和排查入口。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：当业务代码出现 Java、集合、HashMap、并发 相关的并发异常、性能抖动、配置不生效、对象行为和预期不一致时，就可以用这道题定位原因。
- **怎么做方案**：先看触发条件，再看运行时机制。围绕“调用入口、对象状态、线程边界、框架代理、异常日志”五个位置检查，判断 HashMap 为什么线程不安全 是设计问题、用法问题还是环境问题。
- **怎么验证效果**：用单元测试、压测、日志、线程栈、JFR/GC 日志或本地最小复现确认。结合真实业务代码说明什么时候会踩坑、怎么定位、怎么替换方案。
- **怎么兜底**：准备替代 API、隔离开关、降级策略、配置回滚和监控告警。面试里能讲出兜底，说明你不是只会写 happy path。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 HashMap 为什么线程不安全 讲成一次工程决策，而不是一个孤立知识点。重点围绕 对象模型、运行时行为、边界条件和框架默认行为。
