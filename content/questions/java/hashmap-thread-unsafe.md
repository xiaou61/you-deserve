## 一句话结论

HashMap 线程不安全，是因为它的 `put`、扩容、链表/红黑树调整都没有同步控制；多个线程同时修改时，可能出现数据覆盖、元素丢失、结构异常，旧版本里还可能因为扩容链表成环导致死循环。

## 通俗解释

你可以把 HashMap 想成一排抽屉。每个 key 会算出自己应该放进哪个抽屉。单线程时，一个人按顺序放东西，基本不会乱。

但多线程时，相当于好几个人同时往同一个抽屉塞东西，而且没有人排队。A 刚看到抽屉是空的，准备放；B 也看到抽屉是空的，也准备放。最后谁后写进去，谁就把前一个人的结果覆盖掉了。

扩容时更明显：HashMap 要把旧抽屉里的东西搬到新抽屉。如果两个线程同时搬，链表指针可能被改乱，数据就可能丢失或结构异常。

## 面试回答

可以从三个点回答：

1. `put` 操作不是原子的。多个线程同时写入同一个桶位时，可能互相覆盖。
2. 扩容过程不是线程安全的。多个线程同时 resize，会同时迁移节点，可能导致数据丢失或结构破坏。
3. HashMap 本身没有锁，也没有 volatile 等并发可见性保证，读线程可能看到中间状态。

如果面试官继续追问解决方案，可以说：多线程场景应该使用 `ConcurrentHashMap`，或者在外层加锁，但一般不推荐自己给 HashMap 粗暴加锁，因为性能和可维护性都不如并发容器。

## 常见追问

### HashMap 读操作安全吗？

如果没有线程在写，多个线程只读通常问题不大。但只要存在并发写，读也不安全，因为读线程可能看到扩容或链表调整中的中间状态。

### Hashtable 和 ConcurrentHashMap 有什么区别？

Hashtable 是老容器，基本方法都加了 synchronized，锁粒度粗，性能差。ConcurrentHashMap 是专门为并发设计的，JDK 8 以后主要通过 CAS + synchronized 控制桶级别并发，锁粒度更小。

## 易错点

- 不要只说“HashMap 没加锁”，要说清楚哪些操作会出问题。
- 不要把 HashMap 的线程不安全等同于“必然报错”。很多时候它是悄悄丢数据。
- 不要在并发写场景用 `Collections.synchronizedMap` 当默认答案，面试里优先说 ConcurrentHashMap。

## 记忆钩子

记住一句话：**HashMap 的问题不是不会存，而是多人同时存时没人维持秩序。**
