## 一句话结论

HashMap 只适合单线程或只读场景；一旦多个线程同时 `put`、`resize` 或迁移桶位，就可能出现覆盖、丢数据、读到中间状态，JDK 7 里甚至可能把链表改成环。

## 通俗解释

你可以把 HashMap 想成一排抽屉。每个 key 会算出自己应该放进哪个抽屉。单线程时，一个人按顺序放东西，基本不会乱。

但多线程时，相当于好几个人同时往同一个抽屉塞东西，而且没有人排队。A 刚看到抽屉是空的，准备放；B 也看到抽屉是空的，也准备放。最后谁后写进去，谁就把前一个人的结果覆盖掉了。

扩容时更明显：HashMap 要把旧抽屉里的东西搬到新抽屉。如果两个线程同时搬，链表指针可能被改乱，数据就可能丢失或结构异常。

## 面试回答

面试里可以按“写入冲突 -> 扩容迁移 -> 方案替代”这条线回答：

1. `put` 不是原子操作。两个线程同时算到同一个桶位，后写的结果可能覆盖先写的结果。
2. `resize` 会搬迁节点和重连指针，并发执行时容易把链表结构改乱，JDK 7 旧实现里还可能出现环。
3. HashMap 没有锁，也没有为并发写设计的可见性与复合操作保护，读线程可能看到尚未稳定的结构。
4. 真正的解决方案通常是 `ConcurrentHashMap`、只读快照发布，或者外层在更高层次做串行化控制，而不是给 HashMap 临时补一把粗锁。

## 常见追问

### 为什么很多时候没有报错，但线上还是会出事？

因为 HashMap 的并发问题常常表现为“悄悄写错”而不是立刻抛异常，例如 value 被覆盖、size 统计不准、桶迁移后元素丢失。这类问题比直接报错更难排查，通常要靠压测、对账或最小复现去抓。

### JDK 7 和 JDK 8 在这题上该怎么区分？

JDK 7 需要重点提头插法迁移导致链表成环的经典问题；JDK 8 虽然迁移逻辑改了，但它依旧不是并发 Map，并发写仍然会有覆盖、丢失和中间状态可见的问题。面试里别把“没有成环”误说成“线程安全了”。

### `Collections.synchronizedMap` 和 `ConcurrentHashMap` 怎么选？

前者本质是整张表一把锁，语义简单但并发度低；后者针对并发访问做了更细粒度设计，读写性能和扩容协作都更适合高并发场景。只有在极低并发、快速包一层兼容旧代码时，才会考虑前者。

### 怎么自己证明这题不是只会背？

最有效的方法是写一个小压测：多个线程同时向同一批 key 写入，统计最终条目数、value 覆盖率和 CPU 波动。你要能说出“有时复现不了”恰恰是这题的特点，因为它高度依赖调度时序。

## 易错点

- 不要只说“HashMap 没加锁”，要说清楚哪些操作会出问题。
- 不要把 HashMap 的线程不安全等同于“必然报错”。很多时候它是悄悄丢数据。
- 不要在并发写场景用 `Collections.synchronizedMap` 当默认答案，面试里优先说 ConcurrentHashMap。

## 详细讲解
HashMap 为什么线程不安全 这种 Java 题如果只停在定义，很容易被一句“那为什么会这样”追住。比较稳的讲法是先把它放回 JDK 语义或并发语义里，先用一句话压住主题，比如 从 put 覆盖、扩容迁移、结构破坏三个角度理解 HashMap 的线程安全问题，再说明它到底在解决什么问题，或者在约束什么行为。

主线部分适合围绕“`put` 不是原子操作。两个线程同时算到同一个桶位，后写的结果可能覆盖先写的结果”展开，把状态变化、对象关系或线程交互讲清楚。Java 题越往后问，越不看你能不能背 API，而看你能不能把底层语义、可见性、CAS/锁、生命周期或者 JDK 实现讲成一条连续的因果链。

真正容易翻车的，往往是边界和代价。这里可以主动补 不要只说“HashMap 没加锁”，要说清楚哪些操作会出问题、不要把 HashMap 的线程不安全等同于“必然报错”。很多时候它是悄悄丢数据、不要在并发写场景用 `Collections.synchronizedMap` 当默认答案，面试里优先说 ConcurrentHashMap 这些点，再解释它们在高并发、长生命周期对象、版本差异或错误用法下会怎样放大。这样读者不会只记住“这个东西能干嘛”，还会知道“什么时候别这么用”。
