## 一句话结论

InheritableThreadLocal 只能在“创建子线程”那一刻拷贝父线程上下文，碰到线程池复用时，它既不会自动重新继承，也不会自动替你清理旧值。

## 通俗解释

InheritableThreadLocal 像把父任务标签贴给新员工，但线程池员工会反复接不同任务，旧标签可能没撕掉。

## 面试回答

回答时把“新线程”和“线程池线程”分开讲最清楚：

1. 新建子线程时，父线程里的 inheritable 值会复制到子线程。
2. 这个复制动作发生在线程创建阶段，而不是每次提交任务时。
3. 线程池里的工作线程通常很早就创建好了，后续任务不会重新继承父线程最新值，所以很容易串请求、串 trace、串租户。
4. 真要做线程池上下文透传，优先考虑显式参数、任务装饰器或专门的上下文传递方案，而不是直接押注 InheritableThreadLocal。

## 常见追问

### 为什么很多人明明用了它，在线程池里还是拿不到新值？

因为他们误以为“提交一个任务就等于新建一个线程”。实际上工作线程早就存在了，没有新的继承动作发生，所以你在父线程后续 set 的内容，并不会自动进到池线程。

### 父线程后来又改值，子线程会同步更新吗？

不会。默认语义更像一次性拷贝，而不是绑定联动。把这一点说出来，能很好地区分“继承”与“共享”的概念。

### 和 `TransmittableThreadLocal` 有什么关系？

TTL 本质上是通过包装任务，把提交时的上下文捕获下来，再在执行前后设置和清理，弥补线程池复用场景下 InheritableThreadLocal 的空档。面试里提到这个方向会比较加分。

### 最后的落地建议是什么？

优先用显式参数传关键业务数据；只有在链路日志、MDC、traceId 这类横切上下文里，才慎用上下文透传工具，并且必须配套清理策略。

## 易错点

- 在线程池里用它传用户身份。
- 只 set 不 remove，导致上下文泄漏。
