## 一句话结论

ThreadLocal 的本质是“数据挂在线程身上”，不是挂在 ThreadLocal 对象自己身上；线程复用时不清理，最先出事的通常不是 OOM，而是上下文串用。

## 通俗解释

ThreadLocal 像每个员工自己的抽屉。大家都叫“临时资料”，但每个人抽屉里放的是自己的东西。

问题是线程池里的员工不会下班，抽屉一直在。如果用完不清理，下一个任务可能看到上一个任务留下的东西。

## 面试回答

这题面试里可以按“存哪儿、为什么泄漏、怎么规避”三步答：

1. 每个线程对象里都有自己的 `ThreadLocalMap`，value 存在线程里，不是存在 ThreadLocal 实例里。
2. key 是对 ThreadLocal 的弱引用，ThreadLocal 被回收后，key 可能变成 `null`，但 value 还可能挂在线程上。
3. 在线程池里线程寿命很长，如果你只 `set` 不 `remove`，就会出现脏数据串用和对象长期不释放。
4. 规范做法是在 `try/finally` 里清理，并尽量别往里面塞大对象或复杂业务状态。

## 常见追问

### 为什么说线程池比新建线程更危险？

因为新线程执行完就结束，残留数据有机会随线程一起回收；线程池线程会被反复复用，同一个线程下一个请求还能看到上一个请求留下来的值，这比“可能泄漏”更早、更常见。

### key 是弱引用，为什么 value 还会留住？

弱引用只会让 key 在 GC 后变成 null，但清理 stale entry 依赖后续访问触发；如果线程长期存活又很少触碰对应槽位，value 仍然可能挂着。这正是很多人对“弱引用就不会泄漏”的误解。

### 它适合放哪些东西，不适合放哪些东西？

适合请求级上下文、traceId、租户标识这类“线程内临时元信息”；不适合大对象、跨线程共享状态、需要强一致传递的核心业务数据。面试里把“合适和不合适”都说出来，会更完整。

### 线上怎么排查 ThreadLocal 问题？

常见信号是用户串号、日志 trace 混乱、线程池线程堆积大对象、heap dump 里线程引用链异常长。你能说出这些现象，答案就会从原理题变成排障题。

## 易错点

- 不要说 ThreadLocal 的值存在 ThreadLocal 对象里。
- 不要忘记线程池场景下 remove。
- 不要用 ThreadLocal 传递复杂业务状态。
