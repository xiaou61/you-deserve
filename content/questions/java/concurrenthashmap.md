## 一句话结论

JDK 8 的 ConcurrentHashMap 不是“很多段锁”的旧答案了，它更像“空桶 CAS、冲突桶加锁、读多数无锁、扩容可协作”的一套组合拳。

## 通俗解释

Hashtable 像整个超市只有一把大锁，一个人结账时别人都不能动。ConcurrentHashMap 像每个货架局部管理，多个人可以在不同货架同时操作。

## 面试回答

这题建议按“读怎么快、写怎么稳、扩容怎么做”来回答：

1. 读路径多数不加锁，依赖 `volatile` 可见性拿到最新桶引用和节点状态。
2. 写入空桶时用 CAS 抢占；遇到非空桶，再锁桶头节点做链表或树节点更新。
3. 冲突链过长会树化，避免热点桶退化成线性查找。
4. 扩容时多个线程可以一起迁移，借助 `ForwardingNode` 标记已迁移桶位，而不是让单线程独自搬完整张表。

## 常见追问

### 为什么它不允许 `null key/value`？

因为并发场景下 `get(key) == null` 需要明确表示“没有这个 key”，如果允许 null value，就没法区分“key 不存在”还是“存在但 value 为 null”。这是并发语义设计，不只是风格选择。

### `computeIfAbsent` 有什么坑？

回调函数最好保持短小、无阻塞、无副作用。因为它可能在锁保护区附近执行，若里面再做远程调用、再次修改同 map 或抛复杂异常，容易放大锁竞争甚至引出死锁和重入问题。

### `size()` 为什么不能当强一致计数？

高并发下它会通过计数单元近似统计或多次尝试汇总，追求的是性能和可接受精度，而不是事务级绝对一致。如果业务必须精确计数，通常要单独设计计数器或落库方案。

### 它和 `Hashtable` 最本质的区别是什么？

不是“一个新一个旧”这么简单，而是并发控制粒度和读路径完全不同。Hashtable 近似整表大锁，而 ConcurrentHashMap 让无冲突读写尽量并行，这才是它吞吐更高的根因。

## 易错点

- 不要把 JDK 7 的 Segment 机制直接套到 JDK 8。
- 线程安全不代表复合操作天然原子。
- `size()` 在并发下是近似统计思路，不要当强一致计数依赖。
