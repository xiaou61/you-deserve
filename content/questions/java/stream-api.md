## 一句话结论

Stream API 适合对集合数据做过滤、转换、排序、分组、聚合等操作，可以让代码更声明式，但不适合滥用复杂链式逻辑。

## 通俗解释

Stream 像流水线。数据从入口进来，经过过滤、加工、分组，最后产出结果。每一步都专注做一件事。

## 面试回答

Stream 常见操作包括：

- `filter`：过滤元素。
- `map`：转换元素。
- `sorted`：排序。
- `collect`：收集结果。
- `groupingBy`：分组。
- `reduce`：聚合。

它适合处理内存中的集合数据，让代码更简洁。但要注意不要在 Stream 中写大量副作用逻辑，比如修改外部变量、发请求、写数据库等。

并行流 `parallelStream` 也不能随便用，要考虑数据量、线程池、任务是否 CPU 密集以及是否有共享状态。

## 常见追问

### Stream 会改变原集合吗？

大多数 Stream 操作不会直接改变原集合，而是生成新的处理结果。但如果元素本身是对象，修改对象属性仍会影响对象状态。

### parallelStream 一定更快吗？

不一定。线程切换、拆分合并、共享资源竞争都可能让它更慢。

## 易错点

- 不要把 Stream 写成很长很难读的链。
- 不要在 Stream 里塞复杂业务副作用。
- 并行流要谨慎，不是性能万能药。

## 实战落地

- **什么时候会遇到**：当业务代码出现 Java、Stream、集合、函数式编程 相关的并发异常、性能抖动、配置不生效、对象行为和预期不一致时，就可以用这道题定位原因。
- **怎么做方案**：先看触发条件，再看运行时机制。围绕“调用入口、对象状态、线程边界、框架代理、异常日志”五个位置检查，判断 Java Stream API 适合什么场景 是设计问题、用法问题还是环境问题。
- **怎么验证效果**：用单元测试、压测、日志、线程栈、JFR/GC 日志或本地最小复现确认。结合真实业务代码说明什么时候会踩坑、怎么定位、怎么替换方案。
- **怎么兜底**：准备替代 API、隔离开关、降级策略、配置回滚和监控告警。面试里能讲出兜底，说明你不是只会写 happy path。
