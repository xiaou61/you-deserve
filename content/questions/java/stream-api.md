## 一句话结论

Stream API 适合对集合数据做过滤、转换、排序、分组、聚合等操作，可以让代码更声明式，但不适合滥用复杂链式逻辑。

## 通俗解释

Stream 像流水线。数据从入口进来，经过过滤、加工、分组，最后产出结果。每一步都专注做一件事。

## 面试回答

Stream 常见操作包括：

- `filter`：过滤元素。
- `map`：转换元素。
- `sorted`：排序。
- `collect`：收集结果。
- `groupingBy`：分组。
- `reduce`：聚合。

它适合处理内存中的集合数据，让代码更简洁。但要注意不要在 Stream 中写大量副作用逻辑，比如修改外部变量、发请求、写数据库等。

并行流 `parallelStream` 也不能随便用，要考虑数据量、线程池、任务是否 CPU 密集以及是否有共享状态。

## 常见追问

### 这题最容易被追问的边界是什么？

重点说清线程安全、可变性、泛型擦除、异常包装、版本差异、空值和性能退化。不要只给结论，要说明哪些条件下结论成立，哪些条件下会退化或需要换方案。

### 怎么证明自己不是在背模板？

用单元测试、线程栈、JFR、日志、源码方法名、构造参数和最小复现做验证。能说出具体信号、反例或排查入口，答案就从概念层进入实战层。

### 和相近方案怎么区分？

可以拿基础 API、并发容器、锁、原子类、不可变对象、Stream 或 CompletableFuture对比，从语义正确性、线程安全、可读性、性能和版本兼容性几个维度选择。

### 面试官继续深挖时怎么展开？

先围绕“filter：过滤元素。”讲清主链路，再补“补中间操作惰性、终止操作触发执行、短路操作、stream 不能复用、原集合修改风险、Collectors.toMap key 冲突处理、parallelStream 使用公共 ForkJoinPool 的风险”。如果能把边界条件、异常分支和验证闭环连起来，就能接住二面、三面的追问。

## 易错点

- 不要把 Stream 写成很长很难读的链。
- 不要在 Stream 里塞复杂业务副作用。
- 并行流要谨慎，不是性能万能药。
