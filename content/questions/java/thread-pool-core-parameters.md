## 一句话结论

线程池参数要按“任务先建线程、再入队、再扩容、最后拒绝”来理解，脱离流转顺序死背参数名，面试一追问就散。

## 通俗解释

线程池像一家奶茶店。核心线程数是固定店员，任务队列是排队区，最大线程数是临时加班店员，拒绝策略是店里实在忙不过来时怎么处理新顾客。

顾客少时，固定店员处理；顾客多了，先排队；队伍满了，再叫临时店员；临时店员也满了，就只能拒绝或让顾客换种方式处理。

## 面试回答

面试里可以直接照着 `ThreadPoolExecutor` 的接单流程讲：

1. 运行线程数小于 `corePoolSize` 时，优先创建核心线程执行任务。
2. 到了核心线程数后，新任务先尝试进入 `workQueue`。
3. 队列满了且线程数还没到 `maximumPoolSize`，再创建非核心线程顶上。
4. 线程和队列都吃满后，才进入 `RejectedExecutionHandler`。
5. `keepAliveTime` 控制的是非核心线程空闲多久回收，是否连核心线程都超时回收，还要看 `allowCoreThreadTimeOut`。

## 常见追问

### 为什么很多人把 `maximumPoolSize` 讲错了？

因为它不是“线程池一直能开到的常态人数”，只有在队列放不下任务时才会继续扩容。如果你用的是无界队列，很多情况下最大线程数几乎形同虚设，这是高频误区。

### CPU 密集和 IO 密集该怎么配？

CPU 密集更关注核心数附近，避免线程过多引发切换；IO 密集可以适当放大，但前提是明确瓶颈在等待而不是 CPU。本质上是根据任务画像配，不是背一个万能公式。

### 线上怎么判断线程池是不是配小了？

常看几个信号：队列长度长期堆积、活跃线程接近上限、拒绝次数上涨、接口 RT 拉长、调用线程被 CallerRuns 拖慢。这类指标一讲，面试官就会觉得你真看过监控。

### 为什么要自定义线程工厂？

至少要给线程起可识别名字，方便日志、jstack、监控平台定位；很多团队还会顺手设置守护线程标记、异常处理器和业务标签。这是很容易落地的工程细节。

## 易错点

- 不要只背参数名，要说任务流转顺序。
- 队列类型会影响线程池行为，无界队列会让最大线程数很难生效。
- 线程池不是越大越好，CPU 密集和 IO 密集任务的参数思路不同。
