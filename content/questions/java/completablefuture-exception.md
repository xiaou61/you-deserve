## 一句话结论

CompletableFuture 的异常不会像同步调用那样直接冒到当前调用栈，必须在编排链上显式决定是记录、转换、兜底还是继续向后传播。

## 通俗解释

CompletableFuture 异常像异步快递破损，不能站在发货点等报错，要在签收链路处理。

## 面试回答

这题最好把三个常用 API 的角色分开讲：

1. `exceptionally` 更像失败兜底，出错时给一个替代值。
2. `handle` 不管成功失败都会进，适合统一转换结果。
3. `whenComplete` 更适合做日志、指标、埋点观察，不等于把异常吞掉了。
4. 最后如果用 `get()` 或 `join()` 收口，还要知道一个包 `ExecutionException`，一个包 `CompletionException`。

## 常见追问

### 为什么很多异步异常上线后才暴露？

因为代码里可能根本没有 `join/get`，或者日志只打在 `whenComplete` 里却没真正处理。异常被留在 Future 链里，业务主线程不一定第一时间感知。

### `allOf` 里一个分支失败会怎样？

整体 future 会以异常完成，但你如果想保留部分成功结果，就要在各分支自己兜底，或者把结果包装成成功/失败对象后再汇总。这是“聚合接口要不要部分降级”的关键设计点。

### 超时和取消该怎么讲？

可以补 `orTimeout`、`completeOnTimeout` 和 `cancel` 的语义差别：一个是超时报错，一个是超时给默认值，一个是主动取消。把这三个点拉开，答案就不再像模板。

### 最佳实践是什么？

在线路末端统一收敛异常，重要分支打指标和结构化日志，线程池按 IO/CPU 分离，必要时加超时和熔断。异步最怕“错了没人知道”，所以观测和收口比语法更重要。

## 易错点

- 异步任务内部异常没 join/get，导致问题被延后发现。
- whenComplete 里记录日志后以为异常已经恢复。
