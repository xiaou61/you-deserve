## 一句话结论

受检异常在编译期必须处理或声明抛出；运行时异常不强制处理，通常表示程序逻辑错误或运行环境问题。

## 通俗解释

受检异常像出门必须带证件，编译器会提前提醒你。运行时异常像路上摔了一跤，代码写得不稳时才发生。

## 面试回答

受检异常继承自 Exception 但不属于 RuntimeException，比如 IOException。调用可能抛出受检异常的方法时，必须 try-catch 或 throws。

运行时异常继承 RuntimeException，比如 NullPointerException、IndexOutOfBoundsException。编译器不强制处理，更多是通过代码健壮性避免。

## 常见追问

### 业务异常应该设计成哪种？

很多项目会把业务异常设计成运行时异常，配合全局异常处理统一返回错误码，避免方法签名到处 throws。

### Error 和 Exception 有什么区别？

Error 通常表示 JVM 或系统级严重问题，应用一般不主动捕获；Exception 是程序可处理异常。

## 易错点

- 不要把所有异常都 catch 后吞掉。
- 不要用异常控制正常业务流程。
- 业务异常要配合统一错误码和日志。

## 详细讲解

受检异常和运行时异常的区别，表面上是“编译器管不管”，本质上反映了 Java 对异常语义的不同设计意图。受检异常，也就是 checked exception，要求调用者显式处理或者继续在方法签名里抛出；运行时异常则不强制这一点，它通常表示程序员使用方式错误、前置条件不满足或代码逻辑问题。  

为什么 Java 要分这两类？因为不是所有异常都适合同样对待。像文件不存在、网络 IO 失败、外部资源不可用，这类异常往往不是程序本身写错，而是运行环境或外部依赖出现了问题，调用方有机会做降级、重试、提示用户，所以 Java 倾向于把这类异常设计成受检异常，逼着调用链正视它们。  

运行时异常则更多用于表示“不该这样写代码”或者“当前状态下这样调用不成立”。典型如空指针、数组越界、非法参数、状态不合法等。这类问题多数时候不是靠上层 catch 一下就能优雅恢复，而是应该回到代码本身修正逻辑。因此它们不要求写在方法签名里，否则整条调用链都会被异常声明污染。  

但工程实践里，也不能机械地认为“受检异常一定好、运行时异常一定坏”。很多现代项目为了避免接口层层 throws 一串异常，常常在业务层定义统一业务异常，配合全局异常处理输出标准响应；而底层资源异常、第三方调用异常则可能被封装后再往上抛。真正好的做法，是让异常分类有助于维护和定位，而不是让方法签名变成异常清单。  

面试里可以补一个很有判断力的观点：异常设计要看调用方是否真的能合理处理。如果一个异常即使暴露给上层，上层也只能记录日志后失败返回，那强行设计成 checked 反而会让代码到处 try-catch，噪音很大。反之，如果异常确实存在明确恢复路径，比如提醒重试、切换降级资源、回滚操作，那 checked 就更有价值。  

再往工程习惯里落一句也很好用：业务系统里最怕的是异常语义混乱。同一类失败一会儿抛 checked，一会儿抛 runtime，一会儿又直接返回错误码，调用方会非常痛苦。所以异常体系最重要的是一致性，而不是“把所有理论都用上”。  

如果还想再补一个追问点，可以说“很多框架层异常最后会在边界统一转换成业务异常或统一错误响应”。也就是说，异常体系并不是写完类继承树就结束了，它还和日志、监控、接口契约一起构成整条错误处理链。  

图解可以画成两条传播链：受检异常从底层方法一路显式向上声明，逼着调用者处理；运行时异常则更多在运行时冒泡，由统一异常处理或上层兜底。旁边再标一句：异常不是按心情分，关键是调用方是否应该被强制意识到它。这样答案就会很成熟。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 受检异常和运行时异常有什么区别 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：受检异常在编译期必须处理或声明抛出；运行时异常不强制处理，通常表示程序逻辑错误或运行环境问题

拆开来看，第一层是背景问题：理解编译期强制处理和运行期才暴露的异常差异。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：受检异常继承自 Exception 但不属于 RuntimeException，比如 IOException 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：调用可能抛出受检异常的方法时，必须 try-catch 或 throws 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：运行时异常继承 RuntimeException，比如 NullPointerException、IndexOutOfBoundsException 不要只背概念名词，要能说出触发条件、底层机制和排查入口。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：当业务代码出现 Java、异常、Exception、RuntimeException 相关的并发异常、性能抖动、配置不生效、对象行为和预期不一致时，就可以用这道题定位原因。
- **怎么做方案**：先看触发条件，再看运行时机制。围绕“调用入口、对象状态、线程边界、框架代理、异常日志”五个位置检查，判断 受检异常和运行时异常有什么区别 是设计问题、用法问题还是环境问题。
- **怎么验证效果**：用单元测试、压测、日志、线程栈、JFR/GC 日志或本地最小复现确认。结合真实业务代码说明什么时候会踩坑、怎么定位、怎么替换方案。
- **怎么兜底**：准备替代 API、隔离开关、降级策略、配置回滚和监控告警。面试里能讲出兜底，说明你不是只会写 happy path。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 受检异常和运行时异常有什么区别 讲成一次工程决策，而不是一个孤立知识点。重点围绕 对象模型、运行时行为、边界条件和框架默认行为。

## 回答模板

面试时可以按这个节奏组织：

1. **先给结论**：理解编译期强制处理和运行期才暴露的异常差异。
2. **再讲机制**：它的核心不是某个名词，而是一组处理链路。把关键角色、状态变化和触发条件说清楚。
3. **补充边界**：说明什么情况下有效，什么情况下会失效，以及为什么需要配套措施。
4. **落到项目**：如果我在项目里遇到，会先看指标和日志定位问题，再用灰度、压测和回滚策略验证方案。
5. **收一句风险**：真正重要的是不要只让功能跑通，还要保证高并发、异常分支和数据状态都可控。
