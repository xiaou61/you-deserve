## 一句话结论

如果两个对象通过 `equals` 判断相等，它们的 `hashCode` 必须相同。否则放进 HashMap 或 HashSet 这类哈希集合时，可能找不到或出现重复。

## 通俗解释

hashCode 像小区门牌号，equals 像确认是不是同一个人。你说两个人是同一个人，但门牌号却不同，保安就会去不同楼栋找，自然找不到。

## 面试回答

哈希集合通常先用 `hashCode` 定位桶，再用 `equals` 判断桶里的对象是否真正相等。

如果只重写 `equals` 不重写 `hashCode`，两个逻辑相等的对象可能得到不同哈希值，被分到不同桶里。这样 `HashSet` 可能存入重复元素，`HashMap` 也可能用等价 key 取不到值。

所以重写 `equals` 时必须保证：

- 自反性、对称性、传递性、一致性。
- `equals` 相等的对象，`hashCode` 必须相同。

## 常见追问

### hashCode 相同，equals 一定相等吗？

不一定。不同对象可能发生哈希冲突，所以还需要 equals 做最终判断。

### 为什么 String 可以安全作为 key？

String 重写了 equals 和 hashCode，并且不可变。不可变对象作为 key 更稳定，不会因为字段变化导致哈希位置变化。

## 易错点

- 不要说 hashCode 相同对象一定相等。
- 不要只重写 equals。
- 不要使用可变字段作为哈希 key 的核心依据。

## 详细讲解

equals 和 hashCode 要一起重写，是因为 Java 集合里很多结构依赖二者共同判断对象身份。equals 解决“两个对象业务上是否相等”，hashCode 解决“对象在哈希结构里大致放到哪个桶”。在 HashMap、HashSet、ConcurrentHashMap 这类哈希集合里，查找流程通常是先算 hashCode 定位桶，再在桶里用 equals 判断具体是不是同一个 key。

Java 的基本约定是：如果 a.equals(b) 为 true，那么 a.hashCode() 必须等于 b.hashCode()。反过来不要求成立，hashCode 相等只能说明可能在同一个桶里，还要用 equals 做最终判断。如果只重写 equals 不重写 hashCode，就可能出现两个业务相等的对象 hashCode 不同，被放到不同桶里，HashSet 去重失败，HashMap 用等价 key 取不到值。

举例说，一个 User 用 id 判断相等。你 new 两个 User，id 都是 1。如果 equals 按 id 返回 true，但 hashCode 仍然继承 Object 默认实现，那么两个对象的 hashCode 可能不同。把第一个 User 放进 HashMap，再用第二个 User 去 get，HashMap 会先按第二个对象的 hashCode 找桶，桶都不同，自然找不到。这就是“明明 equals 相等却取不到”的根因。

实现时要选择稳定字段。用于 equals 和 hashCode 的字段最好在对象放入集合后不要变化。如果把可变字段作为 hashCode 依据，对象放进 HashSet 后字段变了，hashCode 变了，但对象还停留在旧桶里，后续 contains 或 remove 可能失败。实体类如果用数据库 id，要注意新对象 id 为空时的相等逻辑；值对象则可以用所有业务字段。

IDE 或 Lombok 能自动生成 equals/hashCode，但也要理解生成策略。继承场景尤其要小心，父类字段是否参与、getClass 还是 instanceof、canEqual 等都可能影响对称性和传递性。面试里不必展开所有细节，但要说出五个约定：自反、对称、传递、一致、非空比较为 false，以及 equals 相等必须 hashCode 相等。

图解可以画 HashMap 查找过程：对象 key 先计算 hashCode，进入对应桶；桶里可能有多个节点，再用 equals 逐个比较。旁边画错误分支：两个 equals 相等对象 hashCode 不同，被分到不同桶，导致 get 失败。最后落点：只要对象会作为 HashMap key 或放进 HashSet，就必须认真设计 equals 和 hashCode。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 equals 和 hashCode 为什么要一起重写 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：如果两个对象通过 equals 判断相等，它们的 hashCode 必须相同

拆开来看，第一层是背景问题：从 HashMap/HashSet 的查找流程理解 equals 和 hashCode 的契约。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：哈希集合通常先用 hashCode 定位桶，再用 equals 判断桶里的对象是否真正相等 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：如果只重写 equals 不重写 hashCode，两个逻辑相等的对象可能得到不同哈希值，被分到不同桶里 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：这样 HashSet 可能存入重复元素，HashMap 也可能用等价 key 取不到值 不要只背概念名词，要能说出触发条件、底层机制和排查入口。这样回答会比单纯背八股更像做过项目的人。
