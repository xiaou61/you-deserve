## 一句话结论

String 不可变；StringBuilder 可变但线程不安全，性能较好；StringBuffer 可变且线程安全，但同步带来额外开销。

## 通俗解释

String 像写死的纸条，改一次就要换一张。StringBuilder 像草稿纸，可以一直涂改。StringBuffer 也是草稿纸，但每次修改都要先上锁，避免多人同时写乱。

## 面试回答

String 的内容不可变，每次拼接可能创建新对象。少量拼接问题不大，但循环里大量拼接不适合直接用 String。

StringBuilder 适合单线程下大量字符串拼接。StringBuffer 的方法加了同步，适合多线程共享同一个字符串缓冲区的场景，但实际业务里这种场景不多。

## 常见追问

### String 为什么不可变？

不可变让它更安全，适合作为 HashMap key，也方便字符串常量池复用和缓存 hash 值。

### 字符串拼接一定要手写 StringBuilder 吗？

不一定。简单拼接编译器可能优化。循环大量拼接时再显式使用 StringBuilder。

## 易错点

- 不要说 StringBuilder 一定线程安全。
- 不要说 StringBuffer 一定更好，它只是更安全但更重。
- 不要忽略 String 不可变带来的好处。

## 详细讲解

String、StringBuilder、StringBuffer 的区别，表面看像“一个不可变、一个快、一个线程安全”，但面试里更重要的是把这些差异和使用场景连起来。先说 String。它是不可变对象，一旦创建，内容就不能修改。每次看起来像“修改”字符串，实际通常是在创建新对象。这种设计的好处很多：可以安全放进字符串常量池、便于缓存 hashCode、天然线程安全，也让它非常适合做配置值、参数名、URL 片段这类只读文本。  

问题是，如果你在循环里频繁用  拼接字符串，就可能不断创建新对象，带来额外内存分配和复制成本。编译器在简单场景下会帮你优化，但在复杂循环和动态拼接场景里，手动使用可变字符序列通常更合适。这时就轮到 StringBuilder。它内部维护可扩容的字符数组，append 时通常是在原有缓冲区上追加，避免频繁生成新 String 对象，所以单线程场景下拼接效率更高。  

StringBuffer 和 StringBuilder 的核心差异在于线程安全。StringBuffer 的很多方法带同步，因此在多线程同时修改同一个实例时更安全，但同步也意味着额外开销。现代 Java 业务代码里，真正需要多个线程共享并发修改同一个字符串缓冲区的场景其实并不多，所以大多数时候更常用的是 StringBuilder，而不是 StringBuffer。  

面试时最好不要把它讲成绝对化口号。比如“永远不用 String 拼接”就太僵硬。更准确的说法是：少量常规拼接、编译期常量拼接、日志参数化之外的简单场景，用 String 完全没问题；当你进入循环、批量构造 SQL、拼装大文本、生成报文时，优先考虑 StringBuilder。真正的判断标准是是否存在大量中间字符串对象。  

再往深一点，还可以补一句为什么 String 不可变这么重要。因为不可变对象更容易被共享，不必担心别人改掉内容；这对常量池、类加载、安全校验、文件路径、网络地址等基础设施都很关键。也就是说，String 的不可变不是性能缺陷，而是语言设计上的稳定性选择。  

还有一个很实际的答法是提醒“别把线程安全和可变性混为一谈”。String 安全，不是因为它做了同步，而是因为它根本不让你改；StringBuffer 安全，是因为它在改的时候做了同步。把这个因果讲清楚，答案就比单纯背结论更扎实。  

图解可以画成三个对象对比：String 每次“修改”都生成新对象；StringBuilder 在同一个可变缓冲区追加；StringBuffer 也是可变缓冲区，但外层有同步保护。最后给出一句实战结论：默认用 String 表达文本值，频繁拼接用 Builder，并发共享修改极少数场景再考虑 Buffer。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 String、StringBuilder、StringBuffer 有什么区别 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：String 不可变；StringBuilder 可变但线程不安全，性能较好；StringBuffer 可变且线程安全，但同步带来额外开销

拆开来看，第一层是背景问题：从不可变、线程安全和拼接性能区分三个字符串相关类。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：String 的内容不可变，每次拼接可能创建新对象 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：少量拼接问题不大，但循环里大量拼接不适合直接用 String 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：StringBuilder 适合单线程下大量字符串拼接 不要只背概念名词，要能说出触发条件、底层机制和排查入口。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：当业务代码出现 Java、String、StringBuilder、StringBuffer 相关的并发异常、性能抖动、配置不生效、对象行为和预期不一致时，就可以用这道题定位原因。
- **怎么做方案**：先看触发条件，再看运行时机制。围绕“调用入口、对象状态、线程边界、框架代理、异常日志”五个位置检查，判断 String、StringBuilder、StringBuffer 有什么区别 是设计问题、用法问题还是环境问题。
- **怎么验证效果**：用单元测试、压测、日志、线程栈、JFR/GC 日志或本地最小复现确认。结合真实业务代码说明什么时候会踩坑、怎么定位、怎么替换方案。
- **怎么兜底**：准备替代 API、隔离开关、降级策略、配置回滚和监控告警。面试里能讲出兜底，说明你不是只会写 happy path。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 String、StringBuilder、StringBuffer 有什么区别 讲成一次工程决策，而不是一个孤立知识点。重点围绕 对象模型、运行时行为、边界条件和框架默认行为。

## 回答模板
