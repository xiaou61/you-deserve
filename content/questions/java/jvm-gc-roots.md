## 一句话结论

JVM 主要通过可达性分析判断对象是否可回收：从 GC Roots 出发，如果一个对象沿着引用链走不到，就说明它不再被程序使用，可以被垃圾回收。

## 通俗解释

你可以把内存里的对象想成很多房间，引用就是房间之间的门。GC Roots 是几个一定还被使用的入口，比如正在运行的方法栈、本地方法引用、类静态变量。

垃圾回收器会从这些入口开始走。能走到的房间，说明程序还有机会用到；走不到的房间，说明已经和主路断开，就可以清理。

## 面试回答

JVM 判断对象是否可回收通常不是靠引用计数，而是靠可达性分析。可达性分析会从 GC Roots 作为起点，向下搜索引用链。凡是不能从 GC Roots 到达的对象，就被认为是不可达对象，可以进入回收流程。

常见的 GC Roots 包括：

- 虚拟机栈中引用的对象，比如当前方法里的局部变量。
- 方法区中类静态属性引用的对象。
- 方法区中常量引用的对象。
- 本地方法栈中 JNI 引用的对象。
- JVM 内部使用的一些对象，比如类加载器、异常对象等。

## 常见追问

### 为什么不用引用计数？

引用计数实现简单，但很难解决循环引用。比如 A 引用 B，B 引用 A，如果外部已经没人引用它们，它们的引用计数仍然不是 0，就可能无法回收。

### 不可达对象会立刻被回收吗？

不一定。对象不可达后通常还会经历一次筛选，尤其是重写了 `finalize` 的对象。但实际开发里不应该依赖 `finalize`，它不可控，也已经不推荐使用。

## 易错点

- 不要把“没有引用”说成唯一判断标准，准确说法是“从 GC Roots 不可达”。
- 不要把引用计数当成 HotSpot JVM 的主流判断方式。
- 不要把可回收和立刻回收混为一谈。

## 详细讲解

JVM 判断对象能不能回收，主流答案是可达性分析。它不是简单看对象有没有被某个变量“提到”，而是从一组被称为 GC Roots 的根对象出发，沿着引用关系向外搜索。凡是能从这些根对象一路走到的对象，都被认为还活着；如果某个对象从所有 GC Roots 都走不到，它就成为可回收候选。这个思路比“引用计数法”更能处理循环引用，因为两个对象互相引用但都不再连接到根节点时，仍然会被判定不可达。

GC Roots 的关键是“根”从哪里来。常见根包括虚拟机栈中的局部变量，比如当前正在执行方法里的对象引用；方法区里类的静态变量引用，比如某个单例或静态缓存；常量池中的引用；本地方法栈里 JNI 持有的引用；以及 JVM 内部正在使用的一些对象。你可以把 GC Roots 想成系统当前仍然认可的入口，只要对象还挂在这些入口能触达的引用链上，GC 就不能随便回收它。

面试时要注意，可达性分析说的是“能不能被回收的判断基础”，不等于对象一不可达就立刻释放。对象从不可达到真正被清理，中间还可能经过标记、引用处理、finalize 处理、分代收集策略等步骤。比如软引用、弱引用、虚引用的处理时机不同；finalize 虽然不推荐使用，但理论上对象在第一次被判定不可达后还可能通过 finalize 重新把自己挂回引用链。实际回答里不需要展开太偏，但要知道“不可达”是进入回收流程，不是立刻消失。

这道题还经常追问内存泄漏。Java 的内存泄漏并不是对象没人引用却没回收，而是对象已经没有业务价值，却仍然被 GC Roots 间接引用。比如 ThreadLocal 没清理、静态 Map 做缓存不淘汰、监听器注册后不注销、连接池或集合持有对象引用，这些对象对业务来说该死了，但从 GC Roots 看仍然活着，所以 GC 不会回收。排查时可以用 jmap、MAT、JFR 或堆转储分析引用链，看对象被哪个 GC Root 路径留住。

图解建议画成“根节点森林”：顶部是栈变量、静态字段、JNI 引用、常量池，下面用箭头连到一批对象。能连上的对象标成存活，断开的对象放在另一侧标成可回收候选。这样一眼就能解释为什么循环引用不是问题，也能解释为什么静态集合、ThreadLocal 这类对象容易造成泄漏。

如果面试官继续问“为什么不用引用计数”，可以顺手说出循环引用这个缺陷：A 引用 B、B 引用 A，但它们都不再被任何根节点触达时，引用计数不为零，可达性分析却能准确发现它们已经脱离业务入口。再往项目里落一句：看 GC 问题时，不要只盯“对象很多”，要追引用链，找到是谁把对象留住。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 JVM 怎么判断一个对象可以被回收 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：JVM 主要通过可达性分析判断对象是否可回收：从 GC Roots 出发，如果一个对象沿着引用链走不到，就说明它不再被程序使用，可以被垃圾回收

拆开来看，第一层是背景问题：用可达性分析理解 GC Roots，避免只背引用计数。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：虚拟机栈中引用的对象，比如当前方法里的局部变量。 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：方法区中类静态属性引用的对象。 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：方法区中常量引用的对象。 不要只背概念名词，要能说出触发条件、底层机制和排查入口。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：当业务代码出现 Java、JVM、GC、内存 相关的并发异常、性能抖动、配置不生效、对象行为和预期不一致时，就可以用这道题定位原因。
- **怎么做方案**：先看触发条件，再看运行时机制。围绕“调用入口、对象状态、线程边界、框架代理、异常日志”五个位置检查，判断 JVM 怎么判断一个对象可以被回收 是设计问题、用法问题还是环境问题。
- **怎么验证效果**：用单元测试、压测、日志、线程栈、JFR/GC 日志或本地最小复现确认。结合真实业务代码说明什么时候会踩坑、怎么定位、怎么替换方案。
- **怎么兜底**：准备替代 API、隔离开关、降级策略、配置回滚和监控告警。面试里能讲出兜底，说明你不是只会写 happy path。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 JVM 怎么判断一个对象可以被回收 讲成一次工程决策，而不是一个孤立知识点。重点围绕 对象模型、运行时行为、边界条件和框架默认行为。

## 回答模板

面试时可以按这个节奏组织：

3. **补充边界**：说明什么情况下有效，什么情况下会失效，以及为什么需要配套措施。
