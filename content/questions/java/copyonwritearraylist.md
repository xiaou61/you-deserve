## 一句话结论

CopyOnWriteArrayList 的关键不是“线程安全”四个字，而是它把读写彻底分路：读看旧快照，写复制新数组，所以只适合读多写少且能接受短暂旧数据的场景。

## 通俗解释

它像公告栏。大家看公告不用排队；如果要改公告，就先复印一份新公告，改好后整体替换。

## 面试回答

回答时可以抓住三个关键词：快照、复制、写放大。

1. 读操作直接读取当前数组快照，基本不加锁，所以读性能和遍历体验很好。
2. 写操作会先加锁，再复制一份新数组，在新数组上修改后整体替换引用。
3. 这意味着它非常适合监听器列表、规则白名单、配置快照这类“读远多于写”的对象。
4. 代价是每次写入都要复制数组，数据量大或写频繁时，内存和 GC 压力会迅速上升。

## 常见追问

### 为什么说它读到的可能不是最新值？

因为读线程拿的是某个时刻的数组快照，写线程更新后，旧快照上的迭代器并不会自动切到新数组。这是它换取无锁读取的代价，也是为什么它不适合强实时数据。

### 什么时候不要用它？

高频写、长列表、队列型场景都不合适。比如消息缓冲、订单流转列表、实时排行榜，这些都更关心最新状态或写入效率，放到 CopyOnWriteArrayList 里成本会非常难看。

### 它和 `Collections.synchronizedList` 怎么选？

后者本质是同一份数据加互斥锁，读写都竞争同一把锁；前者让读完全走快照，写再付出复制成本。也就是说，一个是锁竞争重，一个是写放大重，选型要看读写比例。

### 迭代器为什么不能删元素？

因为迭代器持有的是快照视图，不希望你在这个旧视图上再去修改底层结构。能把“快照不可变”这个语义讲出来，比单纯背 API 限制更有味道。

## 易错点

- 不要把它用于高频写入队列。
- 读到的是快照，不保证强实时。
- 数据量很大时复制成本明显。
