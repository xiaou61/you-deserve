## 一句话结论

CAS 是 Compare And Swap，先比较内存值是不是预期值，是的话再更新。ABA 问题是值从 A 变成 B 又变回 A，CAS 只看当前值，会误以为没有变化。

## 通俗解释

CAS 像你去取快递前看柜子里是不是还是你的包裹。如果编号没变，你就拿走。

但 ABA 问题是：你的包裹先被别人拿走，又放回了一个编号一样但实际不同的包裹。你只看编号，以为没变，其实过程已经变过。

## 面试回答

CAS 是一种乐观并发控制方式。更新变量时，会读取当前值，并和预期值比较。如果一致，就把它更新成新值；如果不一致，说明被其他线程改过，就重试或失败。

Java 的很多原子类，比如 `AtomicInteger`，底层都用到了 CAS。

CAS 的问题主要有：

1. ABA 问题。值变走又变回来，CAS 无法感知中间变化。
2. 自旋开销。竞争激烈时反复重试会浪费 CPU。
3. 只能保证单个变量的原子操作，复杂场景仍然需要锁或其他机制。

ABA 可以通过版本号解决，比如 `AtomicStampedReference`，不仅比较值，还比较版本。

## 常见追问

### CAS 和 synchronized 有什么区别？

CAS 是乐观锁思想，不阻塞线程，失败后重试。synchronized 是悲观锁思想，进入临界区前先获取锁，竞争时线程可能阻塞。

### CAS 一定比锁快吗？

不一定。低竞争场景 CAS 很轻；高竞争场景可能一直自旋，CPU 开销反而很大。

## 易错点

- 不要说 CAS 没有问题，它有 ABA 和自旋成本。
- 不要把 CAS 说成 Java 独有，它是 CPU 级原子指令思想。
- 不要忽略版本号解决 ABA 的思路。

## 记忆钩子

**CAS 只看“现在是不是 A”，ABA 问题提醒你：过程也可能很重要。**
