## 一句话结论

CAS 是 Compare And Swap，先比较内存值是不是预期值，是的话再更新。ABA 问题是值从 A 变成 B 又变回 A，CAS 只看当前值，会误以为没有变化。

## 通俗解释

CAS 像你去取快递前看柜子里是不是还是你的包裹。如果编号没变，你就拿走。

但 ABA 问题是：你的包裹先被别人拿走，又放回了一个编号一样但实际不同的包裹。你只看编号，以为没变，其实过程已经变过。

## 面试回答

CAS 是一种乐观并发控制方式。更新变量时，会读取当前值，并和预期值比较。如果一致，就把它更新成新值；如果不一致，说明被其他线程改过，就重试或失败。

Java 的很多原子类，比如 `AtomicInteger`，底层都用到了 CAS。

CAS 的问题主要有：

1. ABA 问题。值变走又变回来，CAS 无法感知中间变化。
2. 自旋开销。竞争激烈时反复重试会浪费 CPU。
3. 只能保证单个变量的原子操作，复杂场景仍然需要锁或其他机制。

ABA 可以通过版本号解决，比如 `AtomicStampedReference`，不仅比较值，还比较版本。

## 常见追问

### CAS 和 synchronized 有什么区别？

CAS 是乐观锁思想，不阻塞线程，失败后重试。synchronized 是悲观锁思想，进入临界区前先获取锁，竞争时线程可能阻塞。

### CAS 一定比锁快吗？

不一定。低竞争场景 CAS 很轻；高竞争场景可能一直自旋，CPU 开销反而很大。

## 易错点

- 不要说 CAS 没有问题，它有 ABA 和自旋成本。
- 不要把 CAS 说成 Java 独有，它是 CPU 级原子指令思想。
- 不要忽略版本号解决 ABA 的思路。

## 详细讲解

CAS 是 compare and swap，也就是比较并交换。它是一种乐观锁思想：线程不先加互斥锁，而是读取当前值和期望值，更新时再判断内存里的值是否仍然等于期望值。如果相等，就把它改成新值；如果不相等，说明期间被别人改过，本次更新失败，通常会重试。Java 里的 AtomicInteger、AtomicReference 等原子类底层就大量使用 CAS。

CAS 的优势是避免阻塞。synchronized 或 Lock 在竞争时可能让线程挂起和唤醒，而 CAS 在用户态或 CPU 原子指令层面尝试更新，失败就自旋重试，适合计数器、状态标记、无锁队列等短小操作。它的典型模式是：读旧值、基于旧值计算新值、CAS 尝试写回。如果失败，说明旧值过期，再读一次重新计算。

但 CAS 不是万能的。第一个问题是自旋成本。竞争很激烈时，很多线程不断失败和重试，会消耗 CPU。第二个问题是只能保证一个变量或一个引用更新的原子性，涉及多个字段的一致性时就很麻烦。第三个问题就是 ABA：线程 1 读到值 A，准备 CAS；线程 2 把 A 改成 B，又改回 A。线程 1 再执行 CAS 时发现值还是 A，于是认为没有变化，更新成功。但实际上这个值中间经历过变化，相关对象状态可能已经完全不同。

ABA 在普通数字计数里未必有影响，但在栈、队列、对象引用、内存复用这类场景里风险很大。比如一个无锁栈的头节点从 A 变到 B，再变回 A，表面引用相同，但 A 节点的 next 关系或业务含义可能已经变化。CAS 只比较“当前值等不等于期望值”，无法知道中间发生过什么。

解决 ABA 的常见方法是加版本号或时间戳。AtomicStampedReference 会把引用和 stamp 一起比较，只有引用相同且版本号也相同才更新。这样 A -> B -> A 后，值虽然回到 A，但版本号已经变化，CAS 就能识别出它不是原来的 A。也可以用 AtomicMarkableReference 用布尔标记，或在业务上避免对象复用带来的歧义。

图解可以画成一条时间线：线程 1 读取 A 后暂停；线程 2 把 A 改成 B，再改回 A；线程 1 CAS 成功但误判。旁边再画版本号方案：A1 -> B2 -> A3，线程 1 期望 A1，发现当前是 A3，因此失败。最后落点：CAS 适合低冲突、短逻辑、单变量更新；高冲突或多变量一致性仍然要考虑锁、队列化或事务。回答时还可以补一句：乐观锁不是不加锁，而是把冲突检测推迟到提交那一刻，失败后要有重试或失败处理策略。线上如果 CAS 失败率很高，就要看是否需要降低竞争、分段计数或改用锁。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 CAS 是什么？ABA 问题怎么解决 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：CAS 是 Compare And Swap，先比较内存值是不是预期值，是的话再更新

拆开来看，第一层是背景问题：理解比较并交换的乐观锁思想，以及 ABA 为什么会骗过 CAS。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：ABA 问题。值变走又变回来，CAS 无法感知中间变化。 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：自旋开销。竞争激烈时反复重试会浪费 CPU。 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：只能保证单个变量的原子操作，复杂场景仍然需要锁或其他机制。 不要只背概念名词，要能说出触发条件、底层机制和排查入口。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：当业务代码出现 Java、CAS、ABA、原子类 相关的并发异常、性能抖动、配置不生效、对象行为和预期不一致时，就可以用这道题定位原因。
- **怎么做方案**：先看触发条件，再看运行时机制。围绕“调用入口、对象状态、线程边界、框架代理、异常日志”五个位置检查，判断 CAS 是什么？ABA 问题怎么解决 是设计问题、用法问题还是环境问题。
- **怎么验证效果**：用单元测试、压测、日志、线程栈、JFR/GC 日志或本地最小复现确认。结合真实业务代码说明什么时候会踩坑、怎么定位、怎么替换方案。
- **怎么兜底**：准备替代 API、隔离开关、降级策略、配置回滚和监控告警。面试里能讲出兜底，说明你不是只会写 happy path。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 CAS 是什么？ABA 问题怎么解决 讲成一次工程决策，而不是一个孤立知识点。重点围绕 对象模型、运行时行为、边界条件和框架默认行为。
