## 一句话结论

MyBatis 会为 Mapper 接口创建动态代理对象，方法调用时根据接口方法定位对应 SQL 并执行，所以不需要手写实现类。

## 通俗解释

Mapper 接口像菜单，XML 或注解 SQL 像后厨做法。你点菜单上的菜，服务员代理会去后厨按对应做法执行。

## 面试回答

MyBatis 启动时会解析 Mapper 接口和 XML 映射关系。运行时通过动态代理生成 Mapper 代理对象。

调用接口方法时，代理对象会根据 namespace、方法名、参数等找到 mapped statement，然后通过 SqlSession 执行 SQL，并把结果映射成返回对象。

## 常见追问

### XML 的 namespace 有什么用？

通常对应 Mapper 接口全限定名，用来把接口方法和 SQL 语句绑定起来。

### 方法重载要注意什么？

MyBatis 根据方法和映射语句绑定，重载可能带来映射不清晰，不建议在 Mapper 中滥用重载。

## 易错点

- 不要说 Mapper 接口“自动实现”就结束，要讲动态代理。
- 不要忽略 namespace 和方法绑定。
- 不要把 Mapper 当普通 DAO 实现类。

## 详细讲解

MyBatis Mapper 接口之所以不用手写实现类，本质上是因为运行时动态代理已经替你把“实现类里本该写的那些转调逻辑”生成好了。你在代码里只看到一个接口方法，比如 selectById，但真正执行时，Spring 或 MyBatis 给你的并不是空接口，而是一个代理对象。这个代理知道如何把方法调用翻译成 SQL 执行。  

执行过程可以拆成几步。首先，框架扫描到 Mapper 接口后，会为它注册代理工厂；当你注入这个接口时，拿到的是代理实例。随后，当你调用某个方法，代理会根据接口全限定名、方法名和参数，去定位对应的 mapped statement，也就是 XML 或注解里定义的那条 SQL 映射。找到映射后，它再借助 SqlSession 进入 MyBatis 执行链路。  

接下来就是典型的执行流程：SqlSession 把请求交给 Executor，Executor 再经过参数处理、SQL 生成、JDBC 执行、结果集映射，最后把结果转回你接口方法声明的返回类型。也就是说，Mapper 接口方法只是一个声明式入口，真正的实现逻辑在 MyBatis 的代理和执行框架里。  

这套机制的价值非常大。它把原来大量重复的 DAO 样板代码消掉了：你不需要自己写实现类去获取连接、准备语句、执行 SQL、映射结果。你只要专注于“我需要哪些查询方法”和“SQL 应该怎么写”。这也是 MyBatis 非常适合业务 CRUD 场景的原因之一。  

当然，这种便利也有代价。方法名和 SQL 映射之间虽然看起来像静态关联，但本质是运行时绑定。如果 namespace、方法名、参数类型、返回映射不一致，问题常在运行时才暴露。面试里如果能主动说出“声明式简化了实现，但也要求映射配置保持严格一致”，会显得很稳。  

图解可以画成一条调用链：业务代码调用 Mapper 接口 -> 动态代理接住方法 -> 定位 mapped statement -> SqlSession/Executor 执行 SQL -> 结果映射返回。最后落一句：不是 Mapper 真有实现类，而是代理把方法调用翻译成了 SQL 执行流程。
