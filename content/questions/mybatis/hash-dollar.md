## 一句话结论

`#{}` 是预编译占位符，会作为参数绑定，能防 SQL 注入；`${}` 是字符串直接拼接，存在 SQL 注入风险。

## 通俗解释

`#{}` 像把用户输入装进安全盒子再交给数据库；`${}` 像直接把用户输入粘进 SQL 语句里。用户如果输入恶意内容，SQL 就可能被改写。

## 面试回答

`#{}` 会被 MyBatis 处理成 `?` 占位符，通过 PreparedStatement 设置参数。它会进行类型处理和安全绑定。

`${}` 是文本替换，会把内容直接拼到 SQL 里。它适合少数无法使用占位符的位置，比如动态表名、排序字段，但必须做白名单校验。

普通查询条件应该优先使用 `#{}`。

## 常见追问

### order by 字段能用 #{} 吗？

通常不能直接用 `#{}` 表示列名，因为它会被当成字符串参数。动态列名可以用 `${}`，但必须白名单限制。

### SQL 注入怎么防？

使用预编译参数、白名单校验、避免拼接用户输入，并控制数据库权限。

## 易错点

- 不要为了动态方便滥用 `${}`。
- 不要把列名和值参数混在一起。
- 动态排序字段必须白名单。

## 详细讲解

MyBatis 里 #{} 和 ${} 的区别，核心不在“一个有引号一个没引号”，而在于它们进入数据库的方式完全不同。#{} 会被 MyBatis 处理成 JDBC 预编译参数占位符 ?，真正的值不是直接拼进 SQL 文本，而是在执行阶段单独绑定进去。${} 则是纯字符串替换，MyBatis 在生成最终 SQL 时直接把文本原样拼接进去。这个差异决定了安全性、可维护性和适用场景。  

先看 #{}。比如 where id = #{id}，最终发给数据库的 SQL 类似 where id = ?，参数 id 由 PreparedStatement 绑定。这样做有三个好处：第一，天然防 SQL 注入，因为参数值不会被当成 SQL 语法解释；第二，数据库可以利用预编译执行计划；第三，MyBatis 会配合 TypeHandler 做类型转换，比如 Java 时间类型、枚举、自定义对象都能被正确映射。绝大多数普通查询条件都应该用 #{}。  

${} 则完全不同。比如 order by ${column}，如果传入 name desc，最终 SQL 就真的会拼成 order by name desc。这使得 ${} 适合处理不能作为预编译参数的位置，例如动态表名、动态列名、动态排序关键字。因为数据库不允许把表名或列名写成 ? 占位符，所以这种结构性变化只能通过文本拼接实现。  

但正因为 ${} 是拼文本，所以风险非常高。如果用户输入未经校验就进入 ${}，就可能构造恶意 SQL。最常见的是动态排序字段、动态表名这种需求，很多人偷懒直接把前端传来的字段名塞进 ${}，这几乎等于把 SQL 语法控制权交给外部。正确做法是白名单校验：前端传的是业务字段标识，后端把它映射成有限可选列名，而不是原样拼接。  

面试时不要只说“#{} 防注入，${} 不防”，还要说出为什么。#{} 的本质是参数绑定，${} 的本质是文本替换。再补一条实践原则：只要是值，优先 #{}；只有 SQL 结构本身必须动态变化时，才考虑 ${}，而且一定配白名单。  

项目里还可以补一个代码审查视角：只要 Mapper 里出现 ${}，就应该下意识问一句“这个值从哪里来，是否经过枚举映射或白名单”。很多安全问题不是框架不会防，而是开发者把本该用 #{} 的地方写成了 ${}。这句一说，答案就从语法差异升级成安全意识。  

图解可以画成两条链路：#{} 路线是“Mapper 参数 -> MyBatis -> SQL 中的 ? -> JDBC 绑定”；${} 路线是“Mapper 参数 -> 直接拼接到 SQL 字符串 -> 数据库解析执行”。旁边用红色标出 ${} 的注入风险点。最后落一句工程经验：不是禁止 ${}，而是要把它关在强校验和有限场景里。这样就不是死背八股，而是能落到真实代码审查。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 MyBatis 中 #{} 和 ${} 有什么区别 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：#{} 是预编译占位符，会作为参数绑定，能防 SQL 注入；${} 是字符串直接拼接，存在 SQL 注入风险

拆开来看，第一层是背景问题：理解预编译参数和字符串拼接的区别，以及为什么 ${} 有 SQL 注入风险。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：#{} 会被 MyBatis 处理成 ? 占位符，通过 PreparedStatement 设置参数 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：它会进行类型处理和安全绑定 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

## 实战落地

- **什么时候会遇到**：当业务代码出现 MyBatis、SQL注入、预编译 相关的并发异常、性能抖动、配置不生效、对象行为和预期不一致时，就可以用这道题定位原因。
- **怎么做方案**：先看触发条件，再看运行时机制。围绕“调用入口、对象状态、线程边界、框架代理、异常日志”五个位置检查，判断 MyBatis 中 #{} 和 ${} 有什么区别 是设计问题、用法问题还是环境问题。
- **怎么验证效果**：用单元测试、压测、日志、线程栈、JFR/GC 日志或本地最小复现确认。结合真实业务代码说明什么时候会踩坑、怎么定位、怎么替换方案。
- **怎么兜底**：准备替代 API、隔离开关、降级策略、配置回滚和监控告警。面试里能讲出兜底，说明你不是只会写 happy path。
