## 一句话结论

联合索引按照字段顺序建立有序结构，查询要从最左边字段开始匹配，才能利用这棵有序的 B+ 树快速定位数据。

## 通俗解释

联合索引 `(a, b, c)` 很像一本按“省、市、区”排序的通讯录。你知道省，就能快速缩小范围；知道省和市，范围更小。

但如果你只知道市，不知道省，就很难直接查。因为通讯录不是按市单独排序的，而是先按省排，再按市排。

所以 `(a, b, c)` 可以支持：

- `where a = ?`
- `where a = ? and b = ?`
- `where a = ? and b = ? and c = ?`

但通常不能很好支持：

- `where b = ?`
- `where c = ?`

## 面试回答

MySQL InnoDB 的联合索引底层是 B+ 树，索引项会按照定义字段的顺序排序。比如 `(a, b, c)`，先按 a 排序，a 相同时再按 b 排序，b 相同时再按 c 排序。

因此查询条件必须从最左字段开始连续匹配，才能利用索引的有序性。如果跳过 a 直接查 b，b 在整棵索引树上并不是全局有序的，数据库就不能直接通过 B+ 树快速定位。

## 常见追问

### 范围查询会发生什么？

如果联合索引是 `(a, b, c)`，条件写成 `a = 1 and b > 10 and c = 3`，通常能稳定用到 a、b，但 c 往往很难继续参与“定位”，更多只是做过滤。原因是 b 一旦进入范围扫描，后面的 c 在这个范围里不再具备连续有序性。继续往下讲时，最好把“用于定位”和“用于过滤”区分开，不然很容易被面试官追着问。

### 查询条件顺序和索引顺序必须一样吗？

SQL 写在前面的条件不一定决定执行顺序，优化器会重排；真正关键的是联合索引定义顺序，以及查询能不能形成连续的最左匹配。这个追问继续展开时，可以顺手补上“最左前缀说的是索引结构，不是 where 子句的书写顺序”，这样能顺手纠正常见误区。

### 联合索引字段顺序到底怎么设计？

不能只按“最左前缀”机械排，要结合高频查询、等值条件、区分度、范围列、排序列和覆盖索引需求一起看。通常高频等值过滤、区分度较高的列更适合靠前；范围列放太前，后面的列利用率会下降。这个追问是从“会用规则”升级到“会设计索引”的关键一跳。

### 线上怎么判断一个联合索引其实没发挥作用？

最直接是看 `EXPLAIN`：type、key、rows、extra 是否符合预期，再看慢 SQL、回表次数、filesort、临时表这些信号。如果你以为命中了联合索引，但实际上扫描行数仍然很高，说明可能跳列、范围过早、排序不匹配，或者优化器觉得走别的索引更便宜。

## 易错点

- 最左前缀说的是索引字段顺序，不是 SQL 书写顺序。
- 范围查询后面的字段不一定还能继续用于索引定位。
- `like 'abc%'` 通常可以用索引，`like '%abc'` 通常不行。

## 详细讲解

联合索引遵守最左前缀，本质来自 B+Tree 的排序方式。假设有联合索引 (a, b, c)，索引项并不是分别建三棵树，而是把 a、b、c 拼成一个复合 key 放进一棵 B+Tree 里排序。排序规则是先按 a 排；a 相同的情况下再按 b 排；a 和 b 都相同的情况下再按 c 排。也就是说，b 和 c 的有序性是建立在前面列已经确定的前提上。

所以查询条件里如果有 a，可以在树上快速定位 a 的范围；如果同时有 a 和 b，可以在 a 的范围里继续缩小 b；如果有 a、b、c，就能继续精确定位。但如果跳过 a 直接查 b，整棵树并不是全局按 b 排的，MySQL 很难利用这棵索引快速定位，只能扫描更多索引项甚至放弃索引。这就是“最左”的含义：从索引定义的第一列开始，连续命中才最有效。

范围条件也是常见坑。比如 where a = 1 and b > 10 and c = 3，索引可以先用 a 定位，再用 b 做范围扫描，但到了 b 是范围后，c 通常不能继续用于缩小定位范围。原因是 b 大于 10 的区间里，c 的值分布在不同 b 值下面，并不是一个可以直接跳转的连续区间。面试里可以说“范围条件后的列可能无法继续用于索引定位”，但要注意 MySQL 还有 ICP 等优化，可能会在存储引擎层用 c 做过滤；这和“继续定位”不是一回事。

联合索引还能服务排序和分组。order by 如果和联合索引顺序一致，并且 where 条件把前导列约束住，就可能避免 filesort。比如索引 (user_id, create_time)，查询某个用户的订单并按 create_time 排序，就很适合。但如果 order by 的列顺序和索引不匹配，或者中间跳列，排序优势就会下降。

设计索引时不要机械地把所有字段塞进联合索引。要看查询模式：高频过滤条件、等值条件、区分度、范围条件、排序字段、覆盖索引需求都要一起考虑。一般等值过滤且区分度高的列更适合靠前；范围列放得太前，可能影响后续列的使用；只为低频查询建很长索引，会增加写入成本和存储成本。

图解可以画一棵联合索引树：第一层按 a 分区，第二层在同一个 a 里按 b 排，第三层在同一个 a、b 里按 c 排。然后画三条查询路线：a+b+c 是顺着树走；a+c 是中间断了一段；b+c 是直接跳过根排序。这样最左前缀就不再是口诀，而是 B+Tree 排序规则的自然结果。

真正设计索引时，还要结合业务查询频率。一个偶尔使用的复杂查询，不值得牺牲大量写入成本；一个核心列表页或账单查询，则值得为了过滤、排序和覆盖索引专门设计联合索引。这个工程取舍是面试加分点。

第三层是边界和取舍：比如 (a, b, c)，先按 a 排序，a 相同时再按 b 排序，b 相同时再按 c 排序 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。
