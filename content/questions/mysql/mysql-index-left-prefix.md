## 一句话结论

联合索引按照字段顺序建立有序结构，查询要从最左边字段开始匹配，才能利用这棵有序的 B+ 树快速定位数据。

## 通俗解释

联合索引 `(a, b, c)` 很像一本按“省、市、区”排序的通讯录。你知道省，就能快速缩小范围；知道省和市，范围更小。

但如果你只知道市，不知道省，就很难直接查。因为通讯录不是按市单独排序的，而是先按省排，再按市排。

所以 `(a, b, c)` 可以支持：

- `where a = ?`
- `where a = ? and b = ?`
- `where a = ? and b = ? and c = ?`

但通常不能很好支持：

- `where b = ?`
- `where c = ?`

## 面试回答

MySQL InnoDB 的联合索引底层是 B+ 树，索引项会按照定义字段的顺序排序。比如 `(a, b, c)`，先按 a 排序，a 相同时再按 b 排序，b 相同时再按 c 排序。

因此查询条件必须从最左字段开始连续匹配，才能利用索引的有序性。如果跳过 a 直接查 b，b 在整棵索引树上并不是全局有序的，数据库就不能直接通过 B+ 树快速定位。

## 常见追问

### 这类 MySQL 题最容易把哪个边界讲错？

最常见的是把“理论结论”直接套到所有执行计划上。像「MySQL 联合索引为什么要遵守最左前缀」这种题，数据分布、索引命中、统计信息、排序方向和 limit 位置一变，结论就可能变样。

### 线上排查时优先看哪些证据？

先看 EXPLAIN / EXPLAIN ANALYZE、rows、filtered、slow log、临时表、锁等待和事务状态，再决定是改索引、改 SQL，还是改访问路径。MySQL 题最好用证据说话。

### 为什么小表上看起来没问题，线上却会炸？

因为数据量、冷热分布、回表成本、磁盘临时文件和长事务都会在大规模下放大。面试里主动说这句，会比单纯背索引规则更像做过性能优化。

### 如果要讲成项目经历，怎么组织？

可以按“现象 -> SQL -> 执行计划 -> 为什么慢/锁/扫这么多 -> 我改了什么 -> 指标怎么回稳”这条线讲 MySQL 联合索引为什么要遵守最左前缀，会非常顺。

## 易错点

- 最左前缀说的是索引字段顺序，不是 SQL 书写顺序。
- 范围查询后面的字段不一定还能继续用于索引定位。
- `like 'abc%'` 通常可以用索引，`like '%abc'` 通常不行。
