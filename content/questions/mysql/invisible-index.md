## 一句话结论

Invisible Index 是让索引暂时对优化器不可见，常用于评估删除索引是否会影响查询，而不是真的立刻删掉索引。

## 通俗解释

它像把工具先收进柜子但不扔掉。看看大家没有这个工具能不能正常干活，不行就立刻拿出来。

## 面试回答

索引太多会增加写入成本和存储成本，但直接删除索引有风险。不可见索引提供了一个灰度方式：

- 把某个索引设为 invisible。
- 优化器默认不再选择这个索引。
- 观察慢 SQL、执行计划和业务指标。
- 如果没影响，再考虑真正删除。
- 如果有影响，快速改回 visible。

它适合索引治理和性能优化验证，尤其在不确定索引是否被重要查询依赖时很有用。

## 常见追问

### 这题最容易被追问的边界是什么？

重点说清索引失效、锁范围扩大、主从延迟、统计信息过期、深分页和 DDL 风险。不要只给结论，要说明哪些条件下结论成立，哪些条件下会退化或需要换方案。

### 怎么证明自己不是在背模板？

用EXPLAIN、慢日志、optimizer trace、锁等待、事务状态、redo/undo/binlog 和复制延迟做验证。能说出具体信号、反例或排查入口，答案就从概念层进入实战层。

### 和相近方案怎么区分？

可以拿索引优化、SQL 改写、分页改造、读写分离、分区分库或归档策略对比，从查询性能、锁影响、一致性、扩展性和回滚难度几个维度选择。

### 面试官继续深挖时怎么展开？

先围绕“把某个索引设为 invisible。”讲清主链路，再补“补 MySQL 8.0 支持、ALTER TABLE ... ALTER INDEX ... INVISIBLE/VISIBLE、optimizer_switch='use_invisible_indexes=on' 测试方式、写入成本仍存在、与 drop index/online DDL 的区别、观察慢 SQL 和执行计划的窗口期”。如果能把边界条件、异常分支和验证闭环连起来，就能接住二面、三面的追问。

## 易错点

- 不可见索引仍有写入维护成本。
- 不要把它当成空间优化手段。
- 变更后要观察执行计划和慢 SQL。
