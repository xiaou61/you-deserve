## 一句话结论

慢 SQL 排查先定位慢日志，再看 EXPLAIN 执行计划，重点分析索引命中、扫描行数、排序临时表、锁等待和返回数据量。

## 通俗解释

排查慢 SQL 像查堵车。先确定哪条路堵，再看是没走高速、车太多、红绿灯太久，还是路口被占住。

## 面试回答

常见排查步骤：

1. 开启或查看慢查询日志，定位具体 SQL。
2. 使用 EXPLAIN 看执行计划。
3. 关注 type、key、rows、Extra 等字段。
4. 判断是否索引失效、扫描行数过大、出现 filesort 或临时表。
5. 检查 SQL 是否返回过多数据、分页过深、join 条件不合理。
6. 如果偶发变慢，还要看锁等待、机器负载和 buffer pool 命中率。

优化方式包括加合适索引、改写 SQL、减少返回字段、拆分大查询、优化分页和避免大事务。

## 常见追问

### EXPLAIN 里 rows 是精确值吗？

不是，它是优化器估算扫描行数，但对判断查询成本很有参考价值。

### 加索引一定能解决慢 SQL 吗？

不一定。索引设计不合理、返回数据太多、排序分组代价高、锁等待都可能导致慢。

## 易错点

- 不要一看到慢 SQL 就只说“加索引”。
- 不要忽略返回数据量和网络传输。
- 偶发慢要考虑锁、IO、CPU、连接池等因素。

## 记忆钩子

**慢 SQL 先定位，再看计划，最后对症优化。**
