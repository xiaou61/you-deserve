## 一句话结论

慢 SQL 排查先定位慢日志，再看 EXPLAIN 执行计划，重点分析索引命中、扫描行数、排序临时表、锁等待和返回数据量。

## 通俗解释

排查慢 SQL 像查堵车。先确定哪条路堵，再看是没走高速、车太多、红绿灯太久，还是路口被占住。

## 面试回答

常见排查步骤：

1. 开启或查看慢查询日志，定位具体 SQL。
2. 使用 EXPLAIN 看执行计划。
3. 关注 type、key、rows、Extra 等字段。
4. 判断是否索引失效、扫描行数过大、出现 filesort 或临时表。
5. 检查 SQL 是否返回过多数据、分页过深、join 条件不合理。
6. 如果偶发变慢，还要看锁等待、机器负载和 buffer pool 命中率。

优化方式包括加合适索引、改写 SQL、减少返回字段、拆分大查询、优化分页和避免大事务。

## 常见追问

### EXPLAIN 里 rows 是精确值吗？

不是，它是优化器估算扫描行数，但对判断查询成本很有参考价值。

### 加索引一定能解决慢 SQL 吗？

不一定。索引设计不合理、返回数据太多、排序分组代价高、锁等待都可能导致慢。

## 易错点

- 不要一看到慢 SQL 就只说“加索引”。
- 不要忽略返回数据量和网络传输。
- 偶发慢要考虑锁、IO、CPU、连接池等因素。

## 实战落地

- **什么时候会遇到**：当业务里出现 MySQL、慢 SQL、EXPLAIN、索引 相关的慢查询、锁等待、热点 key、缓存不一致、内存上涨或写入抖动时，就要把这道题放到读写链路里分析。
- **怎么做方案**：先明确数据规模和访问模式，再选择索引、事务、缓存、过期策略或分片策略。围绕“读路径、写路径、并发控制、失败补偿、容量上限”五个点展开。
- **怎么验证效果**：看 explain、慢日志、锁等待、命中率、内存、QPS、P99 延迟和错误率。把答案落到慢查询、热点 key、事务隔离、缓存一致性或容量评估上。
- **怎么兜底**：准备降级读、限流、重建索引、缓存预热、补偿任务或数据修复脚本，避免单点方案拖垮主链路。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 MySQL 慢 SQL 怎么排查 讲成一次工程决策，而不是一个孤立知识点。重点围绕 数据结构、读写路径、一致性边界和性能影响。
