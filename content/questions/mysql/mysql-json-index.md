## 一句话结论

JSON 灵活但不天然适合任意条件高效查询，常查字段要抽出来形成可索引表达。

## 通俗解释

JSON 索引像给大包裹里的固定物品贴外部标签，不然每次都要拆包找。

## 面试回答

可以从这几层回答：

- MySQL 5.7 常用生成列提取 JSON 路径，再给生成列建索引。
- MySQL 8 可使用函数索引等能力简化部分场景。
- 高频查询字段更建议拆成普通列。
- JSON 适合扩展属性，不适合承载核心关系模型。

回答时最好补一句：这个点不是孤立知识点，真正落地时要结合业务场景、数据规模和失败兜底，说明你不是只背概念。

## 常见追问

### 这题最容易被追问的边界是什么？

重点说清索引失效、锁范围扩大、主从延迟、统计信息过期、深分页和 DDL 风险。不要只给结论，要说明哪些条件下结论成立，哪些条件下会退化或需要换方案。

### 怎么证明自己不是在背模板？

用EXPLAIN、慢日志、optimizer trace、锁等待、事务状态、redo/undo/binlog 和复制延迟做验证。能说出具体信号、反例或排查入口，答案就从概念层进入实战层。

### 和相近方案怎么区分？

可以拿索引优化、SQL 改写、分页改造、读写分离、分区分库或归档策略对比，从查询性能、锁影响、一致性、扩展性和回滚难度几个维度选择。

### 面试官继续深挖时怎么展开？

先围绕“MySQL 5.7 常用生成列提取 JSON 路径，再给生成列建索引。”讲清主链路，再补“补：虚拟/存储生成列的取舍、JSON_EXTRACT/JSON_UNQUOTE 的类型和字符集问题、函数索引适用边界、JSON 数组查询可考虑多值索引、核心关系字段不要长期藏在 JSON 里”。如果能把边界条件、异常分支和验证闭环连起来，就能接住二面、三面的追问。

## 易错点

- 所有扩展字段都塞 JSON，后续查询很痛苦。
- 以为 JSON_EXTRACT 条件一定能走索引。

## 详细讲解

MySQL JSON 字段怎么建索引？ 这道题现在要从“能背出来”修到“能接住追问”。核心结论是：MySQL JSON 字段通常通过生成列或函数索引提取路径后建索引。回答时先把它放回MySQL语境，说明它解决什么矛盾，再围绕执行计划、索引事务、锁范围、统计指标和数据一致性展开。这样能避免泛泛而谈，也能让面试官听出你知道这题的真实边界。

第一步是拆清问题背景。JSON 索引像给大包裹里的固定物品贴外部标签，不然每次都要拆包找。在MySQL题里，背景不能写成空泛的工程套话，而要落到把 SQL、优化器、索引结构、事务隔离和存储引擎行为串起来。如果只说“看场景”或“加组件”，读者很难知道下一步该检查什么；如果能把输入、状态、依赖和输出说清，答案就有了主线。

第二步是讲机制。可以围绕这些点展开：MySQL 5.7 常用生成列提取 JSON 路径，再给生成列建索引。；MySQL 8 可使用函数索引等能力简化部分场景。；高频查询字段更建议拆成普通列。；JSON 适合扩展属性，不适合承载核心关系模型。。每个点都要回答三个问题：它在链路里的位置是什么，它改变了什么状态，失败或边界条件下会留下什么现象。把这三件事讲清楚，追问从概念跳到实战时就不会断。

第三步是补边界。这里尤其要主动说明索引失效、锁范围扩大、主从延迟、统计信息过期、深分页和 DDL 风险。所有扩展字段都塞 JSON，后续查询很痛苦。；以为 JSON_EXTRACT 条件一定能走索引。很多八股答案的问题不是方向错，而是没有说适用范围；一旦遇到数据规模变化、异常分支或版本差异，原来的结论就可能不成立。

第四步是补专项深度：补：虚拟/存储生成列的取舍、JSON_EXTRACT/JSON_UNQUOTE 的类型和字符集问题、函数索引适用边界、JSON 数组查询可考虑多值索引、核心关系字段不要长期藏在 JSON 里。这部分适合放在面试回答后半段，用来证明你不仅知道标准答案，也知道高频追问会往哪里挖。

最后收束成一套回答节奏：先给结论，再讲机制，再补边界，最后说验证方式。对这题来说，验证可以看EXPLAIN、慢日志、optimizer trace、锁等待、事务状态、redo/undo/binlog 和复制延迟；方案选择可以和索引优化、SQL 改写、分页改造、读写分离、分区分库或归档策略对比。这样既保留八股题的清晰度，也能把答案讲成可落地、可排查、可复盘的经验。
