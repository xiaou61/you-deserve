## 一句话结论

MySQL 主从复制的核心是主库写 binlog，从库拉取 binlog 并重放，从而把主库数据同步到从库。

## 通俗解释

主库像老师在黑板上写题，从库像学生抄笔记。老师每写一步，学生按顺序抄下来，最后笔记内容尽量一致。

## 面试回答

主从复制大致流程：

1. 主库提交事务时写入 binlog。
2. 从库 IO 线程连接主库，拉取 binlog。
3. 从库把日志写入 relay log。
4. 从库 SQL 线程读取 relay log 并重放。

常见用途是读写分离、备份、容灾。但主从复制通常存在延迟，所以读写分离时要注意“刚写完马上读”的一致性问题。

## 常见追问

### 主从延迟怎么处理？

可以强制读主库、延迟敏感业务不走从库、监控复制延迟、优化慢 SQL、减少大事务。

### binlog 有什么作用？

binlog 记录数据库变更，可用于复制、恢复、审计和同步到其他系统。

## 易错点

- 不要说主从复制完全实时强一致。
- 不要忽略 relay log。
- 读写分离要考虑主从延迟导致的读旧数据。

## 记忆钩子

**主库写日记，从库抄日记，抄慢了就有延迟。**
