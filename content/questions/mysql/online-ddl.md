## 一句话结论

Online DDL 是 MySQL 尽量在不中断或少阻塞业务的情况下执行表结构变更的能力，但不是所有 DDL 都完全无锁。

## 通俗解释

它像商场不停业装修。能边营业边改造，但有些施工还是会临时封路，不能想当然。

## 面试回答

传统 DDL 可能长时间锁表，影响读写。Online DDL 尝试降低这种影响。

实际要关注：

- 操作类型是否支持 online。
- 是否会复制整张表。
- 是否会长时间持有元数据锁。
- 表数据量和并发写入压力。
- 是否有回滚和灰度方案。

生产大表变更通常要提前评估，必要时使用在线变更工具或分阶段变更。

## 常见追问

### 这类 MySQL 题最容易把哪个边界讲错？

最常见的是把“理论结论”直接套到所有执行计划上。像「MySQL Online DDL 是什么」这种题，数据分布、索引命中、统计信息、排序方向和 limit 位置一变，结论就可能变样。

### 线上排查时优先看哪些证据？

先看 EXPLAIN / EXPLAIN ANALYZE、rows、filtered、slow log、临时表、锁等待和事务状态，再决定是改索引、改 SQL，还是改访问路径。MySQL 题最好用证据说话。

### 为什么小表上看起来没问题，线上却会炸？

因为数据量、冷热分布、回表成本、磁盘临时文件和长事务都会在大规模下放大。面试里主动说这句，会比单纯背索引规则更像做过性能优化。

### 如果要讲成项目经历，怎么组织？

可以按“现象 -> SQL -> 执行计划 -> 为什么慢/锁/扫这么多 -> 我改了什么 -> 指标怎么回稳”这条线讲 MySQL Online DDL 是什么，会非常顺。

## 易错点

- 不要在生产大表上随便执行 DDL。
- Online 不等于零影响。
- 表结构变更要配合应用兼容发布。

## 详细讲解
MySQL Online DDL 是什么 这类 MySQL 题最怕把“书上的规律”直接当成线上结论。更稳的讲法是先把 SQL 放进执行链：优化器会怎么选路径，索引能不能帮你减少扫描，最终是卡在排序、回表、锁等待还是临时表。先用一句话压住主题，比如 Online DDL 尽量在表结构变更时减少锁表和业务阻塞，但不同操作支持程度不同，然后马上把问题锚到执行计划和数据分布。

主线部分最好围绕一条具体 SQL 或一个 EXPLAIN 结果来讲。拿“操作类型是否支持 online”当切口，说明 Extra、type、rows、filtered、索引顺序和 limit 位置分别意味着什么。MySQL 题一旦脱离具体证据，就很容易变成泛泛而谈；反过来，只要能把执行计划和索引路径讲顺，内容就会立刻扎实很多。

真正让线上变慢的，往往不是一个单点配置，而是多个代价叠在一起。这里适合主动补 不要在生产大表上随便执行 DDL、Online 不等于零影响、表结构变更要配合应用兼容发布 这些风险，再说明它们在大表、深分页、冷热分布不均或长事务场景下会怎么被放大。这样读者不只知道“有 filesort/回表/锁”，还知道为什么测试环境不明显，线上却会炸。

继续深挖时，面试官一般会顺着 这类 MySQL 题最容易把哪个边界讲错？、线上排查时优先看哪些证据？、为什么小表上看起来没问题，线上却会炸？ 往下问。你可以直接说：我会先看 EXPLAIN ANALYZE、慢日志、锁等待、事务状态、临时表和复制延迟，再决定是改索引、改 SQL、改分页方式，还是拆访问路径。MySQL 题最加分的地方，就是你知道下一步该拿什么证据。

最后收口时，把 MySQL Online DDL 是什么 讲成一次性能排查会更顺：先看到什么现象，哪条 SQL 可疑，执行计划说明了什么，为什么原来的索引或写法不够用，改完以后又看哪些指标确认确实回稳。这样文章既保留知识点，也保留了真正的使用场景。

MySQL 题真正有用的部分，通常是你能不能把 Online DDL 尽量在表结构变更时减少锁表和业务阻塞，但不同操作支持程度不同 和执行计划、索引路径、锁范围、慢日志这些证据连起来。用户读这类内容，最想要的不是一个大而化之的结论，而是“为什么这条 SQL 会慢、会锁、会扫这么多行，以及我接下来该怎么验证”。把这一层补出来，文章的可用性会高很多。

再补一层取舍会更像真实优化。比如为了让 MySQL Online DDL 是什么 更稳，可能会改索引、改 SQL、换分页方式，甚至拆冷热数据。但每种改法都会带来写放大、维护成本、回滚难度或者兼容风险。把这个取舍说出来，文章就不只是“会分析”，而是“会决策”。
