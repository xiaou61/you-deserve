## 一句话结论

深分页慢通常是因为 `limit offset, size` 需要扫描并丢弃大量前置数据，可以用游标分页、延迟关联、记录上次位置等方式优化。

## 通俗解释

普通分页像从第一本书开始数到第 10000 本再拿 10 本。页数越深，前面白数的越多，自然越来越慢。

## 面试回答

深分页的典型问题是：

```sql
select * from user order by id limit 100000, 20;
```

数据库可能需要扫描很多行，再丢弃前 100000 行，只返回 20 行。

优化方式：

- 使用基于主键或索引的游标分页：`where id > last_id limit 20`。
- 延迟关联：先查索引列定位 id，再回表查完整数据。
- 避免过深页码跳转，产品层限制最大页数。
- 保证排序字段有合适索引。

### 这题最容易被追问的边界是什么？

重点说清索引失效、锁范围扩大、主从延迟、统计信息过期、深分页和 DDL 风险。不要只给结论，要说明哪些条件下结论成立，哪些条件下会退化或需要换方案。

### 怎么证明自己不是在背模板？

用EXPLAIN、慢日志、optimizer trace、锁等待、事务状态、redo/undo/binlog 和复制延迟做验证。能说出具体信号、反例或排查入口，答案就从概念层进入实战层。

### 和相近方案怎么区分？

可以拿索引优化、SQL 改写、分页改造、读写分离、分区分库或归档策略对比，从查询性能、锁影响、一致性、扩展性和回滚难度几个维度选择。

### 面试官继续深挖时怎么展开？

先围绕“select * from user order by id limit 100000, 20;”讲清主链路，再补“补执行成本：limit offset, size 通常要扫描 offset+size 后丢弃前 offset 行”。如果能把边界条件、异常分支和验证闭环连起来，就能接住二面、三面的追问。

## 易错点

- 不要只说加索引，要解释 offset 丢弃数据的问题。
- 游标分页需要稳定排序字段。
- 业务上能限制深翻页时，限制本身也是优化。
