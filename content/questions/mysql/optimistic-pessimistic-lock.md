## 一句话结论

悲观锁认为冲突经常发生，所以先锁住再操作；乐观锁认为冲突较少，所以提交时通过版本号或条件判断是否冲突。

## 通俗解释

悲观锁像进房间先反锁门，别人等着；乐观锁像大家都先写，提交时看版本是不是被别人改过。

## 面试回答

悲观锁常见方式：

- `select ... for update`。
- 数据库行锁。

适合冲突频繁、不能接受重试成本的场景。

乐观锁常见方式：

- version 字段。
- update 时带旧版本条件。
- 更新失败说明被别人改过，再重试或提示用户。

适合读多写少、冲突不频繁的场景。

## 常见追问

### 乐观锁怎么写 SQL？

类似 `update product set stock = stock - 1, version = version + 1 where id = ? and version = ?`。

### 悲观锁一定更安全吗？

不一定。它会降低并发，还可能造成锁等待和死锁。

## 易错点

- 乐观锁失败后要处理重试或提示。
- 悲观锁要放在事务中才有意义。
- 选择锁策略要看冲突频率和业务成本。

## 记忆钩子
