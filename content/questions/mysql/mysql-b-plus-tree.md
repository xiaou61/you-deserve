## 一句话结论

MySQL 使用 B+ 树做索引，是因为它树高低、磁盘 IO 少、范围查询友好，适合数据库这种大量磁盘读写场景。

## 通俗解释

索引不是放在内存里的小目录，它经常要和磁盘打交道。磁盘读取很贵，所以我们希望查一次数据尽量少读几次磁盘。

B+ 树每个节点可以放很多 key，树就不会太高，查找时需要的磁盘 IO 次数更少。

## 面试回答

B+ 树适合数据库索引主要有几个原因：

1. 多路平衡树结构，树高低，查询时磁盘 IO 次数少。
2. 非叶子节点主要存 key，能容纳更多索引项，提高扇出。
3. 数据都在叶子节点，查询路径更稳定。
4. 叶子节点之间有链表，范围查询和排序扫描更方便。

相比二叉树，B+ 树高度更低；相比 Hash 索引，B+ 树支持范围查询和排序。

## 常见追问

### 这题最容易被追问的边界是什么？

重点说清索引失效、锁范围扩大、主从延迟、统计信息过期、深分页和 DDL 风险。不要只给结论，要说明哪些条件下结论成立，哪些条件下会退化或需要换方案。

### 怎么证明自己不是在背模板？

用EXPLAIN、慢日志、optimizer trace、锁等待、事务状态、redo/undo/binlog 和复制延迟做验证。能说出具体信号、反例或排查入口，答案就从概念层进入实战层。

### 和相近方案怎么区分？

可以拿索引优化、SQL 改写、分页改造、读写分离、分区分库或归档策略对比，从查询性能、锁影响、一致性、扩展性和回滚难度几个维度选择。

### 面试官继续深挖时怎么展开？

先围绕“多路平衡树结构，树高低，查询时磁盘 IO 次数少。”讲清主链路，再补“补 InnoDB 页大小、聚簇索引/二级索引叶子内容差异、联合索引按字典序排列、范围条件后的索引使用边界、为什么低选择性字段收益有限”。如果能把边界条件、异常分支和验证闭环连起来，就能接住二面、三面的追问。

## 易错点

- 不要只说“查询快”，要说磁盘 IO 和范围查询。
- 不要把 B+ 树说成二叉树。
- 不要忽略叶子节点链表的作用。
