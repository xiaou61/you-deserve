## 一句话结论

MySQL 在处理复杂分组、排序、去重、子查询等场景时可能使用临时表，数据量大时会影响性能。

## 通俗解释

临时表像临时草稿纸。简单计算不用草稿纸，复杂统计就要先写中间结果，再继续算。

## 面试回答

常见触发场景：

- `group by` 无法直接利用索引。
- `order by` 和 `group by` 不一致。
- `distinct` 去重。
- 复杂子查询或 union。
- 需要保存中间结果。

EXPLAIN 的 Extra 里出现 `Using temporary`，说明可能用了临时表。

临时表可能在内存中，也可能因为数据量或字段类型转为磁盘临时表。

## 常见追问

### 这题最容易被追问的边界是什么？

重点说清索引失效、锁范围扩大、主从延迟、统计信息过期、深分页和 DDL 风险。不要只给结论，要说明哪些条件下结论成立，哪些条件下会退化或需要换方案。

### 怎么证明自己不是在背模板？

用EXPLAIN、慢日志、optimizer trace、锁等待、事务状态、redo/undo/binlog 和复制延迟做验证。能说出具体信号、反例或排查入口，答案就从概念层进入实战层。

### 和相近方案怎么区分？

可以拿索引优化、SQL 改写、分页改造、读写分离、分区分库或归档策略对比，从查询性能、锁影响、一致性、扩展性和回滚难度几个维度选择。

### 面试官继续深挖时怎么展开？

先围绕“group by 无法直接利用索引。”讲清主链路，再补“补内部临时表和业务临时表区别、内存临时表与磁盘临时表的转换条件、tmp_table_size/max_heap_table_size、Created_tmp_disk_tables 等指标、Using temporary 与 Using filesort 的组合风险，以及如何通过索引让分组/排序直接按顺序输出”。如果能把边界条件、异常分支和验证闭环连起来，就能接住二面、三面的追问。

## 易错点

- 临时表不是业务创建的临时表，而是执行过程中的中间表。
- 内存临时表和磁盘临时表成本不同。
- group by 和 order by 组合要特别关注。
