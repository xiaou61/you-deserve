## 一句话结论

MySQL 在处理复杂分组、排序、去重、子查询等场景时可能使用临时表，数据量大时会影响性能。

## 通俗解释

临时表像临时草稿纸。简单计算不用草稿纸，复杂统计就要先写中间结果，再继续算。

## 面试回答

常见触发场景：

- `group by` 无法直接利用索引。
- `order by` 和 `group by` 不一致。
- `distinct` 去重。
- 复杂子查询或 union。
- 需要保存中间结果。

EXPLAIN 的 Extra 里出现 `Using temporary`，说明可能用了临时表。

临时表可能在内存中，也可能因为数据量或字段类型转为磁盘临时表。

## 常见追问

### Using temporary 一定很严重吗？

不一定。小数据量影响不大，大数据量或频繁查询才需要重点优化。

### 怎么减少临时表？

优化索引、减少返回字段、改写 SQL、避免不必要的分组去重。

## 易错点

- 临时表不是业务创建的临时表，而是执行过程中的中间表。
- 内存临时表和磁盘临时表成本不同。
- group by 和 order by 组合要特别关注。

## 记忆钩子

**临时表是 SQL 的草稿纸，草稿太大就慢。**
