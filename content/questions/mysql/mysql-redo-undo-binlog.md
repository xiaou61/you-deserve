## 一句话结论

redo log 保证崩溃恢复，undo log 支持回滚和 MVCC，binlog 用于主从复制和数据恢复。三者解决的问题不同。

## 通俗解释

可以把数据库修改想成记账：

- redo log 是“我已经把这笔账记到草稿里，断电后也能补上”。
- undo log 是“如果这笔操作要撤销，按这个反向记录恢复”。
- binlog 是“把所有变更按顺序记成外部流水，方便同步给别人或以后重放”。

## 面试回答

redo log 是 InnoDB 存储引擎层日志，记录物理页修改，主要用于崩溃恢复，保证事务持久性。

undo log 记录修改前的数据版本，用于事务回滚，也用于 MVCC 构建历史版本链。

binlog 是 MySQL Server 层日志，记录逻辑变更，常用于主从复制、数据恢复和审计。

事务提交时还涉及两阶段提交，用来保证 redo log 和 binlog 的一致性，避免一个成功一个失败导致主库和从库数据不一致。

## 常见追问

### 这题最容易被追问的边界是什么？

重点说清索引失效、锁范围扩大、主从延迟、统计信息过期、深分页和 DDL 风险。不要只给结论，要说明哪些条件下结论成立，哪些条件下会退化或需要换方案。

### 怎么证明自己不是在背模板？

用EXPLAIN、慢日志、optimizer trace、锁等待、事务状态、redo/undo/binlog 和复制延迟做验证。能说出具体信号、反例或排查入口，答案就从概念层进入实战层。

### 和相近方案怎么区分？

可以拿索引优化、SQL 改写、分页改造、读写分离、分区分库或归档策略对比，从查询性能、锁影响、一致性、扩展性和回滚难度几个维度选择。

### 面试官继续深挖时怎么展开？

先围绕“redo log 是 InnoDB 存储引擎层日志，记录物理页修改，主要用于崩溃恢复，保证事务持久性。”讲清主链路，再补“补 crash 场景判断：redo prepare、binlog 写入、redo commit 三个点如何保证一致”。如果能把边界条件、异常分支和验证闭环连起来，就能接住二面、三面的追问。

## 易错点

- 不要把 redo log 和 binlog 都说成“恢复日志”就结束。
- 不要忘记 binlog 是 Server 层，redo/undo 是 InnoDB 层。
- 不要忽略两阶段提交。
