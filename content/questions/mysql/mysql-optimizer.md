## 一句话结论

MySQL 优化器负责为 SQL 选择相对合适的执行计划，比如用哪个索引、表连接顺序、访问方式和排序策略。

## 通俗解释

优化器像导航软件。目的地一样，但它会根据路况选择走哪条路，可能走高速，也可能绕开拥堵。

## 面试回答

优化器会做：

- 解析 SQL 后生成可能的执行方案。
- 根据表统计信息估算成本。
- 选择访问路径，比如全表扫描或索引扫描。
- 决定 join 顺序和 join 方式。
- 判断是否使用排序、临时表等。

优化器不保证永远选到最优计划，统计信息不准、数据分布倾斜、条件复杂时都可能选错。

## 常见追问

### 为什么明明有索引却不走？

可能是优化器估算全表扫描成本更低，或者索引选择性差、条件写法导致无法使用索引。

### 怎么看优化器选择？

可以用 EXPLAIN 或 EXPLAIN ANALYZE 查看执行计划和实际执行情况。

## 易错点

- 不要以为有索引就一定使用。
- 优化器基于成本估算，可能受统计信息影响。
- SQL 优化要看实际执行计划。

## 记忆钩子

**优化器是 SQL 导航，选路不一定永远完美。**
