## 一句话结论

索引失效通常是因为查询条件破坏了索引的有序性，或者优化器判断使用索引不划算，所以最终没有走预期索引。

## 通俗解释

索引像一本排好序的目录。你按目录规则查，速度就很快；但如果你把每个目录项都先加工一遍再查，或者只给了后半截信息，目录就帮不上忙了。

比如索引列是手机号，你写 `where left(phone, 3) = '138'`，数据库要先对每一行手机号做函数计算，原本排好序的索引就很难直接使用。

## 面试回答

常见索引失效场景包括：

1. 对索引列使用函数或表达式，比如 `date(create_time)`。
2. 发生隐式类型转换，比如字符串字段用数字条件查询。
3. 联合索引不满足最左前缀原则。
4. 范围查询后面的联合索引字段可能无法继续用于定位。
5. 使用 `like '%xxx'` 这种左模糊匹配。
6. `or` 两边不是都有可用索引，可能导致整体不走索引。
7. 字段区分度太低，优化器认为全表扫描更划算。

最终是否走索引要用 `EXPLAIN` 看执行计划，而不是只凭感觉判断。

## 常见追问

### 索引失效是不是一定会全表扫描？

不一定。可能只是没走你预期的那个索引，转而用了别的索引；也可能走了整棵索引扫描，效率仍然不好。面试里要区分“不走预期索引”和“完全退化成全表扫描”不是一回事。继续展开时，最好带上 `EXPLAIN` 的 `type`、`key`、`rows` 一起判断。

### 怎么避免函数导致索引失效？

尽量别在索引列本身上做加工，优先把计算挪到条件右侧，或者改写成范围查询。再进一步，才考虑冗余字段、生成列、函数索引这些结构性方案。继续讲时可以补一句：不是“函数一出现就绝对不能用”，而是要看它有没有破坏 B+ 树按原值排序的能力。

### 为什么明明建了索引，优化器还是故意不用？

因为优化器看的是总成本，不是“既然建了索引就一定要走”。如果字段选择性太低、命中行太多、回表成本高，或者表本来就很小，全表扫可能反而更便宜。这个追问很适合把答案从死规则拉回成本模型，看起来更像真正调过 SQL。

### 线上怎么快速判断是 SQL 写法问题，还是索引设计问题？

先看 SQL 是否有函数、隐式转换、左模糊、跳列、范围中断这些典型写法问题；再看现有索引是否匹配高频过滤和排序；最后用 `EXPLAIN`、慢日志和真实数据分布确认。排查顺序最好是“SQL 写法 -> 索引结构 -> 优化器成本判断”，别一上来就盲目加索引。

## 易错点

- 不要把所有慢查询都归因于索引失效。
- 不要忽略优化器成本选择，索引存在不代表一定会用。
- 不要只背规则，最好能配合 `EXPLAIN` 说明。

## 详细讲解

MySQL 索引失效不是“索引坏了”，而是优化器在当前 SQL、数据分布和成本估算下，没有办法或没有必要使用这个索引。面试里要先把思路摆正：索引是一种有序数据结构，能帮助快速定位范围；一旦 SQL 写法破坏了这种有序定位能力，或者用索引的成本比扫表更高，优化器就可能放弃索引。

最常见的是对索引列做函数或表达式计算。比如 where date(create_time) = 某天，create_time 本来在索引里按原值有序，但你把它包成 date 函数后，MySQL 很难直接用原始索引定位。更好的写法是把条件改成范围：create_time >= 当天零点 and create_time < 次日零点。类似地，对字段做运算、字符串截取、隐式函数转换，都可能让索引列无法直接参与定位。

隐式类型转换也是线上常见坑。字段是 varchar，SQL 里却用数字去比较，MySQL 可能把字段转换成数字再比较，相当于对列做函数，索引就难以使用。字符集或排序规则不一致也可能导致转换。这个问题在日志里不一定明显，所以排查时要看表结构、参数类型和最终 SQL。很多 Java 项目里，Mapper 参数类型写错，也会造成这种问题。

联合索引失效要结合最左前缀理解。索引 (a, b, c) 先按 a 排，再按 b 排，再按 c 排。查询跳过 a 直接查 b，通常无法有效使用这个联合索引。范围条件也可能中断后续列的定位能力，比如 a = 1 and b > 10 and c = 3，b 之后的 c 不一定能继续用于缩小扫描范围。注意这里说的是定位能力，不等于后续列完全没用，ICP 可能仍能用来过滤。

还有一种情况是优化器主动不用索引。比如性别、状态这类低选择性字段，命中大量行，走二级索引还要回表，成本可能比全表扫描更高。小表数据量很少时，扫表也可能更便宜。order by、like 前缀、or 条件、not equal、is not null 等也要结合数据量和索引结构判断，不要机械地说“一定失效”。

正确排查方式是 explain 加真实数据验证。重点看 type 是否从 ALL 改善到 range、ref、const；key 是否命中预期索引；rows 估算是否过大；Extra 里是否有 Using filesort、Using temporary、Using index condition、Using where。图解可以画成一条诊断链：SQL 条件进入后，依次检查函数计算、类型转换、联合索引顺序、范围中断、选择性和成本，最后用 explain 验证，而不是凭感觉改索引。

## 深挖理解

这道题不要只停在“是什么”。面试官真正想确认的是：你能不能把 MySQL 索引为什么会失效 放回真实系统里，讲清楚它为什么出现、解决什么问题、代价是什么。可以先用一句话定调：索引失效通常是因为查询条件破坏了索引的有序性，或者优化器判断使用索引不划算，所以最终没有走预期索引

拆开来看，第一层是背景问题：从函数计算、隐式转换、最左前缀、范围查询和低选择性字段理解索引失效。 如果只背结论，很容易在追问里断掉；更稳的方式是先说明问题发生的场景，再解释机制为什么能缓解这个问题。

第二层是核心机制：对索引列使用函数或表达式，比如 date(create_time)。 这里要尽量把动作讲成链路，而不是罗列名词。比如谁先发生、谁依赖谁、哪个状态会改变、失败时会留下什么痕迹。

第三层是边界和取舍：发生隐式类型转换，比如字符串字段用数字条件查询。 它通常不是银弹，真正的面试加分点是能主动说出适用范围、性能影响、复杂度和替代方案。

最后落到风险意识：联合索引不满足最左前缀原则。 不要把数据库或缓存讲成万能组件，要说明适用范围和兜底手段。这样回答会比单纯背八股更像做过项目的人。

## 实战落地

- **什么时候会遇到**：当业务里出现 MySQL、索引、SQL优化、Explain 相关的慢查询、锁等待、热点 key、缓存不一致、内存上涨或写入抖动时，就要把这道题放到读写链路里分析。
- **怎么做方案**：先明确数据规模和访问模式，再选择索引、事务、缓存、过期策略或分片策略。围绕“读路径、写路径、并发控制、失败补偿、容量上限”五个点展开。
- **怎么验证效果**：看 explain、慢日志、锁等待、命中率、内存、QPS、P99 延迟和错误率。把答案落到慢查询、热点 key、事务隔离、缓存一致性或容量评估上。
- **怎么兜底**：准备降级读、限流、重建索引、缓存预热、补偿任务或数据修复脚本，避免单点方案拖垮主链路。

## 追问准备

- **如果数据量或并发量扩大 10 倍怎么办？** 先回答瓶颈会出现在哪里，再说扩容、分片、缓存、异步化或限流策略，最后补一句监控指标怎么验证。
- **如果它失败了会有什么表现？** 从用户现象、服务日志、核心指标、数据状态四个角度描述。能说出失败表现，就能自然过渡到排查方案。
- **和相近方案怎么选？** 不要直接说“看场景”，要给出判断维度：一致性要求、延迟要求、吞吐量、实现复杂度、团队维护成本和故障恢复成本。
- **你在项目里会怎么讲？** 用“背景 -> 方案 -> 取舍 -> 验证 -> 复盘”的顺序，把 MySQL 索引为什么会失效 讲成一次工程决策，而不是一个孤立知识点。重点围绕 数据结构、读写路径、一致性边界和性能影响。

面试时可以按这个节奏组织：

3. **补充边界**：说明什么情况下有效，什么情况下会失效，以及为什么需要配套措施。
